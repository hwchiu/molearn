> 🍱 **類比**：ClusterClass 就像餐廳的「套餐菜單」— 平台團隊（主廚）設計好幾套固定套餐（ClusterClass），每套包含前菜（Infrastructure）、主菜（Control Plane）和配菜（Workers）的標準組合；開發團隊（客人）只需選擇套餐名稱和填入份量（variables），不需要逐道菜詳細指定食材和烹飪方式，大幅降低點餐出錯（配置錯誤）的機率，而且每次都保證品質一致。

## 🌐 整體觀念：沒有 ClusterClass 之前，你遇到什麼問題？

想像你的公司有 20 個開發團隊，每個團隊都需要自己的 Kubernetes 叢集。如果沒有 ClusterClass，每個叢集都需要單獨定義一套完整的 YAML：`Cluster`、`KubeadmControlPlane`、`MachineDeployment`、`InfrastructureCluster`……每個物件都有大量重複的設定，稍有不一致就埋下安全漏洞或設定錯誤的風險。

更頭疼的是「升級」問題：當公司決定把所有叢集從 Kubernetes v1.27 升到 v1.28，你需要一個個修改 20 份 YAML，每一份都可能遺漏某個欄位。

**有了 ClusterClass 之後呢？**

平台團隊只需維護一份「叢集藍圖」（ClusterClass）。開發團隊建立叢集時只需填入三件事：選哪個套餐（`class`）、用哪個 K8s 版本（`version`）、要幾台 Worker（`replicas`）。升級時，平台團隊更新一份 ClusterClass，Topology Controller 自動將變更傳播到所有使用它的叢集。

**Before — 沒有 ClusterClass：每個叢集都要完整定義**

```yaml
# 20 個叢集，20 份這樣的 YAML，全部要手動維護
apiVersion: cluster.x-k8s.io/v1beta1
kind: Cluster
metadata:
  name: team-a-cluster
spec:
  clusterNetwork: { ... }
  infrastructureRef:
    apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
    kind: Metal3Cluster
    name: team-a-cluster
  controlPlaneRef:
    apiVersion: controlplane.cluster.x-k8s.io/v1beta1
    kind: KubeadmControlPlane
    name: team-a-cluster-kcp
---
# 還需要 KubeadmControlPlane、Metal3Cluster、MachineDeployment、
# KubeadmConfigTemplate、Metal3MachineTemplate ... 共 6+ 個物件
```

**After — 有了 ClusterClass：開發團隊只填三行**

```yaml
apiVersion: cluster.x-k8s.io/v1beta1
kind: Cluster
metadata:
  name: team-a-cluster
spec:
  topology:
    class: enterprise-standard    # 引用 ClusterClass
    version: v1.28.0
    workers:
      machineDeployments:
        - class: standard-worker
          name: app-workers
          replicas: 5
    variables:
      - name: imageURL
        value: "https://images.example.com/ubuntu-22.04.tar.gz"
```

Topology Controller 接手，自動從 ClusterClass 生成所有需要的底層物件。

## 🔄 核心流程：Topology Controller 如何讓這一切自動化？

> 💡 **場景**：平台團隊建立好 `enterprise-standard` ClusterClass，開發團隊執行 `kubectl apply` 建立上面那個簡化版 Cluster。Topology Controller 接下來做了什麼？

**Step 1 — 讀取 ClusterClass 藍圖**

Topology Controller 讀取 `enterprise-standard` ClusterClass，找到所有模板引用：InfraClusterTemplate、ControlPlaneTemplate、Worker MachineDeploymentClass 等。

**Step 2 — 解析 Variables 與 Patches**

從 `Cluster.Spec.Topology.Variables` 讀取使用者填入的參數值（如 `imageURL`），套用 ClusterClass 中定義的 `spec.patches` 規則，將參數值注入對應的模板欄位（例如將 image URL 寫入 Metal3MachineTemplate 的 `spec.template.spec.image.url`）。

**Step 3 — 生成底層物件**

根據修改後的模板，自動建立：
- `Metal3Cluster`（InfrastructureCluster）
- `KubeadmControlPlane`（ControlPlane）
- `MachineDeployment` + `KubeadmConfigTemplate` + `Metal3MachineTemplate`

所有生成的物件都帶有 `topology.cluster.x-k8s.io/owned: true` label，表示由 Topology Controller 管理。

**Step 4 — 持續協調（Ongoing Reconcile）**

此後每次 ClusterClass 模板更新，Topology Controller 自動偵測變更並重新 reconcile——只更新有差異的部分，不重建所有機器。若 Cluster 設定了 `topology.rolloutAfter`，升級會等到指定時間才執行。

## 🧩 元件詳解

### ClusterClass 結構

**主要欄位**

| 欄位 | 說明 |
|------|------|
| `spec.infrastructure.ref` | InfraClusterTemplate 引用 |
| `spec.controlPlane.ref` | ControlPlaneTemplate 引用（如 KubeadmControlPlaneTemplate）|
| `spec.controlPlane.machineInfrastructure.ref` | CP 節點的 InfraMachineTemplate |
| `spec.workers.machineDeployments` | Worker MachineDeployment class 定義列表 |
| `spec.variables` | 可配置的變數定義（含 schema 和預設值）|
| `spec.patches` | JSON Patch 定義，將 variable 值注入 template |

**Variables Schema（OpenAPI v3）**

ClusterClass 使用 OpenAPI v3 Schema 定義每個變數的型別、預設值與合法範圍：

```yaml
variables:
  - name: imageURL
    required: true
    schema:
      openAPIV3Schema:
        type: string
        description: "OS image 的 URL"
        example: "https://images.example.com/ubuntu-22.04.tar.gz"
  - name: controlPlaneReplicas
    required: false
    schema:
      openAPIV3Schema:
        type: integer
        default: 3
        minimum: 1
        maximum: 5
  - name: workerMachineType
    required: false
    schema:
      openAPIV3Schema:
        type: string
        default: "standard"
        enum: ["standard", "high-memory", "gpu"]
```

<Callout type="tip" title="新手提示：什麼時候才需要 ClusterClass？">
如果你只管理 1-3 個叢集，直接建立 Cluster + KCP + MachineDeployment 就夠了。ClusterClass 的價值在於**大規模管理**場景：當你需要管理 10 個以上的同類叢集，或者需要讓開發團隊自助建立符合規範的叢集時，ClusterClass 的標準化能力才真正發揮作用。初學時可以先跳過 ClusterClass，等熟悉基礎資源後再回來學習。
</Callout>

### Cluster Topology 完整設定範例

```yaml
apiVersion: cluster.x-k8s.io/v1beta1
kind: Cluster
metadata:
  name: production-cluster
  namespace: default
spec:
  topology:
    class: enterprise-standard    # 引用 ClusterClass
    version: v1.28.0
    controlPlane:
      replicas: 3
      metadata:
        labels:
          tier: control-plane
    workers:
      machineDeployments:
        - class: standard-worker
          name: app-workers
          replicas: 5
          metadata:
            labels:
              workload: application
        - class: gpu-worker
          name: ml-workers
          replicas: 2
          variables:
            overrides:              # 覆蓋 ClusterClass 層級的 variable
              - name: workerMachineType
                value: "gpu"
    variables:
      - name: imageURL
        value: "https://images.example.com/ubuntu-22.04.tar.gz"
      - name: controlPlaneReplicas
        value: 3
```

**Variable 覆蓋機制**

- **全域 variables**：在 `spec.topology.variables` 設定，套用到所有使用該 variable 的模板
- **MachineDeployment 層級覆蓋**：個別 MachineDeployment 可在 `variables.overrides` 覆蓋特定 variable 的值
- **覆蓋範圍隔離**：覆蓋只影響該 MachineDeployment，不影響其他 worker 或 control plane 的設定

### Patches 完整說明

`spec.patches` 定義如何將 variable 值注入對應的模板物件。每個 patch 透過 `selector` 指定目標模板的 apiVersion、kind 及套用對象：

```yaml
patches:
  - name: imageURLPatch
    definitions:
      - selector:
          apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
          kind: Metal3MachineTemplate
          matchResources:
            controlPlane: true              # 套用到 CP 的 MachineTemplate
            machineDeploymentClass:
              names: ["standard-worker"]    # 套用到特定 class 的 MD
        jsonPatches:
          - op: replace
            path: /spec/template/spec/image/url
            valueFrom:
              variable: imageURL
          - op: replace
            path: /spec/template/spec/image/checksum
            valueFrom:
              template: |
                &#123;&#123; .imageChecksum &#125;&#125;
```

<Callout type="info" title="套用順序">

`spec.patches` 陣列中的 patch 按照**陣列索引順序**依序套用。後面的 patch 可以覆蓋前面 patch 已設定的欄位，因此定義多個 patch 時需注意順序，確保最終結果符合預期。

</Callout>

### MachineDeploymentClass 深入說明

`spec.workers.machineDeployments` 定義每一種 worker node 的「類型（class）」，每個 class 包含自己的 bootstrap template、infrastructure template、健康檢查規則及命名策略：

```yaml
spec:
  workers:
    machineDeployments:
      - class: standard-worker
        template:
          bootstrap:
            ref:
              apiVersion: bootstrap.cluster.x-k8s.io/v1beta1
              kind: KubeadmConfigTemplate
              name: standard-worker-bootstrap
          infrastructure:
            ref:
              apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
              kind: Metal3MachineTemplate
              name: standard-worker-machine
        machineHealthCheck:          # 為此 class 定義 MHC
          unhealthyConditions:
            - type: Ready
              status: Unknown
              timeout: 300s
          maxUnhealthy: "40%"
        namingStrategy:              # 自定義命名策略
          template: "&#123;&#123; .cluster.name &#125;&#125;-&#123;&#123; .machineDeployment.topologyName &#125;&#125;-&#123;&#123; .random &#125;&#125;"
```

`namingStrategy.template` 使用 Go template 語法，可引用 `.cluster.name`、`.machineDeployment.topologyName` 及 `.random` 等內建變數，讓產生的 MachineDeployment 物件名稱具有可讀性。

### ClusterClass 升級（版本管理）

ClusterClass 的 `spec.infrastructure.ref`、`spec.controlPlane.ref` 等欄位指向的模板物件（如 KubeadmControlPlaneTemplate）本身可以更新。TopologyReconciler 在下次 reconcile 時，會偵測到模板已變更，並自動將變更傳播到所有使用此 ClusterClass 的叢集。

若需要進行大幅度的結構性變更，建議建立全新版本的 ClusterClass（例如 `enterprise-standard-v2`），避免影響仍在使用舊版的叢集。叢集遷移到新 ClusterClass 時，只需修改 `Cluster.Spec.Topology.Class` 欄位即可。

**多叢集升級策略**

1. **更新模板物件**：修改 ClusterClass 引用的 template（例如調整 KubeadmControlPlaneTemplate 中的 kubeadm 配置）
2. **自動偵測與重新協調**：TopologyReconciler 偵測到 ClusterClass 變更後，自動重新 reconcile 所有使用此 ClusterClass 的 Cluster
3. **控制升級時機**：若 Cluster 設定了 `topology.rolloutAfter` 欄位，可指定一個時間點，確保升級在預定的維護視窗內才執行

### ClusterTopology 優缺點

**優點**

- **標準化建立流程**：統一的叢集建立方式，大幅減少人為設定錯誤
- **集中管理多叢集配置**：修改一個 ClusterClass 即可影響所有使用它的叢集
- **簡化版本升級**：升級 Kubernetes 版本只需修改 `topology.version` 一個欄位
- **靈活的參數化**：透過 variable 和 patches 機制，讓同一個 ClusterClass 能適應不同場景的需求

**限制**

<Callout type="warning" title="使用限制">

- **學習曲線較高**：ClusterClass 涉及 variables、patches、selector 等多個抽象層，初期理解需要時間
- **JSON Patch 語法複雜**：`spec.patches` 中的 JSON Patch 定義較為繁瑣，需要熟悉 RFC 6902 語法
- **不適合高度客製化的叢集**：當每個叢集的差異極大時，強行使用 ClusterClass 反而會讓 patches 變得難以維護
- **Template 物件必須預先存在**：ClusterClass 引用的所有模板物件（如 KubeadmControlPlaneTemplate）若不存在，Topology reconcile 將持續失敗，需確保相依物件已正確建立

</Callout>

## ⚠️ 常見陷阱與注意事項

<Callout type="warning" title="常見誤解澄清">
1. **「修改 ClusterClass 會立即重建所有使用它的叢集」** — 不正確。修改 ClusterClass 會觸發 Topology Controller 重新 reconcile，但 Controller 只會更新**有差異的部分**（例如更新 KubeadmControlPlaneTemplate 的 version），不會無差別重建所有機器。滾動更新仍由 KCP 和 MachineDeployment 各自控制。
2. **「ClusterClass 的 variables 可以覆寫任何 Template 欄位」** — 有限制。variables 的值必須透過 `patches` 中明確定義的映射規則才能注入到 Template，沒有對應 patch 規則的 variable 不會有任何效果，且 ClusterClass webhook 會驗證 variable schema。
3. **「使用 Topology 管理的叢集和手動建立的叢集可以混用」** — 不建議。一旦 Cluster 設定了 `spec.topology`，該叢集就交由 Topology Controller 管理，手動修改 KCP 或 MachineDeployment 的欄位可能在下次 reconcile 時被 Topology Controller 覆蓋回去。
</Callout>

> 💡 **思考一下**：ClusterClass 的 `spec.patches` 使用 JSON Patch 或 MergePatch 來修改 Template 內容。為什麼 CAPI 不直接允許在 Cluster 的 `topology.variables` 中覆寫任意 Template 欄位，而要透過 patches 定義映射關係？這種設計有什麼安全性或可維護性上的考量？
