> 🔄 **類比**：Machine 的生命週期就像一位「外派員工的工作旅程」— 從收到派遣通知（Pending）、準備出發（Provisioning）、抵達目的地上班（Running），到最後任務結束的離職程序（Deleting）。離職程序特別嚴謹：要先完成交接（Pre-drain Hook）、結束手上工作（Drain）、歸還設備（Volume Detach）、最後告知相關系統（Pre-terminate Hook）才能正式離職，確保不影響團隊運作。

## 🌐 整體觀念：一台 Machine 的誕生與消滅

在 CAPI 的世界裡，一台「機器」不是立即出現的——它要先通過一系列準備工作，才能成為 Kubernetes 叢集中正式的節點。更重要的是，它的「消失」同樣不是一刀斬斷，而是一段精心設計的告別流程。

為什麼需要這麼複雜的流程？因為生產環境中的節點往往承載著有狀態的工作負載：資料庫、訊息佇列、有 PersistentVolume 的應用程式。如果節點突然消失，資料可能損毀，服務可能中斷。CAPI 的 Machine 生命週期設計，就是為了確保「每一台機器的上線和下線都對服務零影響」。

## 🔄 核心流程：一台 Machine 從誕生到消滅的故事

![Machine 生命週期](/diagrams/cluster-api/machine-lifecycle.png)

> 💡 **場景**：你的叢集需要一台新的 Worker 節點。讓我們跟著這台機器，從它的 YAML 物件被建立的那一刻，一路追蹤到它最後從叢集中消失。

### 誕生之旅：Pending → Provisioning → Provisioned → Running

**第一站：Pending — 等待「入職文件」**

Machine 物件一建立就進入 `Pending` 狀態。此時機器還沒有任何實體存在，MachineReconciler 正在等待 Bootstrap Provider（如 KubeadmConfig）準備好「入職文件」——也就是 cloud-init userdata，裡面包含了讓節點加入叢集所需的所有初始化指令。

Bootstrap Provider Controller 根據叢集設定（kubeadm token、CA 憑證等）生成 cloud-init user-data，存入一個 Secret，再將 Secret 名稱寫入 `BootstrapConfig.Status.DataSecretName`。

**第二站：Provisioning — 實體機器正在建立**

Bootstrap 資料就緒後，Machine 進入 `Provisioning` 狀態，輪到 Infrastructure Provider 上場了。無論是 MAAS 上的裸機、Metal3 管理的 BareMetalHost，還是雲端的 VM，Infrastructure Provider 都在這個階段完成以下工作：

- 讀取 userdata，開始 provisioning（VM 建立、裸機 OS 安裝等）
- 機器啟動後，cloud-init / ignition 執行 `kubeadm join`，嘗試加入叢集
- Provisioning 完成後，回寫 `InfrastructureMachine.Status.ProviderID`

**第三站：Provisioned → Running — 節點加入叢集**

Infrastructure Provider 完成後，MachineReconciler 透過 ClusterCache 連線到 workload cluster，依 ProviderID 比對 Node 物件，等待對應 Node 出現且進入 `Ready` 狀態。找到後設定 `Machine.Status.NodeRef`，Machine 正式進入 `Running` 狀態——它現在是叢集中一個完整的節點了。

---

### 告別之旅：Deleting — 8 步驟的優雅退場

某一天，這台 Machine 需要被移除（縮容、版本升級、MachineHealthCheck 修復或手動刪除）。Kubernetes 設定 `DeletionTimestamp`，MachineReconciler 開始執行精心設計的 8 步驟告別流程。

**Step 1 — Pre-Drain Hook（告別前的最後準備）**

如果有外部系統（備份工具、監控系統）需要在 drain 前做準備，可以在 Machine 上設定 annotation：

```
pre-drain.delete.hook.machine.cluster.x-k8s.io/<hook-name>
```

MachineReconciler 看到這個 annotation 就暫停等待。外部系統完成後移除 annotation，流程才繼續。

```bash
# 外部系統完成備份後，移除 annotation 以解除阻塞
kubectl annotate machine my-machine \
  pre-drain.delete.hook.machine.cluster.x-k8s.io/backup-
```

**Step 2 — Cordon（宣告「不接新案」）**

對 workload cluster 中的對應 Node 執行 Cordon（等同 `kubectl cordon`），標記為 `Unschedulable = true`。從此刻起，不再有新的 Pod 排程到這台節點。

**Step 3 — Drain（優雅地把工作交出去）**

對節點執行 drain，把所有 Pod 驅逐到其他節點，整個過程遵守 PodDisruptionBudget（PDB）規則。DaemonSet Pods 和 Mirror Pods 會跳過（它們有自己的清理機制）。

```yaml
spec:
  nodeDrainTimeout: 10m  # 預設：無超時，等待 PDB 允許驅逐
```

<Callout type="warning" title="PDB 阻擋 Drain">

若 Pod 設定了 PDB 且 `disruptionsAllowed = 0`，drain 會被完全阻塞。

- 確保 PDB 的 `maxUnavailable` 設定合理，避免無限阻塞刪除流程
- 強烈建議設定 `nodeDrainTimeout`，避免升級或縮減操作永久卡住
- 若超過 `nodeDrainTimeout`，MachineReconciler 會強制繼續進行後續步驟

</Callout>

**Step 4 — Volume Detach（歸還「共用資料」）**

等待所有 CSI volumes 從節點卸除，防止兩個節點同時掛載同一塊 volume（split-brain 問題）。對資料庫等有狀態應用尤其重要。超時設定為 `Machine.Spec.NodeVolumeDetachTimeout`。

**Step 5 — Pre-Terminate Hook（最後通知）**

類似 Pre-Drain Hook，這個 hook 在 drain 完成、volumes 卸除後觸發，適合通知 CMDB、更新 load balancer、執行最終備份等操作。

```
pre-terminate.delete.hook.machine.cluster.x-k8s.io/<hook-name>
```

**Step 6 — 刪除 InfrastructureMachine（真正的「告別」）**

MachineReconciler 刪除 InfrastructureMachine 物件，觸發 Infrastructure Provider 執行底層清理：

- **AWS CAPA**：終止 EC2 Instance
- **Metal3 CAPM3**：BMH 進入 deprovisioning 流程
- **MAAS CAPMAAS**：釋放 MAAS Machine，重新進入 Ready 狀態

**Step 7 — 刪除 BootstrapConfig**

清理 KubeadmConfig 和 bootstrap Secret，確保 kubeadm token、憑證等敏感資料不殘留。

**Step 8 — 移除 Finalizer**

MachineReconciler 移除最後一個 finalizer：

```
machine.cluster.x-k8s.io
```

Kubernetes API Server 將 Machine 物件從 etcd 中正式刪除。這台 Machine 完成了它的使命。

<Callout type="tip" title="新手提示：為什麼刪除要這麼多步驟？">
Machine 的 8 步驟刪除流程每一步都有意義：Cordon 確保不再有新工作分配到此節點；Drain 讓現有工作優雅地遷移到其他節點（遵守 PDB）；Volume Detach 保護有狀態應用的資料；最後才真正刪除機器。這些步驟合在一起，確保服務在節點被移除時不會有任何中斷或資料遺失。
</Callout>

## 🧩 元件詳解

### Machine Phase 狀態機

| Phase | 說明 | 前置條件 |
|-------|------|---------|
| `Pending` | Machine 剛建立，等待 bootstrap 就緒 | 初始狀態 |
| `Provisioning` | Bootstrap 就緒，Infrastructure Provider 正在建立機器 | Bootstrap ready=true |
| `Provisioned` | Infrastructure Provider 已建立機器（有 ProviderID）| InfraMachine ready=true |
| `Running` | Machine 對應的 K8s Node 已加入叢集且 Ready | NodeRef 已設定 |
| `Deleting` | Machine 正在刪除流程中 | DeletionTimestamp 已設定 |
| `Deleted` | Machine 已完全刪除 | 最終狀態 |
| `Failed` | 發生不可恢復的錯誤 | 需人工介入 |
| `Unknown` | 狀態不明（通常是 controller 無法連線）| — |

### 觸發刪除的情境

Machine 可能因下列原因進入刪除流程：

- **手動刪除**：使用者執行 `kubectl delete machine <name>`
- **MachineSet/MachineDeployment 縮減**：副本數下調時，MachineSet 會選擇並刪除多餘的 Machine
- **KCP Rolling Update**：KubeadmControlPlane 升級時會依序建立新 CP Machine、刪除舊 CP Machine
- **MachineHealthCheck Remediation**：節點健康檢查失敗，CAPI 預設策略會刪除問題 Machine 並由 MachineSet 重建

### Machine Conditions 詳解

Machine 的 `Status.Conditions` 提供比 Phase 更細粒度的健康狀態，可透過 `kubectl get machine -o yaml` 或 `clusterctl describe cluster` 查看。

| Condition | True 條件 | 說明 |
|-----------|----------|------|
| `InfrastructureReady` | InfraMachine.Status.Ready=true | Infrastructure 已就緒 |
| `BootstrapReady` | Bootstrap dataSecretName 已設定 | Bootstrap 資料已就緒 |
| `NodeHealthy` | Node Ready 且 NodeRef 已設定 | 節點健康 |
| `APIServerPodHealthy` | kube-apiserver pod 健康 | （CP only）API Server 運作正常 |
| `ControllerManagerPodHealthy` | kube-controller-manager pod 健康 | （CP only）Controller Manager 運作正常 |
| `SchedulerPodHealthy` | kube-scheduler pod 健康 | （CP only）Scheduler 運作正常 |
| `EtcdPodHealthy` | etcd pod 健康 | （CP only, local etcd）etcd Pod 運作正常 |
| `EtcdMemberHealthy` | etcd member 狀態健康 | （CP only, local etcd）etcd 成員健康 |

Control Plane Machine 的 Conditions 比 Worker Machine 多，因為 KCP Controller 會額外監控各個 CP 元件的健康狀況。

### ProviderID：Machine 與 Node 的身份橋樑

ProviderID 是 Machine 在 Infrastructure Provider 中的**唯一識別符**，也是連接 CAPI Machine 物件與 workload cluster K8s Node 的關鍵橋樑。

CAPI 透過以下方式建立對應關係：

1. Infrastructure Provider 將 ProviderID 寫入 `InfrastructureMachine.Spec.ProviderID`
2. MachineReconciler 將此值同步至 `Machine.Spec.ProviderID`
3. cloud-init 啟動時，kubelet 以相同的 ProviderID 向 K8s API 註冊，寫入 `Node.Spec.ProviderID`
4. MachineReconciler 掃描 workload cluster 的所有 Node，依 ProviderID 找到對應 Node

不同 Infrastructure Provider 的 ProviderID 格式各異：

| Provider | ProviderID 格式範例 |
|----------|-------------------|
| AWS (CAPA) | `aws:///ap-northeast-1a/i-0abc123def456` |
| Metal3 (CAPM3) | `metal3://metal3/my-bmh/my-m3machine` |
| MAAS (CAPMAAS) | `maas://abc123-def456-ghi789` |
| vSphere (CAPV) | `vsphere://420f1234-5678-abcd-ef01-234567890abc` |

**ProviderID 一旦設定即不可變更。** 若因特殊原因需要更改，必須重建整個 Machine 物件。

### 多版本升級考量

<Callout type="warning" title="版本升級注意事項">

CAPI 建議嚴格遵守以下升級原則，違反可能導致叢集不穩定：

- **逐版升級**：v1.27 → v1.28 → v1.29，不建議跨版本升級（如直接從 v1.27 升到 v1.29）
- **版本順序**：Worker 節點版本**不能高於** Control Plane 版本
- **升級順序**：先升級 Control Plane（透過更新 KubeadmControlPlane 的 `version` 欄位），再升級 Worker 節點（透過更新 MachineDeployment 的 `version` 欄位）
- **查看升級路徑**：使用 `clusterctl upgrade plan` 確認可用的升級選項及建議順序

違反版本順序可能導致 API 相容性問題，造成 workload cluster 控制面不穩定。

</Callout>

## ⚠️ 常見陷阱與注意事項

<Callout type="warning" title="常見誤解澄清">
1. **「Machine Phase 是 CAPI 完整管理叢集生命週期的唯一指標」** — 不完全正確。Machine Phase（Pending/Provisioning/Running/Deleting/Failed）是粗粒度的狀態描述，詳細的中間狀態需要查看 Machine 的 `status.conditions`（如 `BootstrapReady`、`InfrastructureReady`）才能了解目前卡在哪個環節。
2. **「Drain 完成後節點資料就已安全」** — 需要額外確認。Drain 只確保 Pod 已遷移（驅逐），但若有使用 PersistentVolume，還需等待 `VolumeAttachments` 完全 detach（刪除流程的 Step 4）才能確保磁碟資料安全，否則新節點掛載同一份磁碟時可能發生衝突。
3. **「設定了 pre-drain hook 就能無限延遲節點刪除」** — 不正確。CAPI 的 hook 機制有超時保護（`nodeDrainTimeout`），超過設定時間後 Controller 會強制繼續刪除流程，不會因為 hook 未回應而永久卡住。
</Callout>

> 💡 **思考一下**：Machine 的 `ProviderID` 設計成「一旦設定即不可變更」。這個不可變性保護的是什麼？如果 ProviderID 可以被修改，對 Controller 的 reconcile 邏輯和節點身份識別會造成什麼問題？
