## 什麼是 KubeadmControlPlane（KCP）

KubeadmControlPlane 是 Cluster API 的 **Control Plane Provider** 實作，屬於 `controlplane.cluster.x-k8s.io/v1beta1` API group。它負責管理 Control Plane 節點的完整生命週期，主要功能包括：

- **維護指定數量的 Control Plane Machine**：確保叢集中 CP 節點數量與 `spec.replicas` 一致
- **透過 kubeadm 執行 K8s 版本升級**：以 Rolling Update 方式逐台替換舊版節點
- **管理 etcd 叢集的健康狀態**：監控每台 CP 節點上 etcd member 的連線與資料一致性
- **提供 kubeconfig secret 給 CAPI Cluster Controller**：使上層 Cluster 物件可以存取 workload cluster API server

## KubeadmControlPlane Spec 重要欄位

| 欄位 | 類型 | 說明 |
|------|------|------|
| `replicas` | int32 | Control Plane 節點數（建議奇數：1, 3, 5）|
| `version` | string | Kubernetes 版本（如 v1.28.0）|
| `machineTemplate.infrastructureRef` | ObjectRef | Infrastructure Machine Template 引用 |
| `kubeadmConfigSpec` | KubeadmConfigSpec | kubeadm 設定（InitConfiguration, ClusterConfiguration 等）|
| `rolloutStrategy` | RolloutStrategy | 更新策略（RollingUpdate / 預設）|
| `rolloutStrategy.rollingUpdate.maxSurge` | IntOrString | 滾動更新最多額外建立數（預設 1）|
| `nodeDrainTimeout` | Duration | 節點 drain 超時時間 |
| `nodeVolumeDetachTimeout` | Duration | Volume detach 超時時間 |
| `nodeDeletionTimeout` | Duration | 節點刪除超時時間 |
| `machineTemplate.nodeDrainTimeout` | Duration | 機器層級的 drain 超時 |
| `remediationStrategy` | RemediationStrategy | KCP 自帶的 remediation 設定 |

## KubeadmConfigSpec 主要設定

`kubeadmConfigSpec` 是 KCP 中最核心的設定區塊，對應到每台 CP 節點上執行的 kubeadm 設定。

### initConfiguration（第一台 CP 節點使用）

第一台 Control Plane 節點透過 `kubeadm init` 初始化叢集，使用此區塊的設定：

```yaml
initConfiguration:
  nodeRegistration:
    name: "{{ ds.meta_data.local_hostname }}"
    kubeletExtraArgs:
      cloud-provider: external
      node-labels: "node-role.kubernetes.io/control-plane="
  localAPIEndpoint:
    advertiseAddress: "{{ ds.meta_data.local_ipv4 }}"
    bindPort: 6443
```

<Callout type="tip" title="新手提示：initConfiguration vs joinConfiguration">
第一台 Control Plane 使用 `initConfiguration`（kubeadm init，建立全新叢集），後續的 Control Plane 節點使用 `joinConfiguration`（kubeadm join，加入現有叢集）。KCP Controller 會根據目前是「第一台」還是「後續台」自動選擇正確的設定區塊，使用者不需要手動區分。KCP 內部用 `KubeadmControlPlaneInitialized` 這個 Condition 來判斷叢集是否已初始化。
</Callout>

### clusterConfiguration（K8s 叢集全域設定）

定義整個叢集層級的 Kubernetes 元件設定，包含 API Server、Controller Manager、etcd 及網路：

```yaml
clusterConfiguration:
  apiServer:
    extraArgs:
      cloud-provider: external
      audit-log-path: /var/log/audit.log
    certSANs:
      - localhost
      - 127.0.0.1
  controllerManager:
    extraArgs:
      cloud-provider: external
  etcd:
    local:
      dataDir: /var/lib/etcd
      extraArgs:
        quota-backend-bytes: "8589934592"  # 8GB
  networking:
    podSubnet: 192.168.0.0/16
    serviceSubnet: 10.96.0.0/12
    dnsDomain: cluster.local
  kubernetesVersion: v1.28.0
```

### joinConfiguration（後續 CP 節點使用）

第二台及之後加入的 Control Plane 節點透過 `kubeadm join` 加入叢集，使用此區塊：

```yaml
joinConfiguration:
  nodeRegistration:
    name: "{{ ds.meta_data.local_hostname }}"
    kubeletExtraArgs:
      cloud-provider: external
  controlPlane:
    localAPIEndpoint:
      advertiseAddress: "{{ ds.meta_data.local_ipv4 }}"
```

### files / preKubeadmCommands / postKubeadmCommands

可在 kubeadm 執行前後注入檔案或指令：

```yaml
files:
  - path: /etc/kubernetes/audit-policy.yaml
    content: |
      apiVersion: audit.k8s.io/v1
      kind: Policy
      rules:
        - level: Metadata
preKubeadmCommands:
  - systemctl enable --now containerd
  - systemctl enable --now kubelet
postKubeadmCommands:
  - echo "Control plane setup complete"
```

## 完整 KubeadmControlPlane YAML 範例

```yaml
apiVersion: controlplane.cluster.x-k8s.io/v1beta1
kind: KubeadmControlPlane
metadata:
  name: my-cluster-control-plane
  namespace: default
spec:
  replicas: 3
  version: v1.28.0
  machineTemplate:
    infrastructureRef:
      apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
      kind: Metal3MachineTemplate
      name: my-cluster-controlplane
  rolloutStrategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1
  nodeDrainTimeout: 5m
  kubeadmConfigSpec:
    initConfiguration:
      nodeRegistration:
        name: "{{ ds.meta_data.local_hostname }}"
        kubeletExtraArgs:
          cloud-provider: external
    clusterConfiguration:
      apiServer:
        extraArgs:
          cloud-provider: external
      controllerManager:
        extraArgs:
          cloud-provider: external
    joinConfiguration:
      nodeRegistration:
        name: "{{ ds.meta_data.local_hostname }}"
        kubeletExtraArgs:
          cloud-provider: external
```

## KubeadmControlPlane Status 欄位

| 欄位 | 類型 | 說明 |
|------|------|------|
| `initialized` | bool | 第一台 CP 節點是否完成初始化 |
| `ready` | bool | KCP 是否就緒（至少一台 CP Ready 且 etcd healthy）|
| `replicas` | int32 | 目前管轄的 CP Machine 數量 |
| `readyReplicas` | int32 | 就緒的 CP Machine 數量 |
| `updatedReplicas` | int32 | 已更新到最新版本的數量 |
| `unavailableReplicas` | int32 | 不可用數量 |
| `version` | string | 目前 CP 節點執行的版本 |
| `selector` | string | 用於 HPA 的 selector 字串 |
| `conditions` | []Condition | 健康狀態條件 |
| `observedGeneration` | int64 | 最後觀察到的 generation |

## Control Plane 升級流程

### 升級觸發條件

修改以下任一欄位即可觸發 KCP 的 Rolling Upgrade：

- `KubeadmControlPlane.Spec.Version`：變更 Kubernetes 版本
- `KubeadmControlPlane.Spec.MachineTemplate.InfrastructureRef`：變更 Infrastructure Machine Template（如更換 OS 映像）

### 升級步驟（以 3 台 CP 節點為例，maxSurge=1）

1. **建立第 4 台新版本 CP Machine**（surge），等待其完成 `kubeadm join` 並加入 etcd 叢集
2. **等待新 CP 節點就緒**：新節點的 etcd member 健康、API Server 可用
3. **Cordon + Drain 最舊的舊版本 CP 節點**：驅逐上面的 workload Pod
4. **刪除最舊的舊版本 CP Machine**：回收 Infrastructure 資源
5. **重複步驟 1–4** 直到全部 CP 節點都更新為新版本


<Callout type="warning" title="升級資源與 etcd 注意事項">

- 升級期間 KCP 會暫時出現 **`replicas + maxSurge`** 台 CP 節點（例如 3+1=4 台），請確保 infrastructure 有足夠資源可建立額外節點。
- etcd 叢集在偶數個成員時，leader 選舉的 quorum 計算會較為敏感。`maxSurge=1` 確保叢集經歷「奇數 → 偶數 → 奇數」的短暫過渡，而非長時間維持偶數狀態。

</Callout>

## KCP Remediation 機制

KCP 內建自己的 remediation 邏輯，與 `MachineHealthCheck` 是不同的機制。

### KCP 自帶的健康檢查

- KCP Controller 持續監控每台 CP 節點上的 etcd member 狀態
- 若某台 CP 節點的 etcd member 出現以下問題，KCP 會主動觸發 remediation：
  - peer 連接失敗
  - 資料不一致
  - etcd member 長時間無法加入或脫離 quorum

### RemediationStrategy 設定

```yaml
spec:
  remediationStrategy:
    maxRetry: 3
    retryPeriod: 1m
    minHealthyPeriod: 5m
```

| 欄位 | 說明 |
|------|------|
| `maxRetry` | 最大重試次數，超過後不再自動 remediate |
| `retryPeriod` | 兩次 remediation 之間的最短間隔，避免過於頻繁地替換節點 |
| `minHealthyPeriod` | 節點必須持續健康達此時間才視為「曾健康過」，防止對剛修好的節點重複觸發 remediation |

> 💡 **思考一下**：KCP 在升級 Control Plane 時為何堅持「一次只動一台」的策略？如果允許同時替換多台 Control Plane Machine，在 etcd quorum 的角度來看會有什麼風險？

## 常見誤解

<Callout type="warning" title="常見誤解澄清">
1. **「KCP 的 replicas 設為 3 就能保證高可用」** — 不完全正確。replicas=3 只是數量，還需要確保機器分散在不同的 Failure Domain（可用區），否則 3 台同在一個 AZ，該 AZ 故障時整個 Control Plane 仍會失效。
2. **「修改 kubeadmConfigSpec 中任何欄位都會觸發滾動更新」** — 不正確。只有「immutable 欄位」（如 `clusterConfiguration`、`initConfiguration`）的修改才會觸發，其他欄位（如 `files`、`preKubeadmCommands`）的修改不一定觸發滾動更新。
3. **「KCP remediation 和 MachineHealthCheck 是同一件事」** — 錯誤。KCP Remediation 是 KubeadmControlPlane 內建的 Control Plane 自癒機制；MachineHealthCheck 是通用的、針對所有 Machine 的健康檢查機制，兩者獨立運作，設定方式也不同。
</Callout>
