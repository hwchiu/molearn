## Manager 架構與 Controller 清單

![Reconcile Loop 概念](/diagrams/capi/reconcile-loop.png)

CAPI 在 management cluster 中運行一個 **controller-manager Pod**，其內部以多個 reconciler goroutine 的形式並行處理各種資源。這個 Pod 通常由 `clusterctl` 安裝，使用 Deployment 部署，並支援 leader election 以實現高可用。

![CAPI Manager 元件部署圖](/diagrams/capi/manager-components.png)

### Controllers 清單

| Controller | 管理資源 | 預設並發數 |
|-----------|---------|----------|
| `ClusterReconciler` | Cluster | 10 |
| `MachineReconciler` | Machine | 10 |
| `MachineSetReconciler` | MachineSet | 10 |
| `MachineDeploymentReconciler` | MachineDeployment | 10 |
| `MachinePoolReconciler` | MachinePool | 10 |
| `KubeadmConfigReconciler` | KubeadmConfig | 10 |
| `KubeadmControlPlaneReconciler` | KubeadmControlPlane | 10 |
| `MachineHealthCheckReconciler` | MachineHealthCheck | 10 |
| `ClusterResourceSetReconciler` | ClusterResourceSet | 10 |
| `TopologyReconciler` | Cluster（topology 模式）| 10 |
| `ClusterClassReconciler` | ClusterClass | 10 |

每個 reconciler 都是獨立的 goroutine pool，彼此不共用 worker，因此不同類型的資源可以完全並行處理。

## Provider 合約介面

![Provider 互動架構](/diagrams/capi/provider-interaction.png)

![Provider 生態系總覽](/diagrams/capi/provider-ecosystem.png)

Provider 合約是 CAPI 最重要的設計概念之一。CAPI 核心只定義**介面規格**，具體的基礎設施操作完全由各 Provider 實作。CAPI 與 Provider 之間透過 Kubernetes CRD 欄位的「約定格式」進行溝通——這就是所謂的「合約（Contract）」。

<Callout type="tip" title="新手提示：什麼是 Provider 合約">
CAPI 本身不知道如何建立 AWS EC2、Metal3 bare-metal或其他基礎設施 — 它只定義「合約」。合約就是一組欄位規範：CAPI 說「你只要在 `status.ready` 填寫 true/false，我就知道你是否就緒」。所有 Infrastructure Provider（AWSCluster、Metal3Cluster 等）只要遵守這份合約，就能無縫與 CAPI 核心配合，無需修改 CAPI 核心程式碼。
</Callout>

以下是 InfraCluster 與 InfraMachine 的合約欄位一覽：

| 合約層級 | 欄位 | 說明 | 必填 |
|---------|------|------|------|
| InfraCluster | `spec.controlPlaneEndpoint` | Control Plane API server 的 endpoint（host:port）| ✅ |
| InfraCluster | `status.ready` | Infrastructure 是否已就緒 | ✅ |
| InfraCluster | `status.failureDomains` | 可用的 failure domains（跨 AZ 分散）| ❌ |
| InfraMachine | `spec.providerID` | 機器的唯一識別碼（格式由 Provider 定義）| ✅ |
| InfraMachine | `status.ready` | 機器 VM 是否已建立並就緒 | ✅ |
| InfraMachine | `status.addresses` | 機器的 IP 地址列表 | ❌ |
| InfraMachine | `status.failureReason` | 失敗原因代碼 | ❌ |
| InfraMachine | `status.failureMessage` | 失敗訊息詳細描述 | ❌ |

Bootstrap Provider（`KubeadmConfig`）需實作 `status.dataSecretName` 和 `status.ready`；ControlPlane Provider（`KubeadmControlPlane`）需實作 `status.initialized`、`status.ready` 等欄位。詳細規格請參閱 [Provider 合約與 Runtime Hooks](/cluster-api/features/provider-contracts-runtime-hooks)。

## 六大 API Group

CAPI 的功能依職責分散在六個 API Group 中：

| API Group | 用途 |
|-----------|------|
| `cluster.x-k8s.io` | 核心：Cluster、Machine、MachineSet、MachineDeployment、ClusterClass、MachineHealthCheck |
| `bootstrap.cluster.x-k8s.io` | Bootstrap：KubeadmConfig、KubeadmConfigTemplate，產生 cloud-init userdata |
| `controlplane.cluster.x-k8s.io` | Control Plane：KubeadmControlPlane（KCP），管理 etcd 與 API server node |
| `addons.cluster.x-k8s.io` | Addons：ClusterResourceSet，自動安裝 CNI 等 Day 0 附加元件 |
| `ipam.cluster.x-k8s.io`（Alpha）| IP 管理：IPAddressClaim / IPAddress，標準化靜態 IP 請求介面 |
| `runtime.cluster.x-k8s.io`（Alpha）| Runtime SDK：ExtensionConfig，讓外部 webhook 介入cluster生命週期 hook |
