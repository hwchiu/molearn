> 📱 **類比**：Bootstrap Data 就像新手機的「開箱設定精靈」— 手機出廠時是白板，第一次開機時精靈會引導你設定語言、網路、帳號（對應 cloud-init 執行 kubeadm init/join）。KubeadmConfig 就是這份設定精靈的「腳本」，Infrastructure Provider 在建立機器時將這份腳本注入機器，確保節點啟動後自動完成所有 Kubernetes 初始化步驟。

## Bootstrap Provider 概念

在 Cluster API（CAPI）架構中，**Bootstrap Provider** 負責生成節點初始化所需的 cloud-init 或 ignition 資料。CAPI 核心本身不知道如何初始化節點內部環境；由 Bootstrap Provider 決定要執行哪些命令、安裝哪些套件、寫入哪些設定檔。

官方標準的 Bootstrap Provider 是 **kubeadm Bootstrap Provider（CABPK，Cluster API Bootstrap Provider Kubeadm）**，使用 `kubeadm` 完成節點的 Kubernetes 初始化流程。

Bootstrap Provider 的最終輸出是一個 **Kubernetes Secret**，內含 cloud-init 格式的 `userData`。Secret 名稱格式為 `<machine-name>-bootstrap-data`，由 Machine Controller 讀取後傳遞給 Infrastructure Provider，在建立實體機器或虛擬機器時注入為啟動時的 user-data。

## KubeadmConfig CRD

### 用途

`KubeadmConfig` 是為**單一 Machine** 生成的 bootstrap 設定資源。Machine 的 `spec.bootstrap.configRef` 指向對應的 KubeadmConfig（或 KubeadmConfigTemplate，後者是模板形式）。每台 Machine 對應一個 KubeadmConfig 物件。

### Spec 主要欄位

| 欄位 | 類型 | 說明 |
|------|------|------|
| `initConfiguration` | InitConfiguration | kubeadm init 設定（Control Plane 第一台使用）|
| `clusterConfiguration` | ClusterConfiguration | K8s 叢集全域設定（Control Plane 使用）|
| `joinConfiguration` | JoinConfiguration | kubeadm join 設定（Worker 和後續 CP 使用）|
| `files` | []File | 寫入節點的檔案 |
| `preKubeadmCommands` | []string | kubeadm 前執行的命令 |
| `postKubeadmCommands` | []string | kubeadm 後執行的命令 |
| `users` | []User | 建立的系統使用者（含 SSH key）|
| `ntp` | NTP | NTP server 設定 |
| `format` | string | cloud-init 格式：`cloud-config`（預設）或 `ignition` |
| `verbosity` | int32 | kubeadm 日誌詳細程度（0–9）|
| `useExperimentalRetryJoin` | bool | 啟用 join 重試機制 |
| `diskSetup` | DiskSetup | 磁碟初始化設定 |
| `mounts` | []MountPoints | 掛載點設定 |

### Status 欄位

| 欄位 | 說明 |
|------|------|
| `ready` | bootstrap data 是否已生成 |
| `dataSecretName` | 儲存 bootstrap data 的 secret 名稱 |
| `conditions` | 健康狀態條件 |
| `observedGeneration` | 最後觀察到的 generation |

## KubeadmConfigTemplate

`KubeadmConfigTemplate` 是 KubeadmConfig 的模板形式，供 `MachineDeployment` 或 `MachineSet` 使用。每當 MachineDeployment 建立一台新的 Machine，Controller 就會從模板建立一個對應的 KubeadmConfig 物件。

```yaml
apiVersion: bootstrap.cluster.x-k8s.io/v1beta1
kind: KubeadmConfigTemplate
metadata:
  name: my-cluster-workers
  namespace: default
spec:
  template:
    spec:
      joinConfiguration:
        nodeRegistration:
          name: '{{ ds.meta_data.local_hostname }}'
          kubeletExtraArgs:
            cloud-provider: external
            node-labels: "node-role.kubernetes.io/worker="
      preKubeadmCommands:
        - systemctl enable --now containerd
      files:
        - path: /etc/containerd/config.toml
          content: |
            version = 2
            [plugins."io.containerd.grpc.v1.cri"]
              sandbox_image = "registry.k8s.io/pause:3.9"
          owner: root:root
          permissions: "0644"
```

## Files 欄位詳細說明

`files` 欄位允許在節點啟動時建立任意檔案，常用於注入憑證、設定檔或腳本：

```yaml
files:
  - path: /etc/kubernetes/pki/ca.crt
    content: |
      -----BEGIN CERTIFICATE-----
      MIIBeTCCAR+gAwIBAgIRAKmluck...
      -----END CERTIFICATE-----
    owner: root:root
    permissions: "0640"
    encoding: base64   # 可選：base64, gzip+base64

  - path: /usr/local/bin/health-check.sh
    content: |
      #!/bin/bash
      curl -sf http://localhost:10248/healthz || exit 1
    owner: root:root
    permissions: "0755"
```

**File 子欄位說明：**

| 欄位 | 說明 |
|------|------|
| `path` | 目標路徑（絕對路徑）|
| `content` | 檔案內容 |
| `owner` | 擁有者（格式：`user:group`）|
| `permissions` | 權限（八進位字串，如 `"0644"`）|
| `encoding` | 內容編碼：`""`、`base64`、`gzip`、`gzip+base64` |
| `append` | 是否追加到現有檔案（而非覆蓋）|

## Users 欄位 — SSH 金鑰注入

透過 `users` 欄位可在節點建立時注入系統使用者及 SSH public key，讓運維人員能夠直接登入節點：

```yaml
users:
  - name: ubuntu
    sudo: "ALL=(ALL) NOPASSWD:ALL"
    groups: "users, admin"
    sshAuthorizedKeys:
      - "ssh-rsa AAAA...your-public-key... user@laptop"
    shell: /bin/bash
    passwd: ""
    lockPassword: true
```


<Callout type="tip" title="安全建議">

建議搭配 **cert-manager** 或 **Sealed Secrets** 管理 SSH key，避免直接在 YAML 中明文儲存公開金鑰或敏感資訊。將 SSH key 以 SealedSecret 加密後提交版控，可同時兼顧安全性與 GitOps 可審計性。

</Callout>

## NTP 設定

```yaml
ntp:
  servers:
    - 0.pool.ntp.org
    - 1.pool.ntp.org
  enabled: true
```


<Callout type="warning" title="裸機環境特別注意">

在裸機環境中，NTP 設定尤為重要。時鐘偏差（clock skew）會導致：
- **etcd 叢集不穩定**：etcd 採用 Raft 一致性算法，對時間敏感
- **憑證驗證失敗**：TLS 憑證的有效期間驗證依賴系統時鐘
- **Kubernetes Token 過期**：bootstrap token 有時效限制

請確保所有節點在加入叢集前已正確同步時間。

</Callout>

## Bootstrap Data 生成流程

CABPK Controller 負責監聽 KubeadmConfig 物件，並依序完成以下步驟：

### Step 1 — 偵測 Machine 需要 bootstrap

- Machine 的 `spec.bootstrap.configRef` 指向 KubeadmConfig
- CABPK Controller watch 所有 KubeadmConfig 物件，偵測到新建或未就緒的物件後開始處理

<Callout type="tip" title="新手提示：Bootstrap Data 是什麼格式？">
Bootstrap Data 本質上是一段「開機腳本」，格式通常是 **cloud-init**（cloud-config 格式）。當 Infrastructure Provider 建立一台新 VM 或裸機時，會把這段腳本作為 `user-data` 注入機器。機器第一次開機時，cloud-init 程式會自動執行這段腳本，安裝 containerd、kubelet，並執行 `kubeadm init` 或 `kubeadm join` 將節點加入 Kubernetes 叢集。整個過程完全自動，無需人工介入。
</Callout>

### Step 2 — 等待必要資訊就緒

- 等待 `Cluster.Status.InfrastructureReady = true`
- 等待 KubeadmConfig 對應的 Cluster 資源完整建立
- 若目標節點為 Control Plane，額外等待 certificates secret（含 CA、etcd CA 等）就緒

### Step 3 — 生成 cloud-init 資料

根據節點類型產生不同的初始化指令：

- **Control Plane init（第一台 CP）**：包含 `kubeadm init` 命令、叢集憑證、以及 ClusterConfiguration
- **Worker / CP join（後續節點）**：包含 `kubeadm join` 命令和有時效的 bootstrap token

Controller 將所有 `files`、`preKubeadmCommands`、`postKubeadmCommands`、`users`、`ntp` 等設定組合成完整的 cloud-init user-data。

### Step 4 — 建立 Secret

- 將 cloud-init 資料存入 Secret，名稱為 `<machine-name>-bootstrap-data`
- 設定 `KubeadmConfig.Status.Ready = true`
- 設定 `KubeadmConfig.Status.DataSecretName` 指向該 Secret

### Step 5 — Machine Controller 讀取 Secret

- Machine Controller 偵測到 `bootstrap.ready = true`
- 從 `dataSecretName` 讀取 bootstrap Secret
- 將 user-data 傳遞給 Infrastructure Provider
- Infrastructure Provider 在建立機器時注入 user-data（例如：IPMI virtual media、cloud-init drive 或 ISO 注入）

## Ignition 格式支援

除了預設的 cloud-init（`cloud-config`）格式，CABPK 也支援 **Ignition** 格式，主要用於 **Fedora CoreOS** 或 **Flatcar Container Linux** 等不可變作業系統。

啟用方式是在 KubeadmConfig 中設定：

```yaml
spec:
  format: ignition
```

使用 Ignition 格式時需注意：

- `files` 欄位的語法與 cloud-init 略有不同，部分子欄位（如 `encoding`）的處理方式有差異
- Ignition 的 `users` 和 `ntp` 設定也會轉換為對應的 Ignition 規格
- 在 Metal3 環境中，BMH（BareMetalHost）支援透過 live-ISO 或 virtual media 方式注入 Ignition 設定，適合無狀態、不可變的裸機節點管理場景

## 常見誤解

<Callout type="warning" title="常見誤解澄清">
1. **「Bootstrap Data 就是 kubeadm 的設定檔」** — 不完全正確。Bootstrap Data 是一個完整的「開機初始化腳本」，通常以 cloud-init 格式呈現，其中**包含** kubeadm 設定（`kubeadm-config.yaml`），但還包含安裝 containerd、kubelet、設定 NTP、建立使用者等許多其他步驟，遠比單純的 kubeadm 設定豐富。
2. **「KubeadmConfig 和 KubeadmConfigTemplate 功能完全相同」** — 有差別。`KubeadmConfig` 是一次性物件（1 對 1 對應一台 Machine），`KubeadmConfigTemplate` 是可重用的範本（1 對多），由 MachineSet 或 MachineDeployment 管理，每次建立新 Machine 時都會從 Template 複製出一個新的 KubeadmConfig。
3. **「Bootstrap Provider 可以省略」** — 不正確。每台 Machine 的節點初始化都需要 Bootstrap Data。若不使用 CABPK，也必須提供其他 Bootstrap Provider（或自定義機制），否則節點永遠無法加入叢集。
</Callout>
