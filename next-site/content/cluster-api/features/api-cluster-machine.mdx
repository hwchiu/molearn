# Cluster 與 Machine 核心資源

本頁深入解析 Cluster API 兩個最基礎的資源：**Cluster** 與 **Machine**。這兩個資源是整個 CAPI 架構的骨幹，其他所有資源（MachineDeployment、KubeadmControlPlane 等）皆圍繞它們運作。

## Cluster 資源

### 概念說明

`Cluster` 是 CAPI 的頂層資源，代表一個完整的 Kubernetes cluster。它本身不直接管理基礎設施或 Control Plane，而是透過兩個引用（Reference）將工作委派給對應的 Provider：

- **`infrastructureRef`** → 指向 Infrastructure Provider 資源（如 `Metal3Cluster`、`AWSCluster`）
- **`controlPlaneRef`** → 指向 Control Plane Provider 資源（如 `KubeadmControlPlane`）

這種設計讓 CAPI 核心邏輯與底層實作完全解耦，只要 Provider 遵守 CAPI 合約（實作 `status.ready` 等欄位），就能無縫整合。

### ClusterSpec 欄位詳解

| 欄位 | 類型 | 說明 |
|------|------|------|
| `paused` | bool | 暫停所有 reconcile 操作 |
| `clusterNetwork.pods.cidrBlocks` | []string | Pod CIDR 範圍 |
| `clusterNetwork.services.cidrBlocks` | []string | Service CIDR 範圍 |
| `clusterNetwork.serviceDomain` | string | Service DNS domain（預設 cluster.local）|
| `controlPlaneEndpoint.host` | string | API server 的 IP 或 hostname |
| `controlPlaneEndpoint.port` | int | API server port（預設 6443）|
| `controlPlaneRef` | ObjectReference | ControlPlane Provider 資源引用（例如 KubeadmControlPlane）|
| `infrastructureRef` | ObjectReference | Infrastructure Provider 資源引用（例如 Metal3Cluster）|
| `topology` | Topology | ClusterClass 拓撲設定（ClusterTopology feature gate）|

### 原始碼定義

以下為 `ClusterSpec` 在原始碼中的實際 struct 定義（已略去 kubebuilder 標注），可以看出幾個關鍵設計選擇：

- `ControlPlaneRef` 和 `InfrastructureRef` 均為 **指標型別**（`*corev1.ObjectReference`），允許為 `nil`，讓cluster在無 ControlPlane Provider 的情況下也能建立（例如匯入已存在的cluster）。
- `Topology` 亦為指標，代表 ClusterClass 拓撲功能是**完全選擇性**的，不啟用時為 nil。
- `AvailabilityGates` 機制讓第三方 Controller 能夠擴充cluster的「可用性」判斷條件，而不需要修改核心程式碼。

```go
// 原始碼: api/core/v1beta1/cluster_types.go
type ClusterSpec struct {
    // paused 用於暫停所有 Controller 對此 Cluster 的 reconcile。
    Paused bool `json:"paused,omitempty"`

    // clusterNetwork 代表cluster的網路設定（Pod/Service CIDR 等）。
    ClusterNetwork *ClusterNetwork `json:"clusterNetwork,omitempty"`

    // controlPlaneEndpoint 為 API Server 的對外端點（host:port）。
    ControlPlaneEndpoint APIEndpoint `json:"controlPlaneEndpoint,omitempty"`

    // controlPlaneRef 指向 ControlPlane Provider 資源（例如 KubeadmControlPlane）。
    // 使用指標型別，允許 nil（適用於匯入既有cluster的情境）。
    ControlPlaneRef *corev1.ObjectReference `json:"controlPlaneRef,omitempty"`

    // infrastructureRef 指向 Infrastructure Provider 資源（例如 Metal3Cluster）。
    // 同樣為指標型別，解耦核心邏輯與底層基礎設施實作。
    InfrastructureRef *corev1.ObjectReference `json:"infrastructureRef,omitempty"`

    // topology 為 ClusterClass 拓撲設定。
    // 需啟用 ClusterTopology feature gate 才有作用；未使用時為 nil。
    Topology *Topology `json:"topology,omitempty"`

    // availabilityGates 可讓第三方 Controller 注入額外的可用性判斷條件。
    AvailabilityGates []ClusterAvailabilityGate `json:"availabilityGates,omitempty"`
}
```

### ClusterStatus 欄位

| 欄位 | 類型 | 說明 |
|------|------|------|
| `phase` | string | cluster目前 phase（見下方狀態說明）|
| `infrastructureReady` | bool | InfraCluster.status.ready = true |
| `controlPlaneReady` | bool | ControlPlane.status.ready = true |
| `controlPlaneInitialized` | bool | 第一個 CP node 已加入cluster |
| `conditions` | Conditions | 目前服務狀態 |
| `failureDomains` | FailureDomains | 從 InfraCluster 取得的 failure domain 清單 |
| `observedGeneration` | int64 | 最後 reconcile 的 generation |

### 原始碼定義

`ClusterStatus` 記錄了 Controller 觀察到的cluster實際狀態。值得注意的設計細節：

- `ControlPlaneReady` 的注解說明它**只在初始 provisioning 時更新**，cluster穩定運行後要查看 ControlPlane 狀態應使用 `conditions`，而非這個布林值。
- `V1Beta2` 欄位封裝了下一版 API 將引入的新狀態欄位，以子結構方式新增可避免破壞舊版 API 合約。
- `FailureReason` / `FailureMessage` 在原始碼中已被標記為 **Deprecated**，未來版本將移除。

```go
// 原始碼: api/core/v1beta1/cluster_types.go
type ClusterStatus struct {
    // failureDomains 從 InfraCluster 同步的 failure domain 清單。
    FailureDomains FailureDomains `json:"failureDomains,omitempty"`

    // phase 代表cluster目前的 phase。
    // +kubebuilder:validation:Enum=Pending;Provisioning;Provisioned;Deleting;Failed;Unknown
    Phase string `json:"phase,omitempty"`

    // infrastructureReady 代表 InfraCluster 的就緒狀態。
    InfrastructureReady bool `json:"infrastructureReady"`

    // controlPlaneReady 代表 ControlPlane 在初始 provisioning 時是否就緒。
    // 注意：此欄位在 provisioning 完成後不再更新，
    // 運行期間請改用 conditions 判斷 ControlPlane 狀態。
    ControlPlaneReady bool `json:"controlPlaneReady"`

    // conditions 定義目前cluster的服務狀態（v1beta1 Conditions）。
    Conditions Conditions `json:"conditions,omitempty"`

    // observedGeneration 是 Controller 最後一次 reconcile 的 generation。
    ObservedGeneration int64 `json:"observedGeneration,omitempty"`

    // v1beta2 封裝未來 V1Beta2 API 版本中新增或修改的 status 欄位。
    // 以子結構方式新增，確保不破壞現有 v1beta1 API 合約。
    V1Beta2 *ClusterV1Beta2Status `json:"v1beta2,omitempty"`
}
```

### Cluster 生命週期 Phase

| Phase | 說明 |
|-------|------|
| `Pending` | cluster剛建立，等待 InfraCluster/ControlPlane 初始化 |
| `Provisioning` | 基礎設施正在建立 |
| `Provisioned` | 所有基礎設施就緒，cluster正常運行 |
| `Deleting` | cluster正在被刪除 |
| `Failed` | cluster進入不可恢復的錯誤狀態 |
| `Unknown` | cluster狀態未知 |

### Cluster 與 Provider 的關聯方式

以下 YAML 展示 Cluster 如何同時引用 Infrastructure Provider（Metal3Cluster）和 Control Plane Provider（KubeadmControlPlane）：

```yaml
apiVersion: cluster.x-k8s.io/v1beta1
kind: Cluster
metadata:
  name: my-cluster
  namespace: default
spec:
  clusterNetwork:
    pods:
      cidrBlocks: ["10.244.0.0/16"]
    services:
      cidrBlocks: ["10.96.0.0/12"]
    serviceDomain: "cluster.local"
  controlPlaneEndpoint:
    host: 192.168.1.100
    port: 6443
  infrastructureRef:
    apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
    kind: Metal3Cluster           # InfraCluster
    name: my-metal3-cluster
  controlPlaneRef:
    apiVersion: controlplane.cluster.x-k8s.io/v1beta1
    kind: KubeadmControlPlane     # CP Provider
    name: my-cluster-cp
```

## Machine 資源

### 概念說明

`Machine` 代表cluster中的一個node（Control Plane 或 Worker）。在實際操作中，**通常不直接手動建立 Machine**，而是透過以下方式間接管理：

- **Worker node** → 透過 `MachineDeployment` → `MachineSet` 自動建立
- **Control Plane node** → 透過 `KubeadmControlPlane` 自動建立

Machine 同樣採用 Provider 引用模式：`infrastructureRef` 指向 InfraMachine（如 `Metal3Machine`），`bootstrap.configRef` 指向 Bootstrap Provider（如 `KubeadmConfig`）。

### MachineSpec 欄位詳解

| 欄位 | 類型 | 必填 | 說明 |
|------|------|:----:|------|
| `clusterName` | string | ✅ | 所屬cluster名稱（建立後不可變）|
| `bootstrap.configRef` | ObjectReference | ❌* | Bootstrap Provider 資源引用（例如 KubeadmConfig）|
| `bootstrap.dataSecretName` | string | ❌* | 直接指定 bootstrap data secret 名稱 |
| `infrastructureRef` | ObjectReference | ✅ | InfraMachine 資源引用（例如 Metal3Machine）|
| `version` | string | ❌ | Kubernetes 版本（例如 v1.28.0，webhook 自動加 v 前綴）|
| `providerID` | string | ❌（唯讀）| 由 InfraMachine controller 設定 |
| `failureDomain` | string | ❌ | 指定部署的 failure domain |
| `nodeDrainTimeout` | Duration | ❌ | Node drain 超時（預設無限）|
| `nodeVolumeDetachTimeout` | Duration | ❌ | Volume detach 超時（預設無限）|
| `nodeDeletionTimeout` | Duration | ❌ | Node 刪除超時（預設 10s）|
| `readinessGates` | []MachineReadinessGate | ❌ | 額外的就緒條件 |

> \* `bootstrap.configRef` 和 `bootstrap.dataSecretName` 至少需要提供一個。

### 原始碼定義

以下為 `MachineSpec` 的實際 struct 定義，幾個值得深入了解的設計：

- `Bootstrap` 欄位型別為值型別（非指標），表示 Bootstrap 設定是**必填的**，但其內部的 `configRef` 和 `dataSecretName` 都是 optional，讓 Webhook 在建立時驗證「至少提供其中一個」。
- `ProviderID` 由 InfraMachine Controller 設定，不應由使用者手動填寫；Cluster Autoscaler 等工具依賴此欄位對應實體機器。
- 三個 Timeout 欄位（`NodeDrainTimeout`、`NodeVolumeDetachTimeout`、`NodeDeletionTimeout`）均為 `*metav1.Duration`，預設值各不相同（drain/detach 預設無限，deletion 預設 10s）。
- `Taints` 欄位使用 `+listType=map`，允許不同 owner 分別管理不同的 taint，不互相覆蓋。

```go
// 原始碼: api/core/v1beta1/machine_types.go
type MachineSpec struct {
    // clusterName 是此 Machine 所屬cluster的名稱，建立後不可更改。
    ClusterName string `json:"clusterName"`

    // bootstrap 封裝 Bootstrap Provider 的設定。
    // 內部 configRef 或 dataSecretName 至少需提供一個（Webhook 驗證）。
    Bootstrap Bootstrap `json:"bootstrap"`

    // infrastructureRef 指向 InfraMachine 資源（例如 Metal3Machine）。
    InfrastructureRef corev1.ObjectReference `json:"infrastructureRef"`

    // version 定義此node期望的 Kubernetes 版本（例如 v1.28.0）。
    Version *string `json:"version,omitempty"`

    // providerID 由 InfraMachine Controller 填寫，對應實體/虛擬機器的唯一識別碼。
    // Cluster Autoscaler 等工具依賴此欄位比對node與機器的對應關係。
    ProviderID *string `json:"providerID,omitempty"`

    // failureDomain 指定部署的 failure domain，必須對應 Cluster.status.failureDomains。
    FailureDomain *string `json:"failureDomain,omitempty"`

    // nodeDrainTimeout 為 drain node的超時時間，預設 0（無限制）。
    NodeDrainTimeout *metav1.Duration `json:"nodeDrainTimeout,omitempty"`

    // nodeVolumeDetachTimeout 為等待 Volume detach 的超時時間，預設 0（無限制）。
    NodeVolumeDetachTimeout *metav1.Duration `json:"nodeVolumeDetachTimeout,omitempty"`

    // nodeDeletionTimeout 為嘗試刪除 Node 物件的超時時間，預設 10 秒。
    NodeDeletionTimeout *metav1.Duration `json:"nodeDeletionTimeout,omitempty"`

    // taints 是 CAPI 負責管理的 node taint 清單（最多 64 個）。
    // +listType=map 允許不同 owner 各自管理特定 taint，不互相覆蓋。
    Taints []MachineTaint `json:"taints,omitempty"`
}
```

### MachineStatus 欄位

| 欄位 | 類型 | 說明 |
|------|------|------|
| `phase` | string | Machine 目前 phase |
| `bootstrapReady` | bool | Bootstrap data 已就緒 |
| `infrastructureReady` | bool | InfraMachine.status.ready = true |
| `addresses` | []MachineAddress | node IP 地址（來自 InfraMachine）|
| `nodeRef` | ObjectReference | 對應的 K8s Node 物件引用 |
| `conditions` | Conditions | 目前服務狀態 |
| `deletion.waitForNodeVolumeDetach` | bool | 是否正在等待 volume detach |

<Callout type="tip" title="新手提示：為何不直接建立 Machine？">
在實際操作中，我們幾乎不會手動建立 Machine 物件。Worker node透過 MachineDeployment 統一管理（支援自動擴縮容和rolling update），Control Plane node透過 KubeadmControlPlane 管理（確保 etcd quorum）。直接建立 Machine 就像直接建立 Pod 而不使用 Deployment — 雖然可行，但沒有自癒和版本管理能力。
</Callout>

### Machine 生命週期 Phase

| Phase | 說明 | 觸發條件 |
|-------|------|---------|
| `Pending` | Machine 剛建立 | 初始狀態 |
| `Provisioning` | Bootstrap 或 Infrastructure 尚未就緒 | bootstrapReady=false 或 infraReady=false |
| `Provisioned` | Bootstrap + Infrastructure 就緒，等待 Node 加入 | bootstrapReady=true, infraReady=true |
| `Running` | Node 已加入cluster（nodeRef 已設定）| nodeRef != nil |
| `Deleting` | Machine 被標記刪除 | DeletionTimestamp != nil |
| `Deleted` | Machine 已刪除完成 | 終止狀態 |
| `Failed` | 不可恢復錯誤 | failureReason/Message 設定 |
| `Unknown` | 狀態未知 | 無法確定狀態 |

### 常用 Labels 與 Annotations

| Label / Annotation | 說明 |
|--------------------|------|
| `cluster.x-k8s.io/cluster-name` | 機器所屬cluster名稱（自動設定）|
| `cluster.x-k8s.io/control-plane` | 標記此 Machine 為 Control Plane node |
| `cluster.x-k8s.io/paused` | 暫停此 Machine 的 reconcile |
| `cluster.x-k8s.io/delete-machine` | 標記此 Machine 優先被刪除（MachineSet 縮容時使用）|
| `machine.cluster.x-k8s.io/exclude-node-draining` | 刪除時跳過 node drain |
| `cluster.x-k8s.io/remediate-machine` | 觸發 MachineHealthCheck remediation |
| `machine.cluster.x-k8s.io/certificates-expiry` | 憑證到期時間（ISO 8601 格式）|
| `topology.cluster.x-k8s.io/owned` | 標記此 Machine 被 Topology Controller 管理 |

## Webhook 驗證限制

CAPI 透過 Admission Webhook 對 Machine 資源進行驗證與預設值填充（defaulting）：

| 欄位 | 規則 | 說明 |
|------|------|------|
| `spec.clusterName` | Immutable | 機器不能在cluster間移動 |
| `spec.version` | 自動加 v 前綴 | `1.28.0` → `v1.28.0` |
| `spec.nodeDeletionTimeout` | 預設 10s | webhook defaulting 自動填充 |
| `spec.bootstrap` | 至少一個 | `configRef` 或 `dataSecretName` 必填一個 |

## Machine 刪除流程概要

Machine 刪除涉及多個步驟，以確保node安全退出cluster：

1. 設定 `DeletionTimestamp` → phase 進入 `Deleting`
2. 執行 **pre-drain hooks**（若有設定）
3. **Cordon Node**（標記unschedulable）
4. **Drain Node**（驅逐 Pod，PDB-aware）
5. 等待 `VolumeAttachments` detach
6. 執行 **pre-terminate hooks**（若有設定）
7. 刪除 Bootstrap 與 InfraMachine 資源
8. 移除 finalizer，完成刪除

### 原始碼：Finalizer 定義

Finalizer 是 Kubernetes 物件刪除保護機制的核心。在 CAPI 中，Cluster 和 Machine 各有自己的 Finalizer 字串常數，Controller 在 reconcile 時會確認這些 Finalizer 存在，並在完成清理工作後才移除它們。

以下為原始碼中的實際常數定義：

```go
// 原始碼: api/core/v1beta1/cluster_types.go
const (
    // ClusterFinalizer 由 Cluster Controller 設置，
    // 確保 Cluster 刪除前完成所有基礎設施與 Machine 資源的清理。
    ClusterFinalizer = "cluster.cluster.x-k8s.io"
)
```

```go
// 原始碼: api/core/v1beta1/machine_types.go
const (
    // MachineFinalizer 在 PrepareForCreate 回呼時設置，
    // 確保 Machine 刪除前完成 drain、volume detach 及 InfraMachine 刪除流程。
    MachineFinalizer = "machine.cluster.x-k8s.io"
)
```

**設計意義**：這兩個 Finalizer 決定了刪除順序的安全性 — 若直接強制移除 Finalizer（`kubectl patch ... -p '{"metadata":{"finalizers":[]}}'`），將跳過上述所有保護步驟，可能導致孤立的基礎設施資源或服務中斷。


<Callout type="info" title="詳細刪除流程">

完整的 Machine 刪除流程（包含 hook 機制、PDB 處理、逾時邏輯）請參閱：[Machine 生命週期](/cluster-api/features/machine-lifecycle)

</Callout>

## 常見誤解

<Callout type="warning" title="常見誤解澄清">
1. **「Cluster 物件建立後就代表cluster已可用」** — 錯誤。Cluster 物件只是「期望狀態」的宣告，實際的 Control Plane 和 Worker node要等 Infrastructure Provider 完成機器建立、Bootstrap Provider 完成node初始化後，Cluster 的 `status.phase` 才會進入 `Provisioned`。
2. **「直接刪除 Machine 物件即可快速移除node」** — 危險做法。CAPI 設計了完整的安全刪除流程（Cordon → Drain → Volume Detach），直接刪除物件雖然可行，但會跳過 Pod 驅逐步驟，可能導致服務中斷或有狀態應用的資料遺失。
3. **「Machine labels 可以任意修改」** — 有限制。部分 label（如 `cluster.x-k8s.io/cluster-name`）若被修改，會影響 Controller 的 selector 邏輯，導致該 Machine 被視為「孤立」（orphaned）而不再受 MachineSet 管控。
</Callout>
