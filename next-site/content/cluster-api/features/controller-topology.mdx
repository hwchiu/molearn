## Topology Controller 概覽

`TopologyReconciler` 是 Cluster API 中負責 **ClusterClass 功能**的核心控制器，由 Feature Gate `ClusterTopology`（Beta，預設開啟）控制。

其主要職責是將 `Cluster.Spec.Topology` 中的**宣告式設定**，轉換為實際的 CAPI 資源，包括：

- **InfraCluster**：底層基礎設施叢集物件
- **KubeadmControlPlane（KCP）**：控制平面
- **MachineDeployment**：工作節點群組
- 各種對應的 **Template** 資源

除了資源建立，TopologyReconciler 還負責：
- 處理 **ClusterClass 的版本升級**（透過修改 `spec.topology.version`）
- 進行 **Variables 替換**，將使用者提供的變數值注入各 template

## Topology Reconcile 主流程

### Step 1 — 讀取 ClusterClass

控制器從 `Cluster.Spec.Topology.Class` 取得 ClusterClass 名稱，並讀取其完整定義，包含：

- `spec.infrastructure`：InfraCluster Template 的參照
- `spec.controlPlane`：KCP Template 的參照（含 machine infrastructure template）
- `spec.workers.machineDeployments`：各 worker class 的 template 定義
- `spec.variables`：可用變數清單（含預設值）
- `spec.patches`：patch 規則清單

### Step 2 — 處理 Variables

控制器讀取 `Cluster.Spec.Topology.Variables`，並執行以下操作：

1. 將使用者提供的變數與 ClusterClass `spec.variables` 的預設值**合併**
2. **驗證**所有標記為 `required: true` 的變數都有提供值
3. 驗證每個變數值符合 `schema.openAPIV3Schema` 中定義的型別和限制

如果驗證失敗，Reconcile 會中止並回報錯誤，Cluster 狀態會反映驗證失敗的原因。

### Step 3 — Apply Patches

Patch 是 ClusterTopology 最強大的功能之一，控制器會依序執行：

1. **套用 ClusterClass 中定義的 patches**（按順序）
2. 每個 patch 的 `selector` 決定要修改哪些 template
3. 將 variable 值透過 `valueFrom.variable` 或 `valueFrom.template` 替換進去
4. 結果是一組具體化的 template：
   - `InfraClusterTemplate`（已填入所有變數值）
   - `KCPTemplate`（已填入所有變數值）
   - 各 `MachineDeploymentTemplate`（已填入所有變數值）

<Callout type="tip" title="新手提示：Patches 如何把參數「注入」到 Template？">
想像 ClusterClass 的 Template 就像有預留空格的合約範本，`patches` 就是「填空規則」。你在 `spec.topology.variables` 填入的值（例如 `imageURL: "https://example.com/ubuntu.img"`），Topology Controller 會按照 patches 定義的路徑（例如 `/spec/template/spec/image/url`），把這個值精確填入對應 Template 的那個欄位。整個過程在 Controller 記憶體中完成，不會直接修改 ClusterClass 或 Template 物件，而是產生一個「具體化版本」再套用到叢集。
</Callout>

### Step 4 — 建立或更新資源

控制器對比**目前存在的資源**與**期望狀態**：

| 情況 | 操作 |
|------|------|
| 資源不存在 | 建立新資源（從具體化 template 產生） |
| 資源存在但設定過時 | 更新資源設定 |
| 資源存在且為最新 | 不做任何變動 |

對於不可變動的欄位（如某些 InfraCluster 欄位），控制器會建立**新的 template 物件**並更新參照，而非直接修改原物件。

### Step 5 — 更新 Cluster Status

Reconcile 結束後，控制器更新 `Cluster.Status.Topology`，反映目前的版本和 variable 狀態，供其他控制器或使用者查看。

## ClusterClass 結構說明

ClusterClass 是一個叢集藍圖，定義了建立叢集所需的所有 template 和規則。以下是完整範例：

```yaml
apiVersion: cluster.x-k8s.io/v1beta1
kind: ClusterClass
metadata:
  name: quick-start
  namespace: default
spec:
  infrastructure:
    ref:
      apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
      kind: Metal3ClusterTemplate
      name: quick-start-cluster
  controlPlane:
    ref:
      apiVersion: controlplane.cluster.x-k8s.io/v1beta1
      kind: KubeadmControlPlaneTemplate
      name: quick-start-control-plane
    machineInfrastructure:
      ref:
        apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
        kind: Metal3MachineTemplate
        name: quick-start-control-plane-machine
  workers:
    machineDeployments:
      - class: default-worker
        template:
          bootstrap:
            ref:
              apiVersion: bootstrap.cluster.x-k8s.io/v1beta1
              kind: KubeadmConfigTemplate
              name: quick-start-worker-bootstrap
          infrastructure:
            ref:
              apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
              kind: Metal3MachineTemplate
              name: quick-start-worker-machine
  variables:
    - name: controlPlaneMachineImage
      required: true
      schema:
        openAPIV3Schema:
          type: string
          description: "Control Plane 節點使用的 OS image URL"
    - name: workerReplicas
      required: false
      schema:
        openAPIV3Schema:
          type: integer
          default: 3
          minimum: 1
  patches:
    - name: controlPlaneMachineImagePatch
      definitions:
        - selector:
            apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
            kind: Metal3MachineTemplate
            matchResources:
              controlPlane: true
          jsonPatches:
            - op: replace
              path: /spec/template/spec/image/url
              valueFrom:
                variable: controlPlaneMachineImage
```

**各段說明：**

- **`spec.infrastructure`**：指向 InfraCluster 的 Template，例如 `Metal3ClusterTemplate`
- **`spec.controlPlane`**：指向 KCP 的 Template，以及 Control Plane Machine 的 InfraTemplate
- **`spec.workers.machineDeployments`**：定義每個 worker class，每個 class 有自己的 bootstrap 和 infrastructure template
- **`spec.variables`**：宣告可注入的變數，支援型別驗證（`openAPIV3Schema`）和預設值
- **`spec.patches`**：定義如何將 variable 值注入 template，支援精確的 JSON Patch 操作

## Cluster with Topology 範例

使用 ClusterClass 建立叢集時，Cluster 物件透過 `spec.topology` 區塊引用 ClusterClass：

```yaml
apiVersion: cluster.x-k8s.io/v1beta1
kind: Cluster
metadata:
  name: my-cluster
  namespace: default
spec:
  topology:
    class: quick-start           # 引用 ClusterClass
    version: v1.28.0
    controlPlane:
      replicas: 3
    workers:
      machineDeployments:
        - class: default-worker
          name: worker-pool-1
          replicas: 3
    variables:
      - name: controlPlaneMachineImage
        value: "https://images.example.com/ubuntu-22.04.tar.gz"
      - name: workerReplicas
        value: 3
```

**關鍵欄位說明：**

| 欄位 | 說明 |
|------|------|
| `spec.topology.class` | 引用的 ClusterClass 名稱（必填） |
| `spec.topology.version` | Kubernetes 版本，用於 KCP 和 MachineDeployment |
| `spec.topology.controlPlane.replicas` | Control Plane 副本數 |
| `spec.topology.workers.machineDeployments` | 各 worker pool 設定（需與 ClusterClass 中的 class 名稱對應） |
| `spec.topology.variables` | 覆蓋或提供 ClusterClass 定義的變數值 |

使用 Topology 模式時，**不需要**在 Cluster 物件上設定 `spec.infrastructureRef` 或 `spec.controlPlaneRef`，這些參照由 TopologyReconciler 自動建立和管理。

## Topology 版本升級

ClusterTopology 大幅簡化了版本升級流程：

**升級步驟：**

1. 修改 `spec.topology.version: v1.29.0`
2. TopologyReconciler 偵測到版本變更
3. 更新 `KubeadmControlPlane.Spec.Version`，觸發 KCP rolling upgrade
4. Control Plane 升級完成後，依序更新各 `MachineDeployment.Spec.Template.Spec.Version`


<Callout type="tip" title="使用 ClusterTopology 升級的優勢">

使用 ClusterTopology 的好處是只需修改一個欄位（`spec.topology.version`），所有 Control Plane 和 Worker 節點會自動按正確順序升級。無需手動協調多個資源的更新順序，大幅降低操作風險。

</Callout>


**升級執行指令：**

```bash
# 透過 kubectl patch 觸發版本升級
kubectl patch cluster my-cluster \
  --type=merge \
  -p '{"spec":{"topology":{"version":"v1.29.0"}}}'
```

TopologyReconciler 會確保升級遵循 CAPI 的升級策略（先 Control Plane，後 Workers），與手動升級行為一致。

## Patches 深入說明

Patches 允許 ClusterClass 的使用者透過 variables 客製化 template 內容。

### 1. JSON Patch（RFC 6902）

最常用的 patch 類型，支援 `add`、`replace`、`remove` 操作：

```yaml
jsonPatches:
  - op: add
    path: /spec/template/spec/kubeadmConfigSpec/clusterConfiguration/apiServer/extraArgs
    value:
      audit-log-path: /var/log/audit.log
  - op: replace
    path: /spec/template/spec/image/url
    valueFrom:
      variable: imageURL
```

### 2. JSON Merge Patch

適合需要合併整個物件的情境（整個物件合併，不是逐欄位操作）：

```yaml
patches:
  - name: mergeExtraLabels
    definitions:
      - selector:
          apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
          kind: Metal3MachineTemplate
          matchResources:
            machineDeploymentClass:
              names:
                - default-worker
        mergePatch: |
          spec:
            template:
              spec:
                metaData:
                  labels:
                    env: production
```

### 3. Variable 引用方式

Patch 可以透過兩種方式引用 variable 值：

| 引用方式 | 說明 | 範例 |
|----------|------|------|
| `valueFrom.variable` | 直接使用 variable 的值 | `valueFrom: { variable: imageURL }` |
| `valueFrom.template` | 使用 Go template 處理（支援字串拼接、條件判斷等） | `valueFrom: { template: "prefix-&#123;&#123; .imageURL &#125;&#125;-suffix" }` |

`valueFrom.template` 特別適合需要**字串拼接**或**動態產生值**的情況，例如將 variable 值作為 URL 的一部分。

## Runtime SDK（進階）


<Callout type="info" title="Runtime SDK — Alpha 功能（Feature Gate: RuntimeSDK，預設關閉）">

**Runtime SDK** 允許外部 webhook（稱為 **Runtime Extension**）參與 CAPI 的生命週期事件。

**支援的 Hooks：**

| Hook 名稱 | 觸發時機 |
|-----------|---------|
| `BeforeClusterCreate` | 建立叢集資源前 |
| `BeforeClusterUpgrade` | 升級叢集版本前 |
| `AfterControlPlaneInitialized` | Control Plane 初始化完成後 |
| `AfterClusterUpgrade` | 升級完成後 |
| `BeforeClusterDelete` | 刪除叢集資源前 |

**Topology Controller 在適當時機呼叫這些 hooks**，等待外部系統確認後才繼續執行。

**常見用途：**
- **Custom admission logic**：在建立或升級前進行額外驗證
- **External dependency management**：等待外部系統就緒（如 DNS、憑證）
- **Audit logging**：記錄叢集生命週期事件到外部系統

若要啟用，需設定 Feature Gate `--feature-gates=RuntimeSDK=true` 並部署對應的 Runtime Extension webhook。

</Callout>

> 💡 **思考一下**：Topology Controller 在 Apply Patches 時，需要把 ClusterClass 的 Template 「具體化」為實際資源。如果兩個叢集使用同一個 ClusterClass 但設定了不同的 variables，Topology Controller 如何確保它們各自產生獨立的 Template 實例，而不會互相干擾？

## 常見誤解

<Callout type="warning" title="常見誤解澄清">
1. **「Topology Controller 修改了 ClusterClass 的 Template 物件」** — 錯誤。Topology Controller 會為每個叢集建立獨立的「owned Template 複本」（透過 ownerReference 標記），並對這份複本套用 patches，而不是直接修改 ClusterClass 中的原始 Template。因此多個叢集共用同一個 ClusterClass 時，各自的 Template 是完全獨立的複本。
2. **「版本升級時 Topology Controller 會直接修改所有 Machine 的版本」** — 不正確。Topology Controller 只更新 KubeadmControlPlane 和 MachineDeployment 的 `version` 欄位，實際的節點替換滾動升級由 KCP Controller 和 MachineSetReconciler 各自執行，Topology Controller 不直接接觸 Machine 物件。
3. **「Runtime Extension Hooks 是必要的設定」** — 非必要。Runtime Extension（RuntimeSDK）是進階功能，需要額外部署 webhook 服務並啟用 Feature Gate。大多數使用案例不需要 hooks，只有在需要整合外部系統（如 CMDB、審核系統）時才考慮使用。
</Callout>
