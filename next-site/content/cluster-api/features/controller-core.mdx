> 🧑‍💻 **類比**：Core Controller 就像「自動化值班人員」— 他們 24 小時不間斷地監看系統狀態，一旦發現現實（actual state）與預期（desired state）不符，立刻採取行動讓兩者一致。每位值班人員（Controller）只負責自己那塊職責：ClusterReconciler 管叢集層面，MachineReconciler 管單台機器，MachineSetReconciler 確保機器數量正確 — 各司其職，互不干擾。

## 🌐 整體觀念：為什麼需要這些 Controllers？

想像你是一個 Kubernetes 工程師，今天需要在雲端或裸機上建立一套三節點的工作叢集。在傳統方式下，你需要手動建立 VM、安裝作業系統、執行 kubeadm init、設定網路……每個步驟出錯就要從頭再來，而且沒有辦法「自動修復」任何問題。某天一台節點壞掉，你得半夜起床手動處理。

CAPI 的 Core Controllers 徹底改變了這個流程。你只需要宣告「我想要一個三節點叢集」，剩下的全部交給 Controllers 自動完成。更重要的是，這些 Controllers **永遠不停止監視**——哪天一台節點意外壞掉，Controller 會自動偵測並替換，不需要人工介入。

這套機制的核心是**協調循環（Reconcile Loop）**：持續比對「現實狀態」與「期望狀態」，有差異就採取行動讓兩者一致。這就是 Kubernetes「最終一致性」的實現方式。

## 🔄 核心流程：apply 一個 Cluster YAML，接下來 45 秒發生了什麼？

> 💡 **場景**：你執行 `kubectl apply -f my-cluster.yaml`，裡面定義了一個三節點叢集（1 Control Plane + 2 Workers）。讓我們跟著這個請求，看看每個 Controller 如何接力完成任務。

**第 0 秒 — 你的 YAML 進入 API Server**

Kubernetes API Server 將 `Cluster`、`KubeadmControlPlane`、`MachineDeployment` 等物件儲存到 etcd。這些物件此刻只是資料，還沒有任何實際動作發生。

**第 1 秒 — ClusterReconciler 醒來**

ClusterReconciler 透過 informer 偵測到新的 `Cluster` 物件。它先加入 finalizer（確保日後刪除時能正確清理），接著開始協調 Infrastructure Provider：確認 `InfrastructureCluster`（例如 `MaasCluster`、`Metal3Cluster`）物件存在，等待 `InfraCluster.Status.Ready = true`。

**第 5 秒 — Infrastructure Provider 回報就緒**

Infrastructure Provider（CAPMAAS、CAPM3 等）完成叢集層級設定（DNS、Load Balancer endpoint 等），回報 `Ready = true`。ClusterReconciler 收到通知，轉而等待 `KubeadmControlPlane` 完成初始化。

**第 15 秒 — MachineReconciler 為第一台 CP 機器啟動**

KubeadmControlPlane Controller 建立第一台 Control Plane `Machine`。`MachineReconciler` 立即響應，開始執行三步驟流程：

1. 等待 Bootstrap Provider（KubeadmConfig）生成 cloud-init userdata
2. Bootstrap 就緒後，通知 Infrastructure Provider 建立實際機器
3. 機器啟動、執行 `kubeadm init`，節點加入叢集

**第 30 秒 — Control Plane 就緒，Workers 開始建立**

Control Plane 標記為 `Initialized`，ClusterReconciler 建立 `ClusterCache`（連線到 workload cluster API Server）。與此同時，`MachineSetReconciler` 開始處理 Workers：目前有 0 台、期望是 2 台，於是建立 2 個新 `Machine` 物件，`MachineReconciler` 分別為每台 Worker 執行 Bootstrap → Infrastructure → Node Join 流程。

**第 45 秒 — 叢集完全就緒**

兩台 Worker Node 加入叢集，`MachineHealthCheckReconciler` 開始持續監控所有節點的健康狀態。整個叢集達到期望狀態，Controllers 進入「監視模式」，隨時準備應對下一個變更。

## 🧩 元件詳解

現在你已經了解整體流程，讓我們深入看每個 Controller 的具體責任與實作細節。

### Controller 一覽

| Controller | 負責資源 | 在流程中的角色 |
|-----------|---------|-------------|
| `ClusterReconciler` | Cluster | 協調 Infrastructure + Control Plane，是整個流程的「指揮官」 |
| `MachineReconciler` | Machine | 處理每台機器的 Bootstrap → Infra → Node Join，以及安全刪除流程 |
| `MachineSetReconciler` | MachineSet | 確保 Worker 數量符合期望，負責擴縮容決策 |
| `MachineDeploymentReconciler` | MachineDeployment | 管理 MachineSet 的滾動更新 |
| `MachineHealthCheckReconciler` | MachineHealthCheck | 叢集就緒後持續監控節點健康，觸發 Remediation |
| `ClusterCacheReconciler` | Cluster | 維護連到 workload cluster 的連線快取，供其他 Controller 使用 |

---

### ClusterReconciler 詳解

`ClusterReconciler` 是整個流程的指揮官——它不親自建立節點，但負責協調所有子系統的啟動順序，確保 InfraCluster 先就緒、ControlPlane 再就緒。

**Step 1 — 取得 Cluster 物件**

從 API Server 取得 Cluster 物件；若不存在則直接返回（可能已被刪除）。

**Step 2 — 處理 Finalizer**

若 Cluster 尚未有 `cluster.cluster.x-k8s.io` finalizer，先加入，確保刪除時能正確清理相關資源。

**Step 3 — 若 Cluster 正在刪除（有 DeletionTimestamp）**

執行刪除流程，依序：
1. 刪除所有隸屬的 Machine（確保節點先被清理）
2. 等待所有 Machine 刪除完成
3. 刪除 InfrastructureCluster
4. 刪除 ControlPlane
5. 移除 Finalizer，讓 Kubernetes 完成物件刪除

**Step 4 — 正常 reconcile**

1. **協調 Infrastructure Provider**：建立或更新 InfrastructureCluster 物件（如 AWSCluster、VSphereCluster）
2. **等待 InfraCluster 就緒**：輪詢直到 `InfraCluster.Status.Ready = true`
3. **協調 Control Plane Provider**：建立或更新 ControlPlane 物件（如 KubeadmControlPlane）
4. **等待 ControlPlane Initialized**：輪詢直到 `ControlPlane.Status.Initialized = true`（API Server 可接受請求）
5. **更新 Cluster.Status.ControlPlaneReady**
6. **建立 ClusterCache**：連到 workload cluster 的 API Server，供後續 Machine drain/Node 查詢使用

**Cluster 狀態轉換**

| 狀態 | 條件 |
|------|------|
| `Pending` | Cluster 剛建立，尚未開始 reconcile |
| `Provisioning` | InfraCluster 或 ControlPlane 尚未就緒 |
| `Provisioned` | `InfraCluster.Ready=true` 且 `ControlPlane.Ready=true` |
| `Deleting` | Cluster 正在刪除（有 DeletionTimestamp） |
| `Failed` | 發生不可恢復的錯誤 |
| `Unknown` | 狀態不明（通常是短暫過渡狀態） |

### MachineReconciler 詳解

`MachineReconciler` 負責單一 Machine 的完整生命週期——每台機器從「一個 YAML 物件」到「一個正在運行的 Kubernetes Node」，都是它一手完成的。

**Step 1 — 取得 Machine 物件，確認所屬 Cluster**

取得 Machine 後，透過 `cluster.x-k8s.io/cluster-name` label 找到對應的 Cluster 物件，確認 Cluster 存在且未刪除。

**Step 2 — 若 Machine 正在刪除（有 DeletionTimestamp）**

進入刪除流程（詳見下方「Machine 刪除流程」）。

**Step 3 — 正常 reconcile**

1. **協調 Bootstrap Provider**：等待 bootstrap data（`dataSecretName`）就緒，bootstrap data 包含節點初始化所需的 cloud-init 或類似腳本
2. **協調 Infrastructure Provider**：建立實際機器，等待回填 `ProviderID` 與 `addresses` 到 Machine.Status
3. **等待 Node 加入 workload cluster**：透過 ClusterCache 查找 workload cluster 中 `spec.providerID` 相符的 Node
4. **設定 Machine.Status.NodeRef**：記錄對應的 Node 名稱
5. **更新 Machine.Status.Phase = Running**

**Machine 刪除流程**

<Callout type="info" title="Lifecycle Hooks">

Machine 支援兩種 lifecycle hook annotation，允許外部系統介入刪除流程：

- `pre-drain.delete.hook.machine.cluster.x-k8s.io/<name>`：drain 前等待外部系統回應
- `pre-terminate.delete.hook.machine.cluster.x-k8s.io/<name>`：terminate 前等待外部系統

外部系統完成準備後，移除對應 annotation 以允許流程繼續。

```yaml
metadata:
  annotations:
    pre-drain.delete.hook.machine.cluster.x-k8s.io/backup: "external-system"
    # 外部系統完成後移除此 annotation 以允許繼續
```

</Callout>

完整刪除步驟：

1. **Pre-drain hook**：若有 `pre-drain` lifecycle hook annotation，暫停並等待 hook 完成（annotation 被移除）
2. **Cordon 節點**：對 workload cluster 中的 Node 執行 Cordon，禁止新 Pod 排程到此節點
3. **Drain 節點**：驅逐所有 Pod，遵守 PodDisruptionBudget；若 `Machine.Spec.NodeDrainTimeout` 超時，強制繼續
4. **等待 Volume Detach**：等待所有 CSI volumes 卸除完成；若 `Machine.Spec.NodeVolumeDetachTimeout` 超時，繼續下一步
5. **Pre-terminate hook**：若有 `pre-terminate` lifecycle hook annotation，暫停並等待 hook 完成
6. **刪除 InfrastructureMachine**：觸發 Infrastructure Provider 刪除實際的底層機器
7. **刪除 BootstrapConfig**：清理 bootstrap 相關資源（如 KubeadmConfig）
8. **移除 Finalizer**：Machine 物件完全刪除

<Callout type="tip" title="新手提示：Reconcile Loop 是什麼？">
Reconcile Loop（協調循環）是 Kubernetes Controller 的核心概念。每個 Controller 持續「觀察現實狀態，比對期望狀態，採取行動讓兩者一致」。當你修改一個 YAML 設定，Controller 就會被觸發執行 Reconcile，嘗試讓現實符合新設定。如果執行失敗（例如資源暫時不足），Controller 會自動重試，直到成功為止 — 這就是 Kubernetes「最終一致性」的核心機制。
</Callout>

### MachineSetReconciler 詳解

在叢集建立流程的第 30 秒，`MachineSetReconciler` 開始負責「確保 Worker 數量正確」——它類似 Kubernetes 原生的 ReplicaSet，持續維持指定的 Machine 副本數。

**核心邏輯**

1. **Sync**：列出所有符合 `selector` 的 Machine，更新 MachineSet 的狀態（replicas、readyReplicas 等）
2. **採納（Adoption）**：將沒有 ownerReference 但符合 label selector 的 Machine 採納進來，設定 ownerRef 指向此 MachineSet
3. **擴展（Scale Up）**：若目前 Machine 數量 < `spec.replicas`，依據 template 建立新 Machine
4. **縮減（Scale Down）**：若目前 Machine 數量 > `spec.replicas`，根據 `deletePolicy` 選取要刪除的 Machine：
   - `Random`：隨機選取
   - `Newest`：優先刪除最新建立的
   - `Oldest`：優先刪除最舊建立的
   - 無論何種策略，都優先刪除 `Failed` 狀態的 Machine，以及沒有對應 Node 的 Machine

**MachineSet Conditions**

| Condition | 說明 |
|-----------|------|
| `MachinesReady` | 所有 Machine 是否就緒（Phase=Running 且 NodeRef 已設定） |
| `MachinesUpToDate` | 所有 Machine 的版本（bootstrap/infra template）是否為最新 |
| `ScalingUp` | 是否正在擴展（目前數量 < replicas） |
| `ScalingDown` | 是否正在縮減（目前數量 > replicas） |

### MachineHealthCheckReconciler 詳解

當叢集完全就緒（第 45 秒後），`MachineHealthCheckReconciler` 正式上場——它是叢集的「長期守護者」，持續監控所有節點是否健康，並在發現問題時自動觸發修復。

**健康檢查流程**

1. **列出受管 Machine**：根據 `spec.selector` 找到所有相關 Machine（通常對應一個 MachineDeployment 的 workers）

2. **評估健康狀態**：
   - 透過 ClusterCache 查詢 workload cluster 中對應 Node 的 Conditions
   - 若 `Node.Status.Conditions[Ready] = False` 或 `Unknown`，且持續時間超過 `unhealthyConditions[*].timeout`，標記此 Machine 為不健康
   - 若 Machine 沒有對應 Node（NodeRef 未設定）且超過 `nodeStartupTimeout`，標記為不健康

3. **Circuit Breaker**：計算不健康 Machine 的比例，若超過 `spec.maxUnhealthy` 則停止所有 remediation，避免大規模刪除節點造成服務中斷

4. **觸發 Remediation**：
   - 若 MachineHealthCheck 指定了 `remediationTemplate`（如 Metal3RemediationTemplate）：建立對應的 Remediation 物件，由外部 controller 處理
   - 若未指定：刪除不健康的 Machine（CAPI 預設策略），讓 MachineSet/MachineDeployment 建立替換機器

**MachineHealthCheck 範例**

```yaml
apiVersion: cluster.x-k8s.io/v1beta1
kind: MachineHealthCheck
metadata:
  name: my-cluster-mhc
spec:
  clusterName: my-cluster
  selector:
    matchLabels:
      cluster.x-k8s.io/deployment-name: my-cluster-workers
  unhealthyConditions:
    - type: Ready
      status: Unknown
      timeout: 300s
    - type: Ready
      status: "False"
      timeout: 300s
  maxUnhealthy: "40%"
  nodeStartupTimeout: 20m
```

### ClusterCache（RemoteClusterCache）

`ClusterCache` 是整個 Controller 系統能夠「跨叢集」操作的關鍵基礎設施。沒有它，MachineReconciler 無法執行 Cordon/Drain，MachineHealthCheckReconciler 也無法查詢 Node 狀態。

**為什麼需要 ClusterCache**

CAPI core controllers 在 management cluster 運行，但需要存取 workload cluster 內部資源（如查詢 Node 狀態、執行 cordon/drain）。ClusterCache 為每個 workload cluster 維護一個帶有 informer 的 cached client，避免每次操作都直接呼叫遠端 API Server。

**運作方式**

- 透過 `<cluster-name>-kubeconfig` Secret 取得 workload cluster 的 kubeconfig
- 為每個 workload cluster 建立獨立的 controller-runtime client（帶 cache）
- 使用 informer 機制監控 workload cluster 資源（Node、Pod 等），減少 API Server 負擔
- 其他 controllers（如 MachineReconciler、MachineHealthCheckReconciler）透過 ClusterCache 取得對應的 client

**Cache 刷新機制**

- 預設每 10 分鐘強制刷新一次（重建 informer）
- 若 `<cluster-name>-kubeconfig` Secret 更新（如憑證輪換），自動重建 cache
- 若連線失敗（workload cluster 暫時不可達），進入 exponential backoff 重試

<Callout type="tip" title="ClusterCache 重構說明">

ClusterCache 是 CAPI v1.4+ 引入的重構，取代舊的 RemoteClusterCache，提供更好的連線管理和錯誤處理。新版本支援更精細的健康狀態回報，並改善了在 workload cluster 暫時失聯時的行為。

</Callout>

## ⚠️ 常見陷阱與注意事項

<Callout type="warning" title="常見誤解澄清">
1. **「ClusterReconciler 負責建立所有子資源（KCP、MachineDeployment）」** — 錯誤。ClusterReconciler 只協調 Cluster 本身的狀態（等待 InfraCluster ready、等待 ControlPlane ready），實際建立 KCP 或 MachineDeployment 是由使用者透過 YAML 或 Topology Controller 完成的，ClusterReconciler 不主動建立這些資源。
2. **「MachineSetReconciler 在擴縮容時會立即建立/刪除所有差異數量的 Machine」** — 不完全正確。MachineSetReconciler 在每次 reconcile 中只處理部分差異（受 `concurrentReconciles` 限制），且刪除時會根據 `deletePolicy` 選擇特定 Machine，不是隨機批量操作。
3. **「ClusterCache 連線失敗後 reconcile 就會永遠停止」** — 錯誤。ClusterCache 採用 exponential backoff 重試機制，連線失敗後會以遞增的間隔重新嘗試建立連線。一旦恢復連線，相關 Controller 的 reconcile 會自動繼續。
</Callout>

> 💡 **思考一下**：CAPI 的 MachineReconciler 在刪除 Machine 時需要 Cordon → Drain → Volume Detach。但這些操作需要連接到 workload cluster 的 API Server。如果 workload cluster API Server 暫時不可用，刪除流程會如何處理？Controller 有哪些機制來應對這種情況？
