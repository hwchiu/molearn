## 整體架構概覽

MAAS Provider 以標準 CAPI Infrastructure Provider 模式運作，由以下核心元件構成：

- **CAPI core**（Cluster API manager）：在 management cluster 中運行，負責協調 Cluster、Machine 等 CAPI 物件
- **MaasClusterReconciler**：叢集層級控制器，管理 DNS 與 controlPlaneEndpoint
- **MaasMachineReconciler**：機器層級控制器，負責 MAAS 機器的完整生命週期
- **spectrocloud/maas-client-go SDK**：封裝 MAAS REST API，Provider 透過此 SDK 與 MAAS 端點溝通
- **三個 CRD**：`MaasCluster`、`MaasMachine`、`MaasMachineTemplate`

![架構元件圖](/diagrams/maas/architecture-components.png)

## 兩大 Controller

### MaasClusterReconciler

負責**叢集層級**資源的建立與回收：

- 管理 DNS FQDN，格式為 `<cluster-name>.<dnsDomain>`，作為 control plane 的對外存取位址
- 設定 `controlPlaneEndpoint`，讓 CAPI core 知道如何連接 kube-apiserver
- 持續監控 APIServer 可用性，在 APIServer 就緒後將 `MaasCluster.Status.Ready` 標記為 `true`
- Reconcile 結束時若叢集被刪除，負責清除 DNS 記錄與相關資源

### MaasMachineReconciler

負責**機器生命週期**的全程管理：

- 呼叫 MAAS allocate API，依照 `MaasMachine.spec` 條件（min CPU、min memory、資源池等）取得機器
- 呼叫 MAAS deploy API，部署指定 OS image 並注入 cloud-init userdata
- 透過 `ClusterCacheTracker` 取得 workload cluster 的 Kubernetes client，以便讀取 Node 狀態
- 部署完成後設定 `Node.spec.providerID`，格式為 `maas:///<zone>/<systemID>`
- 叢集刪除或 Machine 刪除時呼叫 MAAS release API，歸還機器至資源池

## 三個 CRD

| CRD | 用途 | 對應 CAPI 物件 |
|-----|------|----------------|
| `MaasCluster` | 叢集層級：DNS domain、CP endpoint、failure domains | `Cluster.spec.infrastructureRef` |
| `MaasMachine` | 機器層級：分配條件、image、部署參數 | `Machine.spec.infrastructureRef` |
| `MaasMachineTemplate` | 不可變模板，用於 MachineDeployment 批次建立機器 | `MachineDeployment` 的模板來源 |

## Scope 物件深度解析

Scope 模式是 CAPI Provider 的標準設計——每次 reconcile 開始時建立一個 scope 物件，將本次 reconcile 所需的所有上下文打包在一起，傳遞給各個業務邏輯函式，避免函式簽名因參數過多而變得複雜。

### ClusterScope

`ClusterScope` 封裝了叢集層級 reconcile 所需的一切，定義於 `pkg/maas/scope/cluster.go`：

```go
type ClusterScope struct {
    logr.Logger                         // 結構化日誌，帶 cluster name context
    client      client.Client           // management cluster 的 k8s client
    patchHelper *patch.Helper           // SSA-safe patch，用於 Close() 持久化狀態

    Cluster             *clusterv1.Cluster      // CAPI 標準 Cluster 物件
    MaasCluster         *infrav1beta1.MaasCluster  // MAAS 特定的叢集 spec/status
    controllerName      string                  // "maascluster" 或 "maasmachine"
    tracker             *remote.ClusterCacheTracker // workload cluster 的 remote client
    clusterEventChannel chan event.GenericEvent  // API server 上線後觸發 reconcile 的 channel
}
```

`ClusterScopeParams` 是建立時的輸入，包含 `Client`、`Logger`、`Cluster`、`MaasCluster`、`Tracker`、`ClusterEventChannel`、`ControllerName` 等必要欄位。

`Close()` 方法在每次 reconcile 結束時（透過 `defer`）被呼叫，它會呼叫 `PatchObject()`，後者使用 `patch.Helper` 將 `MaasCluster` 狀態寫回 API server，並同時計算並更新 `ReadyCondition` 的摘要（`conditions.SetSummary`），聚合 `DNSReadyCondition` 與 `APIServerAvailableCondition` 兩個子條件。

**APIServer 在線偵測**：`ReconcileMaasClusterWhenAPIServerIsOnline()` 會啟動一個背景 goroutine，每秒透過 `tracker.GetClient()` 對 workload cluster 的 Node list 發出查詢，一旦成功就向 `clusterEventChannel` 送出 `GenericEvent`，觸發 reconciler 再次執行。全域 `apiServerTriggers` map 加上 `sync.Mutex` 確保每個叢集只有一個 goroutine 在偵測。

### MachineScope

`MachineScope` 在 `ClusterScope` 的基礎上新增了機器層級的資訊，定義於 `pkg/maas/scope/machine.go`：

```go
type MachineScope struct {
    logr.Logger
    client      client.Client
    patchHelper *patch.Helper           // 針對 MaasMachine 的 patch helper

    Cluster      *clusterv1.Cluster
    ClusterScope *ClusterScope          // 持有對 ClusterScope 的參照

    Machine     *clusterv1.Machine      // CAPI 標準 Machine（含 Bootstrap.DataSecretName）
    MaasMachine *infrav1beta1.MaasMachine  // MAAS 特定的機器 spec/status

    controllerName string
    tracker        *remote.ClusterCacheTracker
}
```

`MachineScope` 提供了豐富的操作方法，直接在 reconciler 中使用：

| 方法 | 功能 |
|------|------|
| `GetInstanceID()` | 解析 `Spec.ProviderID`（`maas:///<zone>/<id>`）取得 system ID |
| `SetProviderID(systemID, az)` | 寫入格式化的 `maas:///<zone>/<systemID>` |
| `GetRawBootstrapData()` | 從 `Machine.Spec.Bootstrap.DataSecretName` 指向的 Secret 讀取 cloud-init 資料 |
| `SetNodeProviderID()` | 透過 `tracker.GetClient()` 取得 workload cluster client，patch Node 的 `Spec.ProviderID` |
| `MachineIsRunning()` | 檢查狀態是否在 `MachineRunningStates`（Deploying、Deployed）集合內 |
| `MachineIsOperational()` | 檢查狀態是否在 `MachineOperationalStates`（+ Allocated）集合內 |
| `MachineIsInKnownState()` | 檢查狀態是否在 `MachineKnownStates`（含 Ready、Releasing 等）集合內 |
| `SetFailureReason/Message()` | 設定終止性錯誤資訊至 `Status.FailureReason/FailureMessage` |

`PatchObject()` 在 control plane 機器時額外包含 `DNSAttachedCondition`，並同樣透過 `conditions.SetSummary` 計算 `ReadyCondition`。

### 為何使用 Scope 模式

若不使用 Scope，每個業務函式都需要傳入 `client`、`logger`、`cluster`、`maasMachine`、`tracker` 等多個參數；新增欄位時所有呼叫者都需修改。Scope 將這些「上下文資料」封裝為一個物件，讓 `dns.Service`、`machine.Service` 等業務模組只需接受一個 `*scope.ClusterScope` 或 `*scope.MachineScope`，大幅降低耦合度。

## Controller 啟動與 Watch 設定

### Scheme 與 Manager 初始化（main.go）

`init()` 函式向 `scheme` 注冊三個 API 群組：

```go
_ = clientgoscheme.AddToScheme(scheme)    // 核心 k8s 物件
_ = clusterv1.AddToScheme(scheme)         // CAPI v1beta1 物件
_ = infrav1beta1.AddToScheme(scheme)      // MAAS Provider 自訂 CRD
```

Manager 以 `ctrl.NewManager()` 建立，關鍵選項包括：

- `LeaderElection: enableLeaderElection`，Lock 名稱為 `controller-leader-election-capmaas`
- `Cache.SyncPeriod: &syncPeriod`（預設 120 分鐘）控制所有 informer 的強制重新同步週期
- Metrics server 綁定 `:8080`，WebhookServer 綁定埠 `9443`（TLS）

Manager 啟動前另外建立 `remote.ClusterCacheTracker`，並以 `concurrency(1)` 啟動 `ClusterCacheReconciler` 維護與各 workload cluster 的連線快取，`NodeProviderIDIndex` 讓查詢 Node 時可依 providerID 做索引。

### MaasClusterReconciler 的 Watch 設定

```go
ctrl.NewControllerManagedBy(mgr).
    For(&infrav1beta1.MaasCluster{}).          // 主要 Watch：MaasCluster 本身的任何變更
    Watches(
        &infrav1beta1.MaasMachine{},
        handler.EnqueueRequestsFromMapFunc(
            r.controlPlaneMachineToCluster),   // CP 機器狀態變化 → 觸發所屬 MaasCluster
    ).
    Watches(
        &clusterv1.Cluster{},
        handler.EnqueueRequestsFromMapFunc(
            util.ClusterToInfrastructureMapFunc(...)),  // CAPI Cluster 變更 → 觸發對應 MaasCluster
    ).
    WithEventFilter(predicates.ResourceNotPaused(...)).  // 被暫停的資源略過
    Build(r)
```

此外，`GenericEventChannel` 作為 `source.Channel` 單獨 Watch，當 API server 上線後的背景 goroutine 送出事件時觸發 reconcile。`controlPlaneMachineToCluster` mapper 只處理帶有 `cluster.x-k8s.io/control-plane` label 的 `MaasMachine`，避免 worker 節點狀態變化不必要地觸發叢集層級 reconcile。

### MaasMachineReconciler 的 Watch 設定

```go
ctrl.NewControllerManagedBy(mgr).
    For(&infrav1beta1.MaasMachine{}).          // 主要 Watch：MaasMachine 本身
    Watches(
        &clusterv1.Machine{},
        handler.EnqueueRequestsFromMapFunc(
            util.MachineToInfrastructureMapFunc(...)),  // CAPI Machine → 對應 MaasMachine
    ).
    Watches(
        &infrav1beta1.MaasCluster{},
        handler.EnqueueRequestsFromMapFunc(
            r.MaasClusterToMaasMachines),      // MaasCluster 變更 → 觸發所有屬於該叢集的 MaasMachine
    ).
    Watches(
        &clusterv1.Cluster{},
        handler.EnqueueRequestsFromMapFunc(clusterToMaasMachines),  // CAPI Cluster 變更 → 全部 MaasMachine
    ).
    WithEventFilter(predicates.ResourceNotPaused(...)).
    Complete(r)
```

`MaasClusterToMaasMachines` mapper 會先取得 owning CAPI Cluster，再列出所有帶有 `cluster.x-k8s.io/cluster-name` label 的 `Machine`，最後回傳各 Machine 的 `InfrastructureRef` 所指向的 `MaasMachine`。

## Condition 設計

Condition 是 CAPI 觀察資源健康狀態的標準機制，所有常數定義在 `api/v1beta1/condition_consts.go`。

### MaasCluster 的 Conditions

| Condition Type | 常數名稱 | 說明 |
|----------------|----------|------|
| `LoadBalancerReady` | `DNSReadyCondition` | DNS 記錄是否已成功建立 |
| `APIServerAvailable` | `APIServerAvailableCondition` | workload cluster API server 是否可連線 |

**Reason 常數（MaasCluster）：**

| Reason | 對應 Condition | 意義 |
|--------|----------------|------|
| `LoadBalancerFailed` (`DNSFailedReason`) | `DNSReadyCondition=False` | DNS reconcile 失敗（通常為 MAAS API 錯誤）|
| `WaitForDNSName` (`WaitForDNSNameReason`) | `DNSReadyCondition=False` | DNS 名稱尚未取得，等待中 |
| `APIServerNotReady` (`APIServerNotReadyReason`) | `APIServerAvailableCondition=False` | API server 查詢失敗 |

### MaasMachine 的 Conditions

| Condition Type | 常數名稱 | 說明 |
|----------------|----------|------|
| `MachineDeployed` | `MachineDeployedCondition` | 機器是否已完成部署（主要 condition）|
| `DNSAttached` | `DNSAttachedCondition` | 僅 control plane：機器 IP 是否已加入 DNS 記錄 |

**Reason 常數（MaasMachine）：**

| Reason | Severity | 意義 |
|--------|----------|------|
| `WaitingForClusterInfrastructure` | Info | 等待 MaasCluster.Status.Ready=true |
| `WaitingForBootstrapData` | Info | 等待 bootstrap secret 被 CAPI 填入 |
| `MachineDeployStartedReason` | Info | allocate/deploy 已啟動 |
| `MachineDeploying` | Warning | 機器仍在部署中（Deploying 或 Allocated 狀態）|
| `MachineDeployFailed` | Error | 部署失敗（MAAS API 回傳錯誤）|
| `MachineTerminatedReason` | Error | 機器出現預期外的終止狀態（Ready/New/Releasing）|
| `MachinePoweredOff` | Warning | 機器已部署但電源關閉 |
| `MachineNotFound` | — | 無法透過 system ID 找到機器 |
| `DNSDetachPending` | Warning | 等待 MaasCluster 移除 DNS 記錄 |
| `DNSAttachPending` | Warning | 等待 MaasCluster 添加 DNS 記錄 |

**Conditions 的使用方式**（以 `sigs.k8s.io/cluster-api/util/conditions` 套件）：

```go
// 設為 True
conditions.MarkTrue(maasCluster, infrav1beta1.DNSReadyCondition)

// 設為 False 並附上 Reason 與 Severity
conditions.MarkFalse(maasMachine, infrav1beta1.MachineDeployedCondition,
    infrav1beta1.MachineDeployingReason, clusterv1.ConditionSeverityWarning, "")

// 設為 Unknown（狀態不明，通常是 API 錯誤）
conditions.MarkUnknown(maasMachine, infrav1beta1.MachineDeployedCondition,
    infrav1beta1.MachineNotFoundReason, "%v", err)
```

`PatchObject()` 中的 `conditions.SetSummary()` 會根據所有子 conditions 的最差狀態自動計算 `ReadyCondition`，並以 step counter 表示目前進度。`patch.WithOwnedConditions` 宣告本 controller 擁有哪些 condition，避免多個 controller 互相覆蓋。

## Error Handling 與 Requeue 策略

MAAS Provider 的 reconcile 函式回傳值遵循以下策略：

### ctrl.Result&#123;&#125; 的三種模式

| 回傳值 | 場景 |
|--------|------|
| `ctrl.Result&#123;&#125;, nil` | 成功完成，等待下一次 Watch 事件觸發 |
| `ctrl.Result&#123;RequeueAfter: X&#125;, nil` | 需要定時重試（如等待機器部署完成、等待 API server 上線）|
| `ctrl.Result&#123;&#125;, err` | 遇到需要立即重試的錯誤，controller-runtime 會以指數退避方式 requeue |

**具體的 RequeueAfter 場景（從原始碼提取）：**

```go
// MaasClusterReconciler：等待所有 MaasMachine 刪除完成
return reconcile.Result{RequeueAfter: 10 * time.Second}, nil

// MaasClusterReconciler：DNS 名稱尚未建立
return reconcile.Result{RequeueAfter: 15 * time.Second}, nil

// MaasMachineReconciler：機器已部署但電源關閉，等待開機
return ctrl.Result{RequeueAfter: 1 * time.Minute}, nil

// MaasMachineReconciler：API server 尚未上線
return ctrl.Result{RequeueAfter: 5 * time.Minute}, nil

// MaasMachineReconciler：機器尚未進入 operational 狀態
return ctrl.Result{RequeueAfter: 5 * time.Minute}, nil

// MaasMachineReconciler：設定 Node ProviderID 失敗（通常為暫時性）
return ctrl.Result{RequeueAfter: 30 * time.Second}, nil
```

### DNS Requeue 的特殊處理

DNS attachment 狀態管理使用哨兵錯誤（sentinel error）來控制 requeue，而非直接回傳一般錯誤：

```go
var ErrRequeueDNS = errors.New("need to requeue DNS")

// 使用 errors.Is 判斷是否為 DNS requeue
if errors.Is(err, ErrRequeueDNS) {
    return ctrl.Result{}, nil  // 不帶 error 回傳，避免增加 error rate metrics
}
```

這樣設計的原因：DNS attachment 的等待是預期行為（MaasCluster reconciler 負責實際更新），不應被記錄為 error，因此回傳 `ctrl.Result&#123;&#125;, nil` 而非帶錯誤的回傳值。

### 機器狀態的終止性失敗

當機器出現 `MachineStateReady`（已釋放回資源池）、`MachineStateNew`（未 commissioned）等**預期外狀態**時，reconciler 會設定終止性失敗資訊：

```go
machineScope.SetFailureReason(capierrors.UpdateMachineError)
machineScope.SetFailureMessage(errors.Errorf("Maas machine state %q is unexpected", m.State))
```

`FailureReason` 與 `FailureMessage` 一旦設定，CAPI Machine controller 會將機器標記為不可修復（terminal error），阻止後續自動修復嘗試，需要人工介入處理。

### Reconcile 入口的前置檢查順序

MaasMachineReconciler 在進入業務邏輯前，按以下順序做前置檢查：

1. 取得 `MaasMachine`（Not Found → return nil）
2. 取得 owner `Machine`（未設定 OwnerRef → 等待）
3. 取得 owner `Cluster`（missing label → return nil）
4. 檢查 paused 狀態（paused → return nil）
5. 取得 `MaasCluster`（尚未建立 → return nil）
6. 建立 `ClusterScope` 與 `MachineScope`
7. `defer machineScope.Close()`（確保狀態寫回）
8. 依 DeletionTimestamp 分流至 `reconcileDelete` 或 `reconcileNormal`

## Provider 啟動流程

Manager 透過以下參數啟動，可在部署的 `Deployment` manifest 中覆寫預設值：

| 參數 | 預設值 | 說明 |
|------|--------|------|
| `--machine-concurrency` | `2` | MaasMachine 並發 reconcile 數量 |
| `--sync-period` | `120m` | 定期 resync 的間隔時間 |
| `--metrics-bind-addr` | `:8080` | Prometheus metrics 監聽位址 |
| `--health-addr` | `:9440` | liveness/readiness probe 位址 |
| `--webhook-port` | `9443` | Admission webhook 監聽埠 |
| `--leader-elect` | `false` | 啟用 leader election 以支援 HA 部署 |
| `--namespace` | `""` | 限制 watch 的 namespace，空字串代表所有 namespace |


<Callout type="tip" title="生產環境建議">

正式環境部署時，建議啟用 `--leader-elect` 並部署多副本，以確保 Provider 的高可用性。

</Callout>


## 環境變數與設定


<Callout type="warning" title="認證方式">

MAAS Provider **不使用 kubeconfig** 認證 MAAS 端點，而是透過以下兩個環境變數進行認證。這兩個變數必須在 Provider 的 Pod 環境中存在，通常透過 Kubernetes Secret 掛載。

</Callout>


| 環境變數 | 必填 | 說明 |
|----------|------|------|
| `MAAS_ENDPOINT` | ✅ | MAAS API endpoint URL，例：`http://maas.example.com/MAAS` |
| `MAAS_API_KEY` | ✅ | MAAS API 金鑰，格式：`consumer_key:token_key:token_secret` |

API 金鑰可從 MAAS UI 的使用者設定頁面產生，或透過 MAAS CLI 取得：

```bash
maas apikey --username <username>
```

## 已知限制


<Callout type="warning" title="使用前須知">

以下為目前版本的已知限制，規劃部署前請先評估是否影響您的使用情境：

1. **固定 Pod CIDR**：Pod 網路使用 `192.168.0.0/16`，目前無法透過 `MaasCluster` spec 自訂，若與現有網路衝突需另行處理
2. **不支援原生升級**：Kubernetes 版本升級需透過 rolling MachineDeployment 進行——刪除舊機器並建立新機器，無法原地升級
3. **In-Memory 前提**：需要 MAAS 版本 ≥ 3.5.10，且目標機器至少具備 16GB RAM 以支援 in-memory 部署模式
4. **單一 MAAS 端點**：一個 Provider 部署實例僅對應一個 MAAS 端點，若需管理多個 MAAS 環境，需部署多個 Provider 實例

</Callout>
