## 控制器架構模式

MAAS Provider 的兩個控制器（`MaasClusterReconciler` 與 `MaasMachineReconciler`）均遵循標準 CAPI reconciler pattern，流程如下：

1. **取得物件**：從 API Server 取得目標 CR，若不存在則結束
2. **Pause 檢查**：若叢集或機器被暫停（paused），跳過本次 reconcile
3. **設定 Finalizer**：確保刪除流程可被攔截，進行清理
4. **判斷 reconcile 類型**：依 `DeletionTimestamp` 是否存在，決定執行正常 reconcile 或 delete reconcile
5. **執行對應邏輯**：呼叫 MAAS API、設定狀態、更新 conditions
6. **Patch 物件狀態**：將最新狀態寫回 API Server

---

## MaasClusterReconciler

`MaasClusterReconciler` 負責管理 `MaasCluster` 資源的生命週期，對應到 CAPI 的 infrastructure cluster 角色。

### Reconcile 主流程

1. 從 API Server 取得 `MaasCluster` 物件；若不存在則結束
2. 透過 `OwnerReference` 取得對應的 CAPI `Cluster` 物件
3. 建立 `ClusterScope`，封裝 `MaasCluster`、`Cluster` 以及 MAAS client
4. 若 `Cluster.Spec.Paused` 為 `true`，或 `MaasCluster` 帶有 `cluster.x-k8s.io/paused` annotation → 設定 `ClusterPaused` condition 後返回
5. 設定 finalizer（`maascluster.infrastructure.cluster.x-k8s.io`）
6. 若 `DeletionTimestamp` 已設定 → 執行 `reconcileDelete`
7. 否則 → 執行 `reconcileNormal`

### reconcileNormal 流程

```
1. 若 spec.failureDomains 有設定 → 呼叫 MAAS API 確認 failure domains 存在
2. 若 controlPlaneEndpoint 尚未設定：
   → 以 <cluster-name>.<dnsDomain> 建立 DNS FQDN
   → 設定為 controlPlaneEndpoint
3. 設定 DNSReadyCondition = True
4. 設定 MaasCluster.Status.Ready = true
```

### reconcileDelete 流程

```
1. 列出所有 MaasMachine，過濾出 OwnerRef 指向此 Cluster 的物件
2. 若仍有剩餘 MaasMachine → return（等待機器先被刪除）
3. 確認無剩餘 MaasMachine → 移除 finalizer
```

### Finalizer 管理

| 項目 | 說明 |
|------|------|
| Finalizer 名稱 | `maascluster.infrastructure.cluster.x-k8s.io` |
| 目的 | 確保叢集被刪除前，所有 MaasMachine 已完成清理 |
| 移除時機 | `reconcileDelete` 確認無剩餘 MaasMachine 後移除 |

## MaasMachineReconciler

`MaasMachineReconciler` 負責管理 `MaasMachine` 資源的生命週期，對應到 CAPI 的 infrastructure machine 角色，實際呼叫 MAAS API 進行機器分配、部署與釋放。

### Reconcile 主流程

1. 從 API Server 取得 `MaasMachine` 物件
2. 透過 `OwnerReference` 取得對應的 CAPI `Machine` 物件
3. 透過 `Machine.Spec.ClusterName` 取得 CAPI `Cluster` 與 `MaasCluster`
4. 建立 `MachineScope`，封裝所有相關物件與 MAAS client
5. 執行 Pause 檢查（同 ClusterReconciler 邏輯）
6. 設定 finalizer（`maasmachine.infrastructure.cluster.x-k8s.io`）
7. 若 `DeletionTimestamp` 已設定 → 執行 `reconcileDelete`
8. 否則 → 執行 `reconcileNormal`

### reconcileNormal 流程（逐步）

**Step 1 — Bootstrap 守衛**

```
若 Machine.Spec.Bootstrap.DataSecretName 未設定：
  → Bootstrap Provider 尚未完成，直接 return 等待重新觸發
```

**Step 2 — 機器分配（Allocate）**

```
若 MaasMachine.Status.SystemID 已設定（已分配過）→ 跳過
否則：
  → 呼叫 MAAS API AllocateMachine
  → 傳入：minCPU、minMemoryInMB、tags、resourcePool、failureDomain
  → 記錄分配到的 SystemID
```

**Step 3 — 機器部署（Deploy）**

```
若 MaasMachine state 已是 Deployed/Ready → 跳過
否則：
  → 呼叫 MAAS disable swap API
  → 讀取 Bootstrap secret 取得 cloud-init userdata
  → 呼叫 MAAS API DeployMachine
     傳入：image、userdata、deployInMemory flag
```

**Step 4 — 狀態輪詢**

```
呼叫 MAAS API GetMachine(systemID) 取得當前 state
若 state 不是 Deployed → return（等待下次 reconcile）
若 state 是 Deployed   → 繼續後續步驟
```

**Step 5 — 電源確認**

```
確認機器已上電（powered = true）
```

**Step 6 — 設定位址**

```
從 MAAS machine 資訊取得 IP addresses
設定 MaasMachine.Status.Addresses
```

**Step 7 — DNS 登記（僅 Control Plane）**

```
判斷此 Machine 是否為 Control Plane（檢查 Machine labels）
若是 CP：
  → 呼叫 DNS API 登記此 IP
  → 設定 DNSAttached = True
```

**Step 8 — 設定 ProviderID**

```
設定 MaasMachine.Spec.ProviderID = "maas://<systemID>"
透過 remote.ClusterCacheTracker 取得 workload cluster client
Patch workload cluster 中對應 Node 的 Spec.ProviderID
設定 MachineDeployed = True
設定 MaasMachine.Status.Ready = true
```

### reconcileDelete 流程

1. 取得 `MaasMachine` 的 `systemID`
2. 若此機器是 CP node：呼叫 DNS API 移除記錄，等待 `DNSAttached` condition 清除
3. 呼叫 MAAS API `ReleaseMachine(systemID)`
4. 等待 machine state 進入 `Releasing` 或 `New`
5. 移除 finalizer（`maasmachine.infrastructure.cluster.x-k8s.io`）

## Pause / Bootstrap 守衛


<Callout type="info" title="Pause 機制">

當 `Cluster.Spec.Paused` 設定為 `true`，或叢集/機器被加上 `cluster.x-k8s.io/paused` annotation 時，所有相關 controller 都會跳過本次 reconcile，直到 pause 被移除為止。

這是 CAPI 的標準機制，常用於叢集維護、升級準備、或問題排查期間，防止 controller 在此期間對資源進行非預期的修改。

</Callout>


<Callout type="info" title="Bootstrap 守衛">

`MaasMachine` controller 在部署機器前，必須等待 Bootstrap Provider（例如 KubeadmBootstrap）完成初始化並產生 bootstrap secret。

只有當 `Machine.Spec.Bootstrap.DataSecretName` 已填入時，controller 才會繼續執行機器部署流程。若尚未設定，controller 會直接 return 並等待下次 reconcile 觸發。

這確保了 cloud-init userdata（包含 kubeadm 設定、憑證等）在部署前已準備就緒。

</Callout>

## RBAC 權限設計

控制器的 RBAC 權限透過 `//+kubebuilder:rbac` annotation 定義，由 `controller-gen` 自動產生 ClusterRole manifest。

| 資源 | 動作 | 說明 |
|------|------|------|
| `maasclusters`、`maasmachines`、`maasmachinetemplate` | create, delete, get, list, patch, update, watch | 完整管理 MAAS Provider 自訂資源 |
| `clusters`、`machines` | get, list, watch | 讀取 CAPI 核心資源以取得 owner reference 及 cluster 設定 |
| `secrets` | get | 讀取 Bootstrap secret 取得 cloud-init userdata |
| `nodes` | get, list, patch | 設定 workload cluster node 的 `Spec.ProviderID` |
| `events` | create, patch | 記錄 controller 操作事件，供 `kubectl describe` 顯示 |
