# MAAS Provider — Machine 生命週期

## 概覽

MAAS Provider 的 Machine 生命週期由 **`MaasMachineReconciler`** 控制，透過 MAAS API 管理裸機的完整生命週期：從分配（Allocate）到部署（Deploy）到釋放（Release）。

Reconciler 監聽 `MaasMachine` 物件的變化，並根據目前狀態決定下一步動作——分配機器、觸發 OS 部署、輪詢等待完成，或在刪除時釋放機器還回 MAAS 資源池。

### 原始碼：Controller 結構

`MaasMachineReconciler` 的結構體定義與 `Reconcile()` 進入點，是整個生命週期管理的核心：

```go
// 原始碼: controllers/maasmachine_controller.go

// MaasMachineReconciler reconciles a MaasMachine object
type MaasMachineReconciler struct {
	client.Client
	Log      logr.Logger
	Scheme   *runtime.Scheme
	Recorder record.EventRecorder
	Tracker  *remote.ClusterCacheTracker // 存取 workload cluster 的 k8s client
}

func (r *MaasMachineReconciler) Reconcile(ctx context.Context, req ctrl.Request) (_ ctrl.Result, rerr error) {
	maasMachine := &infrav1beta1.MaasMachine{}
	if err := r.Get(ctx, req.NamespacedName, maasMachine); err != nil {
		return ctrl.Result{}, client.IgnoreNotFound(err)
	}

	// 建立 clusterScope 與 machineScope（封裝 MAAS API 所需上下文）
	machineScope, err := scope.NewMachineScope(scope.MachineScopeParams{ ... })

	// 離開時自動 Patch MaasMachine 狀態
	defer func() { machineScope.Close() }()

	// 依 DeletionTimestamp 分流：刪除或正常 reconcile
	if !maasMachine.DeletionTimestamp.IsZero() {
		return r.reconcileDelete(ctx, machineScope, clusterScope)
	}
	return r.reconcileNormal(ctx, machineScope, clusterScope)
}
```

### 原始碼：reconcileNormal 關鍵流程

```go
// 原始碼: controllers/maasmachine_controller.go

func (r *MaasMachineReconciler) reconcileNormal(_ context.Context, machineScope *scope.MachineScope, clusterScope *scope.ClusterScope) (ctrl.Result, error) {
	// Bootstrap Guard：等待 cloud-init userdata 就緒才能繼續
	if machineScope.Machine.Spec.Bootstrap.DataSecretName == nil {
		conditions.MarkFalse(machineScope.MaasMachine, infrav1beta1.MachineDeployedCondition,
			infrav1beta1.WaitingForBootstrapDataReason, clusterv1.ConditionSeverityInfo, "")
		return ctrl.Result{}, nil
	}

	// 查找現有機器；若不存在或狀態不對，觸發 deployMachine（allocate + deploy）
	m, _ := r.findMachine(machineScope, machineSvc)
	if m == nil || !(m.State == infrav1beta1.MachineStateDeployed || m.State == infrav1beta1.MachineStateDeploying) {
		m, err = r.deployMachine(machineScope, machineSvc)
	}

	// 根據 MAAS 機器狀態更新 Kubernetes Condition
	switch s := m.State; {
	case s == infrav1beta1.MachineStateDeploying, s == infrav1beta1.MachineStateAllocated:
		machineScope.SetNotReady()
		conditions.MarkFalse(machineScope.MaasMachine, infrav1beta1.MachineDeployedCondition,
			infrav1beta1.MachineDeployingReason, clusterv1.ConditionSeverityWarning, "")
	case s == infrav1beta1.MachineStateDeployed:
		machineScope.SetReady()
		conditions.MarkTrue(machineScope.MaasMachine, infrav1beta1.MachineDeployedCondition)
	}

	// 設定 DNS（僅 Control Plane）
	r.reconcileDNSAttachment(machineScope, clusterScope, m)

	// 等 API Server 上線後，patch Node.Spec.ProviderID
	if err := machineScope.SetNodeProviderID(); err != nil {
		return ctrl.Result{RequeueAfter: 30 * time.Second}, nil
	}
	return ctrl.Result{}, nil
}
```

## MAAS Machine 狀態說明

MAAS 內部為每台機器維護一個狀態，CAPM3 在每次 reconcile 時會讀取此狀態並據此行動。

| 狀態 | MAAS 意義 | CAPM3 觀察到時的動作 |
|------|-----------|---------------------|
| `New` | 機器已登記但未分配，可被分配 | 可作為候選機器 |
| `Allocated` | 已被 CAPM3 分配，等待部署 | 觸發 disable swap + 開始部署 |
| `Deploying` | MAAS 正在安裝 OS | 輪詢等待，設定 `MachineDeployed` condition = False |
| `Deployed` | OS 安裝完成，機器已上線 | 設定 addresses、DNS（CP）、ProviderID、status.ready = true |
| `Ready` | 機器正常運行（已加入 K8s 叢集）| 正常穩定狀態 |
| `DiskErasing` | 磁碟清除中（釋放流程的一部分）| 等待清除完成 |
| `Releasing` | 機器正在釋放回 MAAS 資源池 | 等待釋放完成 |

## 建立流程（Create Flow）詳解

![MAAS 完整部署流程](/diagrams/maas/provisioning-flow.png)

當 CAPI Core 建立 `MaasMachine` 物件後，`MaasMachineReconciler` 開始以下流程：

### Step 1 — Bootstrap 就緒守衛

`MaasMachineReconciler` 首先檢查 `Machine.Spec.Bootstrap.DataSecretName` 是否已填入。若未填入，代表 Bootstrap Provider（例如 `KubeadmConfigController`）尚未完成 cloud-init userdata 的準備。

此時 controller 直接 return，等待下次被觸發——Bootstrap Provider 完成後會更新 Machine 物件，進而觸發下一次 reconcile。


<Callout type="info" title="Bootstrap Guard 模式">

這是 CAPI 的「Bootstrap Guard」模式：Infrastructure Provider 必須等 Bootstrap Provider 準備好 userdata 後，才能進行 deploy。此設計確保部署順序正確，避免機器在沒有 cloud-init 的情況下被部署出去成為無法加入叢集的裸機。

</Callout>


---

### Step 2 — Allocate 機器

Controller 檢查 `MaasMachine.Status.SystemID` 是否已填入：

- **若未填入**：呼叫 MAAS API `POST /api/2.0/machines/?op=allocate`
- **傳入篩選條件**（對應 `MaasMachine.Spec`）：

| MAAS 參數 | 對應 Spec 欄位 |
|-----------|----------------|
| `min_cpu_count` | `spec.minCPU` |
| `mem` | `spec.minMemoryInMB` |
| `tags` | `spec.tags` |
| `pool` | `spec.resourcePool` |
| `zone` | `spec.failureDomain` |

MAAS 回傳分配到的機器資訊（`system_id`、`hostname`、`zone` 等），Controller 將 `system_id` 寫入 `MaasMachine.Status.SystemID` 以防止重複分配。機器狀態進入 `Allocated`。

---

### Step 3 — Disable Swap

分配完成後，Controller 呼叫 MAAS API 對目標機器停用 swap。此步驟在 deploy 之前執行，目的是防止部署後 kubelet 因 swap 啟用而啟動失敗（Kubernetes 預設不允許 swap）。

---

### Step 4 — Deploy 機器

呼叫 MAAS API `POST /api/2.0/machines/&#123;&#123;system_id&#125;&#125;/?op=deploy`，傳入以下參數：

| 參數 | 來源 |
|------|------|
| `distro_series` | `spec.image`（OS image 名稱） |
| `user_data` | base64 編碼的 cloud-init userdata（從 Bootstrap Secret 讀取）|
| `hwe_kernel` | `spec.kernel`（若有設定，指定硬體啟用核心）|
| `ephemeral_deploy` | `spec.deployInMemory = true` 時啟用 in-memory 部署模式 |

觸發後機器狀態進入 `Deploying`，MAAS 開始安裝作業系統。

---

### Step 5 — 輪詢等待 Deployed

由於 OS 安裝需要數分鐘，Controller 在每次 reconcile 時：

1. 呼叫 `GET /api/2.0/machines/&#123;&#123;system_id&#125;&#125;/` 取得最新狀態
2. 若 `status != Deployed`：設定 `MachineDeployed` condition = False，return 並等待下次 reconcile（由 `requeueAfter` 觸發）
3. 若 `status = Deployed`：繼續執行後續步驟

---

### Step 6 — 取得 IP Addresses

OS 安裝完成後，從 MAAS machine 資訊的 `ip_addresses` 欄位讀取所有分配到的 IP，填入 `MaasMachine.Status.Addresses`（Type: `InternalIP`）。這些 IP 會被 CAPI Core 同步到 Machine 物件，供上層元件（如 kubeconfig 產生）使用。

---

### Step 7 — DNS 登記（僅 Control Plane 機器）

Controller 透過 `cluster.x-k8s.io/control-plane` label 判斷此 Machine 是否為 Control Plane 節點：

- **若為 CP 節點**：呼叫 DNS API，將此機器的 IP 登記到 `<cluster-name>.<dnsDomain>` 的 A record，並設定 `DNSAttached` condition = True
- **若為 Worker 節點**：略過此步驟


<Callout type="warning" title="Worker 節點不登記 DNS">

只有 Control Plane 節點才需要 DNS 登記，因為 kube-apiserver 的端點（endpoint）必須透過固定的 DNS 名稱對外提供。Worker 節點只需要 IP，由 kubelet 自行向 API Server 註冊，不需要 DNS A record。

</Callout>


---

### Step 8 — 設定 ProviderID 並標記就緒

最後一步完成所有收尾動作：

1. 設定 `MaasMachine.Spec.ProviderID = "maas://<systemID>"`
2. 透過 `ClusterCacheTracker` 取得 workload cluster 的 Kubernetes client
3. 根據 MAAS hostname 找到對應的 K8s Node 物件
4. Patch Node 的 `Spec.ProviderID = "maas://<systemID>"`（與 cloud-controller-manager 整合所需）
5. 設定 `MachineDeployed` condition = True
6. 設定 `MaasMachine.Status.Ready = true`

至此，Machine 進入 `Ready` 狀態，CAPI Core 將此 Machine 標記為 `Ready`，MachineDeployment 可繼續滾動下一台。

## 刪除流程（Delete Flow）詳解

**觸發條件**：`MaasMachine` 物件被刪除，`DeletionTimestamp` 填入後，Reconciler 進入刪除流程。

### Step 1 — CP DNS 移除（若為 CP 機器）

若此 Machine 為 Control Plane 節點，Controller 先呼叫 DNS API 移除對應的 A record，並等待 `DNSAttached` condition 清除後才繼續。

### Step 2 — Release 機器

呼叫 MAAS API `POST /api/2.0/machines/&#123;&#123;system_id&#125;&#125;/?op=release`，MAAS 收到請求後：

1. 機器進入 `DiskErasing` 狀態——MAAS 清除磁碟上的所有資料
2. 清除完成後進入 `Releasing` 狀態

### Step 3 — 等待釋放完成

Controller 輪詢等待機器狀態回到 `New`（代表機器已可被再次分配）。確認完成後，移除 finalizer `maasmachine.infrastructure.cluster.x-k8s.io`，讓 Kubernetes 完成物件的刪除。

## 錯誤處理與重試

MAAS Provider 採用 condition-based 錯誤通報機制：

- **所有 MAAS API 錯誤**都會設定對應的 Kubernetes condition（例如 `MachineDeployFailed`），讓 `kubectl get maasmachine -o yaml` 即可看到明確的錯誤訊息
- **Controller 透過 `ctrl.Result{RequeueAfter: ...}`** 自動排程重試，無需人工介入
- **Allocate 失敗**（例如：無符合條件的可用機器）會設定 condition 並持續重試，直到有機器釋放出來
- **Deploy 失敗**（例如：指定的 image 不存在於 MAAS）會設定 condition 並將 Machine 標記為 `Failed`，需人工修正 spec 後才能繼續

## deployInMemory 模式說明


<Callout type="tip" title="deployInMemory — In-Memory 部署模式">

當 `spec.deployInMemory: true` 時，MAAS 會以 `ephemeral_deploy` 模式部署機器，OS 完全在 RAM 中運行，不寫入磁碟。

**優點：**
- 極快速的節點上線時間（省去磁碟寫入）
- 無磁碟磨損，延長硬體壽命
- 關機後自動清除，天然隔離

**前提條件：**
- MAAS 版本 ≥ 3.5.10
- 目標機器至少 16GB RAM（需容納整個 OS）

**適合場景：**
- CI/CD 環境（每次 pipeline 使用乾淨節點）
- 短暫測試叢集
- 需要快速替換節點的場景（ephemeral workloads）

</Callout>

## FailureDomain 整合

CAPI 的 `FailureDomain` 概念在 MAAS Provider 中對應到 **MAAS Zone**：

- **`MaasMachine.spec.failureDomain`**：指定此機器必須分配在哪個 MAAS zone，allocate 時會傳入 `zone` 參數給 MAAS API
- **`MaasCluster.spec.failureDomains`**：列出叢集可用的 zone 清單，CAPI Core 會在建立 Machine 時參考此清單
- **`MachineDeployment` 分散部署**：CAPI 核心的 `MachineDeployment` 可透過 `failureDomains` 將多台機器分散到不同 zone，實現跨機房或跨機架的高可用部署

```yaml
# MaasCluster 設定可用 zone
spec:
  failureDomains:
    zone-a:
      controlPlane: true
    zone-b:
      controlPlane: true

# MaasMachine 指定目標 zone
spec:
  failureDomain: zone-a
```

透過此機制，使用者無需手動指定機器，只需定義 zone 偏好，MAAS Provider 會自動在正確 zone 中選取並分配可用機器。
