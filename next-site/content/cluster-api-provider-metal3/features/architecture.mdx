## 🌐 整體觀念：CAPM3 解決什麼問題？

CAPM3（Cluster API Provider Metal3）的核心設計原則是**不直接控制硬體**。它作為 CAPI 和裸機世界之間的「翻譯官」，形成清晰的三層責任分工：

<Callout type="info" title="概念類比：外賣平台的三層架構">

- **使用者（CAPI 層）**：透過 App 下單，說「我要 3 台 Worker 節點、2 核 8G」——不需要知道後廚如何運作
- **外賣平台（CAPM3 層）**：接收訂單，翻譯成「請備好 image A、networkdata B」，協調廚房
- **廚房（BMO + Ironic 層）**：真正去開火做菜，控制 PXE、IPMI、Redfish

</Callout>

![CAPM3 架構圖](/diagrams/metal3/architecture.png)

| 層次 | 物件 | 做什麼 |
|------|------|--------|
| CAPI 層 | `Cluster`、`Machine`、`MachineDeployment` | 宣告式叢集管理介面——「我要幾台、什麼規格」|
| CAPM3 層 | `Metal3Cluster`、`Metal3Machine`、`Metal3Data` | 橋接層——選 BMH、渲染設定模板、追蹤狀態 |
| Metal3 層 | `BareMetalHost`、Ironic | 實際執行——PXE boot、IPMI 控制、OS 部署 |

---

## 🔄 核心流程

建立一台 Worker 節點的完整流程：

1. **Metal3MachineReconciler** 執行 chooseHost，依優先順序（node-reuse > ready > available）鎖定一台 BareMetalHost
2. **Metal3DataTemplateReconciler** 偵測到 Metal3DataClaim，建立 Metal3Data 實例
3. **Metal3DataReconciler** 渲染 Jinja2 模板，產生機器專屬的 metadata/networkdata Secret
4. **Metal3MachineReconciler** 設定 BMH spec（image、userData、metaData、networkData），BMO 開始驅動 Ironic 執行 PXE boot
5. 等待 BMH 狀態從 `provisioning` 變為 `provisioned`（Ironic 負責 OS 安裝）
6. **Metal3MachineReconciler** 設定 `Metal3Machine.Spec.ProviderID`，patch Node 的 `spec.providerID`，設定 `status.Ready = true`
7. **Metal3LabelSyncReconciler** 將 BMH 的硬體 labels 同步至 Kubernetes Node

## 🧩 Controller 與 CRD 總覽

| Controller / CRD | 用途 |
|-----------------|------|
| `Metal3ClusterReconciler` | Pass-through：讀取 `controlPlaneEndpoint`，設 `status.ready=true`（不管理 LB）|
| `Metal3MachineReconciler` | 最複雜：chooseHost、設定 BMH spec、等待 provisioned、設 ProviderID |
| `Metal3DataReconciler` | 渲染 Jinja2 模板，產生每台機器的 metadata/networkdata Secret |
| `Metal3DataTemplateReconciler` | 監控 Metal3DataClaim，為每個 Metal3Machine 建立 Metal3Data |
| `Metal3MachineTemplateReconciler` | 管理 Node Reuse labels，支援滾動升級時裸機重用 |
| `Metal3LabelSyncReconciler` | BMH labels 單向同步到 Kubernetes Node |
| `Metal3RemediationReconciler` | 執行 Reboot 策略（由 MachineHealthCheck 觸發）|
| `BareMetalHost` CRD | 代表一台實體機器，由 BMO 管理狀態機 |
| `Metal3Data` CRD | 每台機器的網路設定渲染結果 |
| `Metal3Remediation` CRD | 描述故障修復策略（reboot 類型、重試次數）|

## ⚠️ 常見陷阱

<Callout type="warning" title="常見誤解">

- **CAPM3 不直接控制 IPMI/Redfish**：實際硬體操作由 BMO 透過 Ironic 執行
- **Metal3Cluster 不建立 Load Balancer**：`controlPlaneEndpoint` 須由使用者或 keepalived/kube-vip 預先設定
- **七個 Controller 是非同步並行的**，不是串行執行
- **ProviderID 格式固定**：`metal3://<namespace>/<bmh-name>/<metal3machine-name>`，cloud-provider-metal3 依賴此格式

</Callout>
