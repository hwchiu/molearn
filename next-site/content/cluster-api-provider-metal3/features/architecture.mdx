## 🌐 整體觀念：CAPM3 解決什麼問題？

CAPM3（Cluster API Provider Metal3）的核心設計原則是**不直接控制硬體**。它作為 CAPI 和裸機世界之間的「翻譯官」，形成清晰的三層責任分工：

<Callout type="info" title="概念類比：外賣平台的三層架構">

把 CAPM3 想像成一個外賣平台系統：

- **使用者（CAPI 層）**：透過 App 下單，說「我要 3 台 Worker 節點、2 核 8G」——不需要知道後廚如何運作
- **外賣平台（CAPM3 層）**：接收訂單，翻譯成「請備好 image A、networkdata B」，協調廚房
- **廚房（BMO + Ironic 層）**：真正去開火做菜，控制 PXE、IPMI、Redfish

CAPM3 的價值在於「橋接」——讓 Kubernetes 原生的 CAPI 介面能夠驅動完全不同技術棧的裸機硬體。

</Callout>

![CAPM3 架構圖](/diagrams/metal3/architecture.png)

**三層職責：**

| 層次 | 物件 | 做什麼 |
|------|------|--------|
| CAPI 層 | `Cluster`、`Machine`、`MachineDeployment` | 宣告式叢集管理介面——「我要幾台、什麼規格」|
| CAPM3 層 | `Metal3Cluster`、`Metal3Machine`、`Metal3Data` | 橋接層——選 BMH、渲染設定模板、追蹤狀態 |
| Metal3 層 | `BareMetalHost`、Ironic | 實際執行——PXE boot、IPMI 控制、OS 部署 |

這個設計讓 CAPM3 保持輕量，無需了解 Ironic 的細節，只需操作 Kubernetes API（BMH CRD）即可完成裸機生命週期管理。

## 🔄 核心流程：追蹤一台裸機從選定到就緒

> 💡 **場景**：你透過 CAPI 建立了一個 `Machine` 物件，要新增一台 Worker 節點。CAPM3 的七個 Controller 接下來如何分工協作，讓這台裸機變成一個可用的 Kubernetes Node？

**Step 1 — Metal3MachineReconciler：選機（chooseHost）**

`Metal3MachineReconciler` 收到新的 `Metal3Machine`，啟動 chooseHost 邏輯：
- 依優先順序掃描可用的 BareMetalHost：先找有 `node-reuse` label 的（升級重用），再找 `ready`，最後找 `available`
- 找到後立即設定 `BMH.Spec.ConsumerRef` 指向此 Metal3Machine，「鎖住」這台機器

**Step 2 — Metal3DataTemplateReconciler：請求資料渲染**

`Metal3Machine` 建立時同時產生一個 `Metal3DataClaim`。`Metal3DataTemplateReconciler` 偵測到這個 Claim，依照指定的 `Metal3DataTemplate` 建立 `Metal3Data` 實例。

**Step 3 — Metal3DataReconciler：渲染網路設定**

![DataTemplate 渲染流程](/diagrams/metal3/data-template-flow.png)

`Metal3DataReconciler` 讀取 `Metal3DataTemplate`，用 Jinja2 模板填入這台機器的 IP、MAC、hostname 等資訊，渲染結果寫入 Kubernetes Secret。每台機器都有獨立的網路設定，互不干擾。

**Step 4 — Metal3MachineReconciler：觸發 Provisioning**

所有設定準備好後，`Metal3MachineReconciler` 設定 BMH 的完整 spec：
- `spec.image`：OS image URL
- `spec.userData`：cloud-init bootstrap 腳本
- `spec.metaData` / `spec.networkData`：Step 3 產生的 Secrets

BMO 偵測到 `spec.image` 被設定，立刻開始驅動 Ironic 執行 PXE boot（BMH 狀態：`available` → `provisioning`）。

**Step 5 — 等待 BMO + Ironic 完成**

這段時間 CAPM3 退到後台，每 30 秒輪詢 `BMH.Status.Provisioning.State`。Ironic 透過 PXE 開機、寫入磁碟、安裝 OS，完成後 BMH 狀態變為 `provisioned`。

**Step 6 — Metal3MachineReconciler：設定 ProviderID，通知 CAPI 就緒**

`Metal3MachineReconciler` 發現 BMH 已 `provisioned`：
- 設定 `Metal3Machine.Spec.ProviderID`：`metal3://<namespace>/<bmh-name>/<m3m-name>`
- 透過 ClusterCacheTracker 取得 workload cluster 的 client，patch Node 的 `spec.providerID`
- 設定 `Metal3Machine.Status.Ready = true`，通知 CAPI Machine 已就緒

**Step 7 — Metal3LabelSyncReconciler：同步硬體資訊到 Node**

`Metal3LabelSyncReconciler` 持續監控，將 BMH 上帶有 `metal3.io` 前綴的 labels 單向同步到 Kubernetes Node，讓 Scheduler 可以感知底層硬體特性（機架位置、CPU 型號、NIC 規格等）。

## 🧩 元件詳解

### 七個 Controller 職責總覽

![chooseHost 決策流程](/diagrams/metal3/choosehost-flow.png)

| Controller | 核心職責 |
|-----------|---------|
| `Metal3ClusterReconciler` | Pass-through 模式，讀取 `controlPlaneEndpoint`，設 `status.ready=true` |
| `Metal3MachineReconciler` | 最複雜：chooseHost、設定 BMH spec、等待 provisioned、設 ProviderID |
| `Metal3DataReconciler` | 渲染 Jinja2 模板，產生每台機器專屬的 metadata/networkdata Secrets |
| `Metal3DataTemplateReconciler` | 監控 Metal3DataClaim，為每個 Metal3Machine 建立 Metal3Data 實例 |
| `Metal3MachineTemplateReconciler` | 管理 Node Reuse labels，支援滾動升級時同一台裸機重用 |
| `Metal3LabelSyncReconciler` | BMH labels 單向同步到 Kubernetes Node |
| `Metal3RemediationReconciler` | 執行 Reboot 策略（由 MachineHealthCheck 觸發） |

### Metal3ClusterReconciler

**pass-through 模式**：Metal3 本身**不管理 Load Balancer**。`controlPlaneEndpoint` 必須由使用者或外部工具（keepalived、kube-vip）預先設定好，CAPM3 只負責讀取並回報 `status.ready = true`。

### Metal3MachineReconciler — chooseHost 邏輯

選機優先順序（高到低）：

1. **節點重用（Node Reuse）**：帶有 `infrastructure.cluster.x-k8s.io/node-reuse` label 的 BMH（適用滾動升級）
2. **Name-Based Preallocation**：若啟用 `enableBMHNameBasedPreallocation`，依名稱規則比對
3. **Ready 狀態的 BMH**：無 `errorMsg`、無 pause annotation、無 unhealthy annotation
4. **Available 狀態的 BMH**：同上篩選條件

### Metal3DataReconciler — 模板渲染

從 `Metal3DataTemplate` 讀取模板，填入每台機器的變數（IP、MAC、hostname、index），渲染結果寫入 Kubernetes Secret。BMH 透過 `spec.metaData.secretRef` 與 `spec.networkData.secretRef` 引用。這個設計讓每台機器的網路設定高度客製化，同時保持模板統一管理。

### Metal3RemediationReconciler — 故障自動修復

與 CAPI 整合流程：
1. `MachineHealthCheck` 偵測到不健康的機器（Node NotReady 等）
2. 建立 `Metal3Remediation` 物件，指定 remediation 策略
3. `Metal3RemediationReconciler` 讀取策略，對對應 BMH 執行 reboot 操作
4. 等待機器重新上線，更新 remediation 狀態

### ProviderID 格式

<Callout type="info" title="ProviderID 說明">

**格式**：

```
metal3://<namespace>/<bmh-name>/<metal3machine-name>
```

**範例**：

```
metal3://default/worker-0/my-cluster-worker-0
```

**設定時機**：由 `Metal3MachineReconciler` 在 BMH provisioning 完成後，寫入 `Metal3Machine.spec.providerID`，CAPI 再將此值同步至 `Node.spec.providerID`。

**用途**：此格式讓 `cloud-provider-metal3` 可以透過 ProviderID 識別 Kubernetes Node 所對應的 BMH，進而實現 Node 的雲端整合功能（如 CCM 節點生命週期管理）。

</Callout>

### 目錄結構

```
cluster-api-provider-metal3/
├── api/v1beta1/              # 所有 CRD 定義（9 個）
├── baremetal/                # Metal3Machine controller 核心邏輯
│   ├── metal3machine_manager.go
│   ├── metal3data_manager.go
│   ├── metal3cluster_manager.go
│   └── baremetalhost.go      # BMH 操作封裝
├── controllers/              # Controller 入口（reconcile loops）
├── config/                   # RBAC、webhook、CRD manifests
├── test/                     # e2e 和 unit tests
└── hack/                     # 開發工具腳本
```

`baremetal/` 目錄是 CAPM3 的核心業務邏輯層，`controllers/` 則是薄薄的進入點，負責監聽 Kubernetes 事件並呼叫 `baremetal/` 中的 manager 函式。

### 環境與相依

| 相依 | 版本要求 | 說明 |

## ⚠️ 常見陷阱與注意事項

<Callout type="warning" title="常見誤解">

**誤解 1：「CAPM3 直接控制 IPMI/Redfish」**
→ 不對。CAPM3 只操作 Kubernetes API（BMH CRD），實際的 IPMI/Redfish 操作由 BMO 透過 Ironic 執行。CAPM3 不需要知道硬體的 IP 或憑證。

**誤解 2：「Metal3Cluster 負責建立負載均衡器（Load Balancer）」**
→ 不對。`controlPlaneEndpoint` 必須由使用者或外部工具（keepalived、kube-vip）預先設定好，CAPM3 只讀取並回報狀態，不建立 LB。

**誤解 3：「七個 Controller 是串行執行的」**
→ 每個 Controller 都有獨立的 reconcile loop，互相非同步執行。`Metal3DataReconciler` 和 `Metal3MachineReconciler` 同時監控各自的物件，由 Kubernetes 的 informer 機制觸發。

**誤解 4：「ProviderID 是可以自定義格式的」**
→ Metal3 的 ProviderID 格式是固定的（`metal3://<namespace>/<bmh-name>/<metal3machine-name>`），cloud-provider-metal3 依賴此格式識別 Node 與 BMH 的對應關係。

</Callout>
