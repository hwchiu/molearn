## 什麼是 BareMetalHost（BMH）

**BareMetalHost（BMH）** 是由 [metal3-io/baremetal-operator](https://github.com/metal3-io/baremetal-operator) 定義的 CRD，代表一台真實的裸機伺服器。

- **BMO（Bare Metal Operator）** 負責管理 BMH，透過 Ironic 執行實際的硬體操作（PXE 開機、IPMI 電源控制、Redfish API 呼叫等）
- **CAPM3** 透過修改 BMH 的 `spec` 欄位來「請求」BMO 執行特定動作，自身不直接操作硬體
- **BMH 的狀態**（`status.provisioning.state`）反映裸機的實際狀態，由 BMO 負責維護與更新

![BMH 生命週期狀態圖](/diagrams/metal3/bmh-lifecycle.png)

## BMH 狀態機（State Machine）

BMH 的 `status.provisioning.state` 欄位記錄裸機當前所處的生命週期階段，所有狀態由 BMO 管理。

| 狀態 | 說明 | 轉換觸發條件 |
|------|------|------------|
| `registering` | BMH 剛建立，BMO 正在初始化 IPMI/BMC 連線 | 自動 |
| `inspecting` | BMO 透過 Ironic 取得硬體資訊（CPU、記憶體、磁碟） | 自動 |
| `available` / `ready` | 已完成 introspection，可被 CAPM3 選取 | introspection 完成 |
| `provisioning` | Ironic 正在安裝 OS（PXE boot + 寫入磁碟） | `BMH.spec.image` 被設定 |
| `provisioned` | OS 已安裝完成，機器已上線運行 | provisioning 完成 |
| `deprovisioning` | Ironic 正在清除磁碟、重置機器 | `BMH.spec.image` 被清除 |
| `available` | 釋放完成，可再次被選取 | deprovisioning 完成 |
| `deleting` | BMH 資源被刪除中 | BMH DeletionTimestamp 設定 |
| `error` | 發生無法自動恢復的錯誤 | 各階段錯誤 |
| `paused` | BMH 被暫停，不接受任何操作 | 設定 pause annotation |


<Callout type="warning" title="BMH 狀態由 BMO 控制">

BMH 的狀態轉換完全由 **BMO** 負責管理，CAPM3 只能透過修改 BMH 的 `spec` 欄位（如設定或清除 `spec.image`）來**間接**影響狀態轉換。CAPM3 無法直接強制改變 BMH 的 provisioning state。

</Callout>

## CAPM3 與 BMH 的協作流程（建立機器）

當 CAPI 建立 `Machine` 物件後，`Metal3MachineReconciler` 會執行以下七個步驟：

### Step 1 — 選擇 BMH（chooseHost）

CAPM3 在以下條件下選擇 BMH：

- BMH 在相同 namespace
- `BMH.Status.Provisioning.State` 是 `available` 或 `ready`
- 沒有 `errorMessage`
- 沒有 pause annotation
- 沒有 unhealthy annotation
- `BMH.Spec.ConsumerRef` 為空（未被其他 Metal3Machine 佔用）

**選擇優先順序：**

1. 帶有 `infrastructure.cluster.x-k8s.io/node-reuse` label 的 BMH（升級重用）
2. Name-Based Preallocation 預分配的 BMH（若啟用）
3. 任意 `ready` 狀態 BMH
4. 任意 `available` 狀態 BMH

### Step 2 — 設定 ConsumerRef

- 設定 `BMH.Spec.ConsumerRef` 指向對應的 `Metal3Machine`
- 這標記 BMH 為「已被佔用」，防止其他 `Metal3Machine` 選取同一台 BMH

### Step 3 — 設定 Image

設定 `BMH.Spec.Image` 包含以下資訊：

| 欄位 | 說明 |
|------|------|
| `url` | OS image 下載 URL |
| `checksum` | image checksum 值 |
| `checksumType` | 校驗演算法：`md5` / `sha256` / `sha512` |
| `format` | image 格式：`raw` / `qcow2`（可選） |

BMO 偵測到 `spec.image` 被設定後，即開始 provisioning 流程（BMH 狀態轉為 `provisioning`）。

### Step 4 — 設定 UserData

- 讀取 CAPI Bootstrap Secret（由 `KubeadmConfig` 產生的 cloud-init 內容）
- 建立新 Secret 並設定到 `BMH.Spec.UserData`
- Ironic 在 provisioning 時將 userdata 傳入機器，供 cloud-init 在首次開機時執行（加入節點至 K8s 叢集）

### Step 5 — 設定 MetaData 與 NetworkData

若 `Metal3Machine` 有 `dataTemplate` 引用：

- `Metal3DataReconciler` 渲染 Jinja2 模板，產生 metadata/networkdata 內容
- 建立對應的 Kubernetes Secrets
- 設定到 `BMH.Spec.MetaData` 和 `BMH.Spec.NetworkData`

| 資料類型 | 說明 |
|---------|------|
| **Metadata** | 機器特定的設定（hostname、UUID 等） |
| **NetworkData** | NIC 設定（IP 位址、gateway、DNS 伺服器） |

### Step 6 — 等待 Provisioned

- CAPM3 輪詢 `BMH.Status.Provisioning.State`
- 若不是 `provisioned` → 等待下次 reconcile 觸發
- 若是 `provisioned` → 繼續執行 Step 7

### Step 7 — 設定 ProviderID 並標記就緒

- 從 BMH status 取得 IP addresses → 設定 `Metal3Machine.Status.Addresses`
- 設定 ProviderID 格式：`metal3://<namespace>/<bmh-name>/<m3m-name>`
- 透過 `ClusterCacheTracker` 取得 workload cluster 的 Kubernetes client
- Patch workload cluster 對應 Node 的 `Spec.ProviderID`
- 設定 `Metal3Machine.Status.Ready = true`，通知 CAPI Machine 已就緒

## CAPM3 與 BMH 的協作流程（刪除機器）

當 CAPI 刪除 `Machine` 物件後，`Metal3MachineReconciler` 會執行以下三個步驟：

### Step 1 — 清除 BMH Image 設定

- 清除 `BMH.Spec.Image`（設為 nil）
- 清除 `BMH.Spec.UserData`
- 清除 `BMH.Spec.MetaData` 和 `BMH.Spec.NetworkData`
- BMO 偵測到 `spec.image` 被清除後，開始 deprovisioning 流程（BMH 狀態轉為 `deprovisioning`）

### Step 2 — 等待 Deprovisioned

- 輪詢 `BMH.Status.Provisioning.State`
- 等待狀態回到 `available` 或 `ready`

### Step 3 — 清除 ConsumerRef

- 清除 `BMH.Spec.ConsumerRef`（釋放 BMH，使其可被其他機器選取）
- 移除 `Metal3Machine` finalizer，允許物件被 Kubernetes GC 回收

---

## automatedCleaningMode 設定

`Metal3MachineTemplate.spec.template.spec.automatedCleaningMode` 控制 Ironic 在 deprovisioning 時的磁碟清理行為：

| 值 | 說明 |
|----|------|
| `metadata`（預設） | Ironic 執行完整的 disk cleaning（清除所有資料），確保資料安全 |
| `disabled` | 跳過 disk cleaning，大幅加速機器釋放速度 |


<Callout type="tip" title="Node Reuse 場景建議">

在 **Node Reuse**（節點升級重用）場景中，建議將 `automatedCleaningMode` 設為 `disabled`。由於升級時會重用同一台裸機，省略 disk cleaning 可以顯著縮短升級所需時間，加快整個升級流程。

</Callout>

## BMH pause 機制

在 BMH 加上以下 annotation 可暫停 BMO 對此 BMH 的所有操作：

```
baremetalhost.metal3.io/paused: ""
```

**使用場景：**

- CAPM3 在某些操作（如設定 `ConsumerRef`）時可能暫時 pause BMH，以確保設定的原子性，避免 BMO 在設定過程中提前觸發狀態轉換
- 管理員可手動 pause BMH 以進行維護操作


<Callout type="warning" title="注意事項">

應避免手動 pause 正在 `provisioning` 或 `deprovisioning` 狀態的 BMH。暫停進行中的操作可能導致 Ironic 任務懸空，需要手動干預才能恢復正常狀態。

</Callout>

## 深度解析：chooseHost 演算法

`chooseHost` 是 CAPM3 最核心的選機邏輯，定義在 `baremetal/metal3machine_manager.go`。這個函式決定哪台 BMH 會被指派給某個 `Metal3Machine`。

### 常數定義

```go
// baremetal/metal3machine_manager.go
const (
    // HostAnnotation 記錄在 Metal3Machine 上，指向對應的 BareMetalHost
    HostAnnotation = "metal3.io/BareMetalHost"

    // nodeReuseLabelName 是啟用 Node Reuse 時設在 BMH 上的 label key
    nodeReuseLabelName = "infrastructure.cluster.x-k8s.io/node-reuse"

    // 所有 requeue 的等待間隔固定為 30 秒
    requeueAfter = time.Second * 30

    // PausedAnnotationKey 是 CAPM3 寫入 BMH pause annotation 的 value
    PausedAnnotationKey = "metal3.io/capm3"
)
```

### 篩選候選 BMH

函式首先透過 `hostLabelSelectorForMachine` 將 `Metal3Machine.spec.hostSelector` 轉成 Kubernetes label selector，再用 `client.List` 一次撈出符合條件的所有 BMH：

```go
func (m *MachineManager) chooseHost(ctx context.Context) (
    *bmov1alpha1.BareMetalHost, *patch.Helper, string, error) {

    labelSelector, err := hostLabelSelectorForMachine(m.Metal3Machine, m.Log)
    hosts := bmov1alpha1.BareMetalHostList{}
    m.client.List(ctx, &hosts,
        client.InNamespace(m.Metal3Machine.Namespace),
        client.MatchingLabelsSelector{Selector: labelSelector},
    )
    // ...
}
```

接著逐一掃描每台 BMH，遇到以下任一情況就直接 `continue` 跳過：

| 跳過條件 | 說明 |
|---------|------|
| `ConsumerRef != nil` 且不指向此 Metal3Machine | BMH 已被其他機器佔用 |
| `nodeReuseLabelName` 存在但不匹配當前機器 | Node Reuse label 屬於另一個 ControlPlane / MachineDeployment |
| `DeletionTimestamp != nil` | BMH 正在被刪除 |
| `Status.ErrorMessage != ""` | BMH 有錯誤訊息 |
| 有 `baremetalhost.metal3.io/paused` annotation | BMH 已被暫停 |
| 有 `capm3.metal3.io/unhealthy` annotation | BMH 被標記為不健康 |
| 有 `capi.metal3.io/unhealthy` annotation（舊版）| 舊版不健康 annotation（已棄用） |
| 狀態不是 `ready` 或 `available` | 機器尚未準備好（`provisioning`、`inspecting` 等） |

通過篩選後，BMH 會被分入兩個候選清單：

- **`availableHostsWithNodeReuse`**：有 nodeReuse label 且 label 值符合當前機器的 BMH
- **`availableHosts`**：沒有 nodeReuse label 且狀態為 `ready` 或 `available` 的 BMH

### 優先選取邏輯（原始碼）

```go
// 優先選取有 nodeReuseLabelName 的 BMH
if len(availableHostsWithNodeReuse) != 0 {
    for _, host := range availableHostsWithNodeReuse {
        if host.Status.Provisioning.State == bmov1alpha1.StateReady ||
            host.Status.Provisioning.State == bmov1alpha1.StateAvailable {
            // 找到符合的 BMH 且已 Ready -> 選取並標記 NodeReuse 原因
            chosenHost, err = m.pickHost(hostsInAvailableStateWithNodeReuse)
            chooseHostReason = infrav1.AssociateBareMetalHostViaNodeReuseSuccessReason
        } else {
            // Node Reuse 的 BMH 存在但尚未 Ready -> 強制 requeue，不回落到一般 BMH
            return nil, nil, "", WithTransientError(errors.New(errMessage), requeueAfter)
        }
    }
} else {
    // 沒有 nodeReuse BMH -> 從一般候選清單隨機選取
    chosenHost, err = m.pickHost(availableHosts)
}
```


<Callout type="warning" title="Node Reuse 的「不回落」行為">

若存在帶有 `nodeReuseLabelName` 的 BMH，但該 BMH 尚未進入 `ready` 或 `available` 狀態（例如仍在 `deprovisioning`），CAPM3 **不會**回落選取一般的 BMH，而是直接回傳 `WithTransientError`，觸發 30 秒後的重試。這是為了確保升級場景下節點的穩定性，避免意外選取到不同台機器。

</Callout>


### 選取成功後的 ConsumerRef 綁定

一旦選定 BMH，CAPM3 會呼叫 `setHostConsumerRef` 將 BMH 的 `spec.consumerRef` 設定為：

```go
host.Spec.ConsumerRef = &corev1.ObjectReference{
    Kind:       "Metal3Machine",
    Name:       m.Metal3Machine.Name,
    Namespace:  m.Metal3Machine.Namespace,
    APIVersion: m.Metal3Machine.APIVersion,
}
```

同時也會**刪除** BMH 上的 `nodeReuseLabelName` label（若有），表示這台機器已從「待重用候選」進入「已分配」狀態。

## 深度解析：防止競爭條件的 Mutex 機制

在 Kubernetes controller-runtime 架構下，同一個 controller 可能同時處理多個 reconcile 請求（goroutine 並發）。若沒有保護機制，多個 `Metal3Machine` 可能在同一時間都選中同一台 `ready` 的 BMH。

CAPM3 使用一個**進程內全域 mutex** 解決此問題：

```go
// baremetal/metal3machine_manager.go
var (
    associateBMHMutex sync.Mutex
)

func (m *MachineManager) Associate(ctx context.Context) (string, error) {
    // 任何時刻只允許一個 goroutine 執行 BMH 選取與綁定
    associateBMHMutex.Lock()
    defer associateBMHMutex.Unlock()
    // ...
    host, helper, chosenHostReason, err = m.chooseHost(ctx)
    // ...
    m.setHostConsumerRef(ctx, host)
    // ...
}
```

**這個 mutex 的保護範圍涵蓋完整的 `Associate` 流程**，包括：選取 BMH、設定 `ConsumerRef`、設定 `Image`/`UserData`、Patch 到 API server。這確保在同一個 controller 進程內，不會有兩個 goroutine 同時搶奪同一台 BMH。


<Callout type="info" title="注意事項">

此 mutex 是進程內的記憶體鎖，對於**多副本部署**的場景（多個 CAPM3 controller pod），仍然依賴 Kubernetes 的 API server 樂觀鎖（`resourceVersion` 衝突偵測）來防止 Race Condition。當 Patch 發生衝突（`IsConflict` 錯誤）時，CAPM3 會觸發 30 秒後重試：

```go
if slices.ContainsFunc(aggr.Errors(), apierrors.IsConflict) {
    return "", WithTransientError(nil, requeueAfter)
}
```

</Callout>

## 深度解析：BMH Annotation 完整機制

CAPM3 在操作過程中會在 BMH 和 Metal3Machine 上設定多種 annotation。理解這些 annotation 是排障的關鍵。

### CAPM3 設定在 Metal3Machine 上的 Annotation

| Annotation Key | 設定時機 | 值 | 說明 |
|---------------|---------|-----|------|
| `metal3.io/BareMetalHost` | Associate 成功後 | `<namespace>/<bmh-name>` | 記錄此 Metal3Machine 對應的 BMH，是 `getHost()` 查詢的入口 |

`ensureAnnotation` 函式負責維護此 annotation：

```go
func (m *MachineManager) ensureAnnotation(_ context.Context, host *bmov1alpha1.BareMetalHost) error {
    annotations := m.Metal3Machine.ObjectMeta.GetAnnotations()
    hostKey, _ := cache.MetaNamespaceKeyFunc(host)  // 產生 "namespace/name" 格式
    annotations[HostAnnotation] = hostKey            // HostAnnotation = "metal3.io/BareMetalHost"
    m.Metal3Machine.ObjectMeta.SetAnnotations(annotations)
    return nil
}
```

### CAPM3 設定在 BMH 上的 Annotation

#### `baremetalhost.metal3.io/paused`（BMO 定義的 pause annotation）

CAPM3 在以下場景**設定** pause annotation：

- 當 `Cluster` 本身被 pause（例如升級暫停）時，CAPM3 呼叫 `SetPauseAnnotation`，將 BMH 也一起暫停

CAPM3 在以下場景**移除** pause annotation：

- 當 `Cluster` 恢復（unpause）時，呼叫 `RemovePauseAnnotation`
- 在 `Delete` 流程完成 deprovisioning 後，清除 pause

CAPM3 寫入的 pause annotation value 為固定字串 `"metal3.io/capm3"`（即 `PausedAnnotationKey` 常數），這讓 CAPM3 能識別是自己設定的 pause，而非管理員手動設定的：

```go
// 設定 pause
host.Annotations[bmov1alpha1.PausedAnnotation] = PausedAnnotationKey  // "metal3.io/capm3"

// 移除 pause：只移除 value 為 PausedAnnotationKey 的，不動管理員手動設定的
if annotations[bmov1alpha1.PausedAnnotation] == PausedAnnotationKey {
    delete(host.Annotations, bmov1alpha1.PausedAnnotation)
} else {
    m.Log.Info("BareMetalHost is paused by user, not removing pause annotation")
}
```

#### `baremetalhost.metal3.io/status`（BMO 定義的 status annotation）

在 `SetPauseAnnotation` 時，CAPM3 同時將**當時的 BMH status** 序列化（JSON，去除 `hardware` 欄位以減少大小）寫入此 annotation：

```go
newAnnotation, _ := json.Marshal(&host.Status)
obj := map[string]any{}
json.Unmarshal(newAnnotation, &obj)
delete(obj, "hardware")          // 去除 hardware inventory（太大）
newAnnotation, _ = json.Marshal(obj)
host.Annotations[bmov1alpha1.StatusAnnotation] = string(newAnnotation)
```

這個 annotation 讓 BMO 在 cluster 遷移或 pivot 場景下能夠恢復 BMH 的 status。

#### 不健康 Annotation（CAPM3 讀取，不設定）

CAPM3 只讀取這些 annotation，判斷是否跳過選取：

| Annotation Key | 說明 |
|---------------|------|
| `capm3.metal3.io/unhealthy` | 目前使用的不健康 annotation（由 Metal3Remediation 設定） |
| `capi.metal3.io/unhealthy` | 舊版不健康 annotation（已棄用，遷移期間仍支援） |

## 深度解析：Provisioning 觸發機制

### CAPM3 設定哪些欄位來觸發 Provisioning

`setHostSpec` 函式（`baremetal/metal3machine_manager.go`）是觸發 provisioning 的核心：

```go
func (m *MachineManager) setHostSpec(_ context.Context, host *bmov1alpha1.BareMetalHost) error {
    // 只有在 host.Spec.Image 為 nil 且有 UserData 時才設定（避免重複 provision）
    if host.Spec.Image == nil && host.Spec.CustomDeploy == nil &&
        m.Metal3Machine.Status.UserData != nil {

        host.Spec.Image = &bmov1alpha1.Image{
            URL:          m.Metal3Machine.Spec.Image.URL,
            Checksum:     ptr.Deref(m.Metal3Machine.Spec.Image.Checksum, ""),
            ChecksumType: bmov1alpha1.ChecksumType(checksumType),
            DiskFormat:   &m.Metal3Machine.Spec.Image.DiskFormat,
        }
        host.Spec.UserData    = m.Metal3Machine.Status.UserData
        host.Spec.MetaData    = m.Metal3Machine.Status.MetaData    // 若有 DataTemplate
        host.Spec.NetworkData = m.Metal3Machine.Status.NetworkData // 若有 DataTemplate
    }
    // AutomatedCleaningMode 設定（不受 Image nil 條件限制）
    host.Spec.AutomatedCleaningMode = bmov1alpha1.AutomatedCleaningMode(
        m.Metal3Machine.Spec.AutomatedCleaningMode)
    host.Spec.Online = true  // 機器上線
    return nil
}
```

**重要設計：** 函式中的 `if host.Spec.Image == nil` 條件確保已 provisioned 的 BMH 不會被重新 provision。若要更換 image，必須先完整走一遍 deprovisioning 流程，再重新 provision。

### Deprovisioning 觸發機制（Delete 流程）

`Delete` 函式清除 BMH spec 欄位觸發 deprovisioning：

```go
func (m *MachineManager) Delete(ctx context.Context) error {
    // 清除 provisioning 相關欄位
    if host.Spec.Image != nil {
        host.Spec.Image = nil           // 清除此欄位觸發 BMO deprovisioning
        bmhUpdated = true
    }
    if host.Spec.CustomDeploy != nil {
        host.Spec.CustomDeploy = nil
        bmhUpdated = true
    }
    if m.Metal3Machine.Status.UserData != nil && host.Spec.UserData != nil {
        host.Spec.UserData = nil
        bmhUpdated = true
    }
    if m.Metal3Machine.Status.MetaData != nil && host.Spec.MetaData != nil {
        host.Spec.MetaData = nil
        bmhUpdated = true
    }
    if m.Metal3Machine.Status.NetworkData != nil && host.Spec.NetworkData != nil {
        host.Spec.NetworkData = nil
        bmhUpdated = true
    }
    // ...
}
```

### automatedCleaningMode 與 Online 欄位的交互

Deprovisioning 時，CAPM3 會根據 `AutomatedCleaningMode` 和 `CAPM3_FAST_TRACK` 環境變數調整 BMH 的 `spec.online` 欄位：

| CAPM3_FAST_TRACK | automatedCleaningMode | DisablePowerOff | BMH online |
|-----------------|----------------------|-----------------|-----------|
| false | disabled | false | false（先關機） |
| true | disabled | false | false（先關機） |
| false | metadata | false | false（先關機） |
| true | metadata | false | **true**（不關機，快速清理） |
| 任何 | 任何 | true | true（禁止關機） |

設定 `CAPM3_FAST_TRACK=true` 且 `automatedCleaningMode=metadata` 時，Ironic 可以在機器開著的狀態下執行磁碟清理，省去關機重開的時間，大幅加速節點回收速度。

### CAPM3 等待 Provisioned 的機制

`IsBaremetalHostProvisioned` 函式直接查詢 BMH status：

```go
func (m *MachineManager) IsBaremetalHostProvisioned(ctx context.Context) bool {
    host, _, err := m.getHost(ctx)
    if host == nil || err != nil {
        return false
    }
    return host.Status.Provisioning.State == bmov1alpha1.StateProvisioned
}
```

在 controller 的 `reconcileNormal` 中，每次 reconcile 都呼叫此函式。若尚未 provisioned，controller 會設定 `requeueAfter`（30 秒），由 controller-runtime 的 watch 機制在 BMH status 變化時觸發下次 reconcile，無需空等。

## 深度解析：BMH 選取失敗的 Fallback 行為

### 無可用 BMH 時的重試策略

當 `chooseHost` 回傳 `nil`（沒有任何符合條件的 BMH），`Associate` 函式會觸發 30 秒後的重試：

```go
if host == nil {
    m.Log.Info("No available BareMetalHost found for Metal3Machine, requeuing")
    return "", WithTransientError(
        errors.New("no available host found. Requeuing"),
        requeueAfter,   // 30 秒
    )
}
```

`WithTransientError` 回傳一個特殊的 `ReconcileError`，controller 收到後會設定 `ctrl.Result{RequeueAfter: 30s}`。

### Node Reuse BMH 尚未就緒的重試

若 nodeReuse BMH 存在但尚未進入 `ready`/`available` 狀態：

```go
errMessage := fmt.Sprint("Found BareMetalHost(s) with nodeReuseLabelName in not-available state, requeuing")
return nil, nil, "", WithTransientError(errors.New(errMessage), requeueAfter)
```

同樣是 30 秒後重試，且**不會回落**到其他一般 BMH。

### BMH 進入 Error 狀態的處理

在 `chooseHost` 篩選階段，有 `status.errorMessage` 的 BMH 會被跳過：

```go
if host.Status.ErrorMessage != "" {
    continue
}
```

若 BMH 在 provisioning 過程中進入 error 狀態（`status.errorMessage` 被設定），CAPM3 下次 reconcile 時發現已綁定的 BMH 有 errorMessage，會透過 `Update` 流程繼續等待，並在 Metal3Machine 的 condition 上反映錯誤狀態。BMO 的錯誤通常需要管理員介入清除（如修正 IPMI 設定、重設 Ironic 任務）。

### Patch 衝突的處理

在 API server 層面，若多個進程同時嘗試更新同一個 BMH 物件：

```go
err = helper.Patch(ctx, host)
if ok := errors.As(err, &aggr); ok {
    if slices.ContainsFunc(aggr.Errors(), apierrors.IsConflict) {
        return "", WithTransientError(nil, requeueAfter)
    }
}
```

遇到 `Conflict` 錯誤時，CAPM3 也會 requeue 30 秒後重試，重新讀取最新的 BMH 版本再操作。

## 深度解析：硬體規格過濾（hostSelector）

### spec.hostSelector 的作用

`Metal3Machine.spec.hostSelector` 允許管理員指定只選取帶有特定 label 的 BMH，實現硬體規格的靜態過濾。對應的 Go 型別定義在 `api/v1beta2/common_types.go`：

```go
type HostSelector struct {
    // 精確 label 比對（key=value）
    MatchLabels map[string]string `json:"matchLabels,omitempty"`

    // 表達式比對（支援 In、NotIn、Exists、DoesNotExist 等運算子）
    MatchExpressions []HostSelectorRequirement `json:"matchExpressions,omitempty"`
}

type HostSelectorRequirement struct {
    Key      string             `json:"key,omitempty"`
    Operator selection.Operator `json:"operator,omitempty"`
    Values   []string           `json:"values,omitempty"`
}
```

`hostLabelSelectorForMachine` 函式將這個結構轉換成 Kubernetes 原生的 `labels.Selector`：

```go
func hostLabelSelectorForMachine(machine *infrav1.Metal3Machine, log logr.Logger) (labels.Selector, error) {
    labelSelector := labels.NewSelector()
    if machine.Spec.HostSelector == nil {
        return labelSelector, nil  // 沒有設定 -> 選取所有 BMH
    }
    // 處理 MatchLabels：轉為 Equals 需求
    for labelKey, labelVal := range machine.Spec.HostSelector.MatchLabels {
        r, _ := labels.NewRequirement(labelKey, selection.Equals, []string{labelVal})
        labelSelector = labelSelector.Add(*r)
    }
    // 處理 MatchExpressions：operator 轉為小寫再建立需求
    for _, req := range machine.Spec.HostSelector.MatchExpressions {
        lowercaseOperator := selection.Operator(strings.ToLower(string(req.Operator)))
        r, _ := labels.NewRequirement(req.Key, lowercaseOperator, req.Values)
        labelSelector = labelSelector.Add(*r)
    }
    return labelSelector, nil
}
```

### 使用範例

以下範例選取帶有 `size=large` label 且 `role` 值在 `[control-plane, infra]` 清單中的 BMH：

```yaml
apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
kind: Metal3Machine
spec:
  hostSelector:
    matchLabels:
      size: large
    matchExpressions:
      - key: role
        operator: In
        values:
          - control-plane
          - infra
```

### CAPM3 不做硬體資源驗證

重要的是：**CAPM3 本身不讀取 BMH 的 `status.hardware` 欄位**（CPU 核心數、記憶體大小、磁碟容量等）進行資源符合性檢查。硬體篩選完全依賴管理員在 BMH 上手動貼 label，再由 `hostSelector` 的 label matching 實現。

若需要更細緻的硬體過濾（例如「至少 32GB RAM」），管理員需要在 BMO 完成 introspection 後，自行根據 `status.hardware` 欄位為 BMH 貼上對應的 label，再透過 `hostSelector` 引用這些 label。

## Node Reuse 完整生命週期

Node Reuse 是 CAPM3 的特殊功能，讓滾動升級時同一台裸機被重用，而不是釋放後重新選取。其運作機制橫跨整個 BMH 生命週期。

### 啟用條件

在 `Metal3MachineTemplate` 中設定：

```yaml
apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
kind: Metal3MachineTemplate
spec:
  nodeReuse: true
  template:
    spec:
      automatedCleaningMode: disabled  # 建議搭配使用
```

### Deprovisioning 時：設定 nodeReuse label

當節點被刪除（例如滾動升級替換舊版節點）且 `nodeReuse: true` 時，CAPM3 在清除 `ConsumerRef` 前，先在 BMH 上貼 `nodeReuseLabelName` label：

```go
if ptr.Deref(m3mt.Spec.NodeReuse, false) {
    if m.isControlPlane() {
        cpName, _ := m.getControlPlaneName(ctx)
        host.Labels[nodeReuseLabelName] = cpName  // 值為 ControlPlane 名稱
    } else {
        mdName, _ := m.getMachineDeploymentName(ctx)
        host.Labels[nodeReuseLabelName] = mdName  // 值為 MachineDeployment 名稱
    }
}
```

- Control Plane 節點：label value = KubeadmControlPlane 名稱
- Worker 節點：label value = MachineDeployment 名稱

### Provisioning 時：優先選取有 nodeReuse label 的 BMH

新版節點的 Metal3Machine 在 `chooseHost` 時，`nodeReuseLabelMatches` 會比對 label value 與當前機器所屬的 KubeadmControlPlane 或 MachineDeployment 名稱是否一致：

```go
func (m *MachineManager) nodeReuseLabelMatches(ctx context.Context, host *bmov1alpha1.BareMetalHost) bool {
    if m.isControlPlane() {
        cp, _ := m.getControlPlaneName(ctx)
        return host.Labels[nodeReuseLabelName] == cp
    }
    md, _ := m.getMachineDeploymentName(ctx)
    return host.Labels[nodeReuseLabelName] == md
}
```

這樣同一個 ControlPlane 或 MachineDeployment 滾動升級時，每台新節點都能優先「認領」對應的舊裸機，實現 1:1 的機器重用。

### Associate 時：清除 nodeReuse label

選機成功後，`setHostConsumerRef` 會清除 nodeReuseLabelName，讓 BMH 離開「待重用」狀態：

```go
if _, ok := labels[nodeReuseLabelName]; ok {
    delete(host.Labels, nodeReuseLabelName)
}
```
