## 🌐 整體觀念：BMH 是「硬體的 Kubernetes 物件」

**BareMetalHost（BMH）** 是由 [metal3-io/baremetal-operator](https://github.com/metal3-io/baremetal-operator) 定義的 CRD，代表一台真實的裸機伺服器。它是 CAPM3 與實體硬體世界溝通的唯一橋梁：

- **CAPM3** 透過修改 BMH 的 `spec` 欄位來「請求」BMO 執行特定動作，自身不直接操作硬體
- **BMO（Bare Metal Operator）** 監控 BMH 物件，透過 Ironic 執行實際操作（PXE 開機、IPMI 電源控制、Redfish API）
- **BMH 的狀態**（`status.provisioning.state`）反映裸機的實際狀態，由 BMO 負責維護與更新

<Callout type="info" title="概念類比：醫院病床管理系統">

把 BMH 想像成醫院的一張病床：

- **Available（可用）**：病床空著，清潔完畢，等待下一位病人入住
- **Provisioning（進行中）**：病人（OS）正在入住，護士正在準備床單（userdata/networkdata）
- **Provisioned（已佔用）**：病人已在床上，機器正常服務中
- **Deprovisioning（退院中）**：病人出院，護士正在清潔病床（磁碟清除）
- **Inspecting（檢查中）**：健檢中，確認病床可用（hardware inventory）
- **Error（故障）**：病床壞了，需要維修（需要人工介入）

BMH 的狀態機確保每台實體機器在任何時刻的狀態都是明確且可追蹤的。

</Callout>

## 🔄 核心流程：一台裸機從硬體到 Kubernetes Node 的旅程

> 💡 **場景**：機房裡有一台裸機伺服器，管理員已建立對應的 BMH 物件。現在 CAPI 要把它變成一個 Kubernetes Worker Node。整個過程，BMH 的狀態如何流轉？CAPM3 在每個關鍵時刻做了什麼？

**誕生：BMH 的初始化（由 BMO 完成）**

BMH 物件建立後，BMO 自動驅動 Ironic 完成兩個階段：
1. `registering` → `inspecting`：BMO 初始化 IPMI/BMC 連線，Ironic 取得硬體資訊（CPU、記憶體、磁碟）
2. `inspecting` → `available`：introspection 完成，BMH 進入可被選取的就緒狀態

**選機：CAPM3 搶先「佔床」**

`Metal3MachineReconciler` 掃描 namespace 內的 BMH，依優先順序（node-reuse → ready → available）選定一台。選定後，**立刻**設定 `BMH.Spec.ConsumerRef` 指向此 `Metal3Machine`，將 BMH「鎖住」，防止競爭。

**備料：設定 image、userData、metaData、networkData**

CAPM3 依序完成所有設定，然後在 BMH spec 上設定 `spec.image`（OS image URL）。BMO 偵測到 `spec.image` 出現，這是觸發 provisioning 的訊號：BMH 狀態轉為 `provisioning`。

**施工：Ironic 接手（CAPM3 退到後台輪詢）**

這段時間 CAPM3 每 30 秒查詢一次 `BMH.Status.Provisioning.State`，自己不做任何硬體操作。Ironic 透過 PXE boot、寫入磁碟、安裝 OS，完成後 BMH 狀態變為 `provisioned`。

**就緒：CAPM3 通知 CAPI 任務完成**

CAPM3 發現 `provisioned` 狀態後：
- 設定 ProviderID：`metal3://<namespace>/<bmh-name>/<m3m-name>`
- 透過 ClusterCacheTracker 取得 workload cluster client，patch Node 的 `spec.providerID`
- 設定 `Metal3Machine.Status.Ready = true`

**告別：機器刪除的三步退場**

當 CAPI 要刪除這台機器時：
1. CAPM3 清除 `BMH.Spec.Image`（設為 nil）→ BMO 偵測到，開始 deprovisioning（`provisioned` → `deprovisioning`）
2. 等待 deprovisioning 完成（Ironic 清除磁碟，`deprovisioning` → `available`）
3. CAPM3 清除 `BMH.Spec.ConsumerRef`，解除綁定，BMH 重新可被選取

## 🧩 元件詳解

### BMH 狀態機（完整）

BMH 的 `status.provisioning.state` 欄位記錄裸機當前所處的生命週期階段，所有狀態由 BMO 管理。

| 狀態 | 說明 | 轉換觸發條件 |
|------|------|------------|
| `registering` | BMH 剛建立，BMO 正在初始化 IPMI/BMC 連線 | 自動 |
| `inspecting` | BMO 透過 Ironic 取得硬體資訊（CPU、記憶體、磁碟） | 自動 |
| `available` / `ready` | 已完成 introspection，可被 CAPM3 選取 | introspection 完成 |
| `provisioning` | Ironic 正在安裝 OS（PXE boot + 寫入磁碟） | `BMH.spec.image` 被設定 |
| `provisioned` | OS 已安裝完成，機器已上線運行 | provisioning 完成 |
| `deprovisioning` | Ironic 正在清除磁碟、重置機器 | `BMH.spec.image` 被清除 |
| `available` | 釋放完成，可再次被選取 | deprovisioning 完成 |
| `deleting` | BMH 資源被刪除中 | BMH DeletionTimestamp 設定 |
| `error` | 發生無法自動恢復的錯誤 | 各階段錯誤 |
| `paused` | BMH 被暫停，不接受任何操作 | 設定 pause annotation |

<Callout type="warning" title="BMH 狀態由 BMO 控制">

BMH 的狀態轉換完全由 **BMO** 負責管理，CAPM3 只能透過修改 BMH 的 `spec` 欄位（如設定或清除 `spec.image`）來**間接**影響狀態轉換。CAPM3 無法直接強制改變 BMH 的 provisioning state。

</Callout>

### 建立機器：七步協作流程

**Step 1 — 選擇 BMH（chooseHost）**

| 篩選條件 | 說明 |
|---------|------|
| 相同 namespace | BMH 必須和 Metal3Machine 在同一個 namespace |
| 狀態 `available` 或 `ready` | 必須是可被選取的狀態 |
| 無 `errorMessage` | 沒有錯誤訊息 |
| 無 pause annotation | 沒有被暫停 |
| 無 unhealthy annotation | 沒有被標記為不健康 |
| `ConsumerRef` 為空 | 未被其他 Metal3Machine 佔用 |

選擇優先順序：帶有 `node-reuse` label（升級重用）→ Name-Based Preallocation → `ready` → `available`

**Step 2 — 設定 ConsumerRef**：鎖住 BMH，防止其他 Metal3Machine 選取同一台

**Step 3 — 設定 Image**：

| 欄位 | 說明 |
|------|------|
| `url` | OS image 下載 URL |
| `checksum` | image checksum 值 |
| `checksumType` | 校驗演算法：`md5` / `sha256` / `sha512` |
| `format` | image 格式：`raw` / `qcow2`（可選） |

**Step 4 — 設定 UserData**：讀取 CAPI Bootstrap Secret（KubeadmConfig 產生的 cloud-init），建立新 Secret 並掛到 `BMH.Spec.UserData`

**Step 5 — 設定 MetaData 與 NetworkData**（需 DataTemplate）：Metal3DataReconciler 渲染 Jinja2 模板，產生每台機器專屬的 hostname/IP/NIC 設定，寫入 Secret 後掛到 `BMH.Spec.MetaData` 和 `BMH.Spec.NetworkData`

**Step 6 — 等待 Provisioned**：每 30 秒輪詢 `BMH.Status.Provisioning.State`，直到 `provisioned`

**Step 7 — 設定 ProviderID 並標記就緒**：
- 從 BMH status 取得 IP addresses → 設定 `Metal3Machine.Status.Addresses`
- 設定 ProviderID：`metal3://<namespace>/<bmh-name>/<m3m-name>`
- Patch workload cluster Node 的 `spec.providerID`
- 設定 `Metal3Machine.Status.Ready = true`

### automatedCleaningMode 設定

`Metal3MachineTemplate.spec.template.spec.automatedCleaningMode` 控制 Ironic 在 deprovisioning 時的磁碟清理行為：

| 值 | 說明 |
|----|------|
| `metadata`（預設） | Ironic 執行完整的 disk cleaning（清除所有資料），確保資料安全 |
| `disabled` | 跳過 disk cleaning，大幅加速機器釋放速度 |

<Callout type="tip" title="Node Reuse 場景建議">

在 **Node Reuse**（節點升級重用）場景中，建議將 `automatedCleaningMode` 設為 `disabled`。由於升級時會重用同一台裸機，省略 disk cleaning 可以顯著縮短升級所需時間，加快整個升級流程。

</Callout>

### BMH pause 機制

在 BMH 加上以下 annotation 可暫停 BMO 對此 BMH 的所有操作：

```
baremetalhost.metal3.io/paused: ""
```

CAPM3 在 Cluster 被 pause 時會同步 pause 旗下的 BMH，並寫入固定值 `"metal3.io/capm3"` 作為 annotation value，讓 CAPM3 能識別是自己設定的 pause（還是管理員手動設定的）。

<Callout type="warning" title="注意事項">

應避免手動 pause 正在 `provisioning` 或 `deprovisioning` 狀態的 BMH。暫停進行中的操作可能導致 Ironic 任務懸空，需要手動干預才能恢復正常狀態。

</Callout>

### 深度解析：chooseHost 演算法

`chooseHost` 是 CAPM3 最核心的選機邏輯，定義在 `baremetal/metal3machine_manager.go`：

```go
const (
    HostAnnotation     = "metal3.io/BareMetalHost"
    nodeReuseLabelName = "infrastructure.cluster.x-k8s.io/node-reuse"
    requeueAfter       = time.Second * 30
    PausedAnnotationKey = "metal3.io/capm3"
)
```

篩選階段——遇到以下任一情況就直接跳過：

| 跳過條件 | 說明 |
|---------|------|
| `ConsumerRef != nil` 且不指向此 Metal3Machine | BMH 已被其他機器佔用 |
| `nodeReuseLabelName` 存在但不匹配當前機器 | Node Reuse label 屬於另一個 ControlPlane / MachineDeployment |
| `DeletionTimestamp != nil` | BMH 正在被刪除 |
| `Status.ErrorMessage != ""` | BMH 有錯誤訊息 |
| 有 `baremetalhost.metal3.io/paused` annotation | BMH 已被暫停 |
| 有 `capm3.metal3.io/unhealthy` annotation | BMH 被標記為不健康 |
| 狀態不是 `ready` 或 `available` | 機器尚未準備好 |

選取邏輯：

```go
if len(availableHostsWithNodeReuse) != 0 {
    // Node Reuse 的 BMH 存在但尚未 Ready -> 強制 requeue，不回落到一般 BMH
    return nil, nil, "", WithTransientError(errors.New(errMessage), requeueAfter)
} else {
    chosenHost, err = m.pickHost(availableHosts)
}
```

<Callout type="warning" title="Node Reuse 的「不回落」行為">

若存在帶有 `nodeReuseLabelName` 的 BMH，但尚未進入 `ready`/`available`，CAPM3 **不會**回落選取一般 BMH，而是直接觸發 30 秒後重試。這確保升級場景下節點的穩定性，避免意外選到不同台機器。

</Callout>

### Node Reuse 完整生命週期

Node Reuse 讓滾動升級時同一台裸機被重用，而不是釋放後重新選取。啟用方式：

```yaml
apiVersion: infrastructure.cluster.x-k8s.io/v1beta1
kind: Metal3MachineTemplate
spec:
  nodeReuse: true
  template:
    spec:
      automatedCleaningMode: disabled  # 建議搭配使用
```

流程：
1. **Deprovisioning 時**：在 BMH 上貼 `nodeReuseLabelName` label（值為 KubeadmControlPlane 或 MachineDeployment 名稱）
2. **新節點 Provisioning 時**：chooseHost 優先選取 label 值匹配當前機器所屬 CP/MD 的 BMH
3. **Associate 成功後**：清除 `nodeReuseLabelName` label，BMH 離開「待重用」狀態

## ⚠️ 常見陷阱與注意事項

<Callout type="warning" title="常見誤解">

**誤解 1：「BMH 從 Provisioned 直接到 Available 需要很久」**
→ Deprovisioning → Cleaning 確實需要時間（磁碟清除 10-15 分鐘），但可透過 `automatedCleaningMode: disabled` 跳過 cleaning，搭配 Node Reuse 功能縮短整體時間。

**誤解 2：「BMH status.errorMessage 只有在 Error 狀態才會出現」**
→ 不對。某些子狀態（如 provisioning 中的 PXE 失敗）也會在 `status.errorMessage` 中留下記錄，但 BMH 可能還沒進入 Error 狀態。定期監控 errorMessage 是重要的運維習慣。

**誤解 3：「刪除 Metal3Machine 物件後 BMH 會立即回到 Available」**
→ 不對。刪除 Metal3Machine 觸發 deprovisioning 流程，BMH 需要完成 deprovisioning → cleaning 後才回到 Available。如果 cleaning 卡住，BMH 可能長時間停留在 deprovisioning 狀態。

**誤解 4：「CAPM3 和 BMO 同時修改 BMH 會衝突」**
→ CAPM3 設定 BMH 的 spec（如 image、consumerRef），BMO 更新 BMH 的 status。兩者操作不同欄位，透過 Kubernetes API Server 的樂觀鎖（resourceVersion）避免衝突。

</Callout>
