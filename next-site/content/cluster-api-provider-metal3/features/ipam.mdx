## 什麼是 IPAM

在bare-metal（Bare Metal）環境中，IP 位址管理（IP Address Management，IPAM）是不可或缺的基礎設施能力。與雲端環境不同，bare-metalnode通常無法依賴 DHCP 動態取得 IP，原因在於：

- **穩定性需求**：生產環境要求node IP 固定，以確保 DNS 記錄、防火牆規則、監控設定不受影響
- **網路隔離**：bare-metal網路環境常有嚴格的 VLAN 或 ACL 限制，DHCP 服務可能不存在於所有網段
- **可預測性**：Kubernetes cluster的node IP 應在整個生命週期中保持一致，尤其是 Control Plane node

Metal3 支援兩種 IPAM 機制，適應不同的部署需求：

| 機制 | API Group | 說明 |
|------|-----------|------|
| **Metal3 自帶 IPAM** | `ipam.metal3.io` | 使用 IPPool CRD，整合在 Metal3DataTemplate 中，Metal3 生態系原生支援 |
| **CAPI 標準 IPAM** | `ipam.cluster.x-k8s.io` | CAPI 定義的標準 IPAM 介面（IPAddressClaim / IPAddress），可接入不同 IPAM 後端 |

<Callout type="info" title="概念類比：大樓的門牌號碼系統">

把bare-metalcluster的 IP 管理想像成一棟大樓的**門牌號碼系統**：

- **IPPool**：大樓的門牌號碼清單（101–150 號）
- **每台 BMH**：一間公寓，需要被分配一個固定門牌
- **preAllocations**：「101 號永遠給 Control Plane 1」的預約記錄
- **IP 分配**：新租客（新機器）入住時，系統從清單中找一個未使用的號碼

關鍵差異：雲端環境的 IP 可以像飯店房間號碼隨時換，但bare-metal的 DNS 記錄、防火牆規則、監控設定都依賴固定 IP——如果 IP 變了，就像換了門牌但快遞記錄還是舊地址，所有寄件都送不到。

</Callout>

## Metal3 自帶 IPAM（ipam.metal3.io）

### IPPool CRD

`IPPool` 是 Metal3 自帶 IPAM 的核心資源，定義一段可供分配的 IP 範圍。Metal3DataTemplate 在渲染時，會透過 IPPool 分配 IP 給每台機器。

**IPPool Spec 欄位：**

| 欄位 | 類型 | 說明 |
|------|------|------|
| `start` | string | IP 範圍起始地址 |
| `end` | string | IP 範圍結束地址 |
| `subnet` | string | 子網路（CIDR 格式，例如 `192.168.1.0/24`）|
| `gateway` | string | 預設gateway |
| `dns` | []string | DNS server 列表 |
| `namePrefix` | string | 分配的 IP 物件名稱前綴 |
| `clusterName` | string | 關聯的cluster名稱 |
| `preAllocations` | map[string]string | 預先分配的 IP（索引 → IP）|

**IPPool YAML 範例：**

```yaml
apiVersion: ipam.metal3.io/v1alpha1
kind: IPPool
metadata:
  name: my-worker-pool
  namespace: metal3
spec:
  clusterName: my-cluster
  namePrefix: my-worker
  subnet: 192.168.1.0/24
  start: 192.168.1.100
  end: 192.168.1.200
  gateway: 192.168.1.1
  dns:
    - 8.8.8.8
    - 8.8.4.4
  preAllocations:
    "0": "192.168.1.100"   # index 0 固定分配 192.168.1.100
    "1": "192.168.1.101"   # index 1 固定分配 192.168.1.101
```

### IPClaim 與 IPAddress

IPPool 的 IP 分配透過以下流程完成：

1. **Metal3DataReconciler** 在渲染模板時，為每個 `Metal3Data`（每台機器）建立 `IPClaim` 物件，請求一個 IP
2. **IPPool controller** 偵測到新的 `IPClaim` → 從 IP 範圍中選出一個未使用的 IP → 建立對應的 `IPAddress` 物件
3. **Metal3DataReconciler** 讀取 `IPAddress` 中的位址資訊 → 填入模板中的 IP 相關變數，完成網路設定渲染

這種非同步的 claim/address 模式讓 IP 分配與模板渲染解耦，IPPool controller 可以獨立管理 IP 資源的生命週期。

### 在 Metal3DataTemplate 中引用 IPPool

Metal3DataTemplate 透過多種欄位從 IPPool 取得不同的網路資訊：

```yaml
spec:
  metaData:
    ipAddressesFromIPPool:
      - key: provisioningIP          # 模板變數名稱
        name: my-provisioning-pool   # IPPool 名稱
    prefixesFromIPPool:
      - key: provisioningCIDR
        name: my-provisioning-pool
    gatewaysFromIPPool:
      - key: provisioningGateway
        name: my-provisioning-pool
  networkData:
    networks:
      ipv4:
        - id: eth0
          link: eth0
          ipAddressFromIPPool: my-worker-pool    # 從此 IPPool 分配 IP
          routes:
            - network: 0.0.0.0
              prefix: 0
              gateway:
                fromIPPool: my-worker-pool
```

`metaData` 中的 IPPool 引用會將分配到的 IP、前綴長度、gateway位址分別對應為模板變數，供 cloud-init 或其他初始化腳本使用。`networkData` 中的 `ipAddressFromIPPool` 則直接控制網路介面的 IP 設定。

### preAllocations 功能


<Callout type="tip" title="preAllocations — 固定 IP 分配">

`preAllocations` 讓管理員在 IPPool 中預先指定某個索引（index）對應的 IP 地址：

- **索引即身份**：每個 Metal3Machine 在機器集合中有對應的索引（例如第 0 台、第 1 台）
- **固定映射**：當機器索引在 `preAllocations` 中有記錄時，IPPool controller 會直接使用指定 IP，而非從範圍中動態選取
- **搭配 enableBMHNameBasedPreallocation**：啟用此功能後，特定 BMH 會固定使用對應索引的 IP，確保升級時機器重用相同 IP
- **避免 IP 漂移**：即使機器被刪除重建，只要索引不變，IP 就不會改變，適合有外部依賴（DNS、防火牆）的環境

</Callout>

## CAPI 標準 IPAM（ipam.cluster.x-k8s.io）

Cluster API 定義了通用的 IPAM 介面規範，讓不同的 IPAM 後端都能與 CAPI 生態系整合。Metal3 支援這套標準介面，讓使用者可以將企業現有的 IPAM 系統接入 Metal3 cluster管理流程。

### IPAddressClaim

當 Metal3DataTemplate 配置為使用 CAPI IPAM 時，reconciler 會建立 `IPAddressClaim` 物件，向指定的 IPAM 後端請求 IP：

```yaml
apiVersion: ipam.cluster.x-k8s.io/v1alpha1
kind: IPAddressClaim
metadata:
  name: my-machine-0
spec:
  clusterName: my-cluster
  poolRef:
    apiGroup: ipam.infoblox.example.com
    kind: InfobloxPool
    name: my-infoblox-pool
```

`poolRef` 指向具體的 IPAM 後端資源，不同後端有不同的 `apiGroup` 和 `kind`。

### IPAddress

IPAM 後端在處理 claim 後，建立 `IPAddress` 物件回應請求，其中包含完整的網路配置資訊：

```yaml
apiVersion: ipam.cluster.x-k8s.io/v1alpha1
kind: IPAddress
metadata:
  name: my-machine-0
spec:
  clusterName: my-cluster
  poolRef:
    apiGroup: ipam.infoblox.example.com
    kind: InfobloxPool
    name: my-infoblox-pool
  address: 192.168.1.100
  prefix: 24
  gateway: 192.168.1.1
```

Metal3DataReconciler 讀取此 `IPAddress` 物件，將 `address`、`prefix`、`gateway` 填入 networkData 模板，完成node網路設定。

### 支援的 IPAM 後端


<Callout type="info" title="可接入的 IPAM 工具">

任何實作了 `ipam.cluster.x-k8s.io` 合約（IPAddressClaim → IPAddress）的 IPAM 工具都可以與 Metal3 整合，例如：

- **Infoblox IPAM provider**：適合已部署 Infoblox 的企業環境
- **in-cluster IPAM**（CAPI 內建）：`sigs.k8s.io/cluster-api/exp/ipam`，輕量型選項，不依賴外部系統
- **自定義 IPAM 實作**：只需實作 IPAddressClaim 的 reconciler，建立對應的 IPAddress 物件即可接入

這套標準介面讓 Metal3 能在混合環境（已有企業 IPAM 系統）中無縫運作，而無需替換現有 IP 管理基礎設施。

</Callout>

## 兩種 IPAM 方案比較

| 比較項目 | ipam.metal3.io（IPPool）| ipam.cluster.x-k8s.io（標準）|
|---------|----------------------|--------------------------|
| CRD 來源 | metal3-io/ip-address-manager | sigs.k8s.io/cluster-api |
| 整合方式 | Metal3DataTemplate 直接引用 | 透過 IPAddressClaim/IPAddress |
| 適用後端 | Metal3 自帶 | 任意 IPAM 工具 |
| preAllocations | ✅ 支援 | 依後端而定 |
| 成熟度 | Metal3 生態系標準 | Alpha（v1alpha1）|
| 適用場景 | Metal3 專屬環境 | 混合環境、已有企業 IPAM |

**選擇建議：**
- 若部署環境完全由 Metal3 管理，且無現有 IPAM 系統，選擇 **ipam.metal3.io IPPool**，整合最直接，功能最完整
- 若企業已有 IPAM 系統（如 Infoblox、NetBox），或需要在多個 Infrastructure Provider 間共用 IPAM，選擇 **ipam.cluster.x-k8s.io**

## enableBMHNameBasedPreallocation 功能

`enableBMHNameBasedPreallocation` 是 CAPM3 的進階功能，專為需要 IP 位址穩定性的生產環境設計。

**啟用方式：** 在 CAPM3 controller manager 的啟動參數中加入 `--enable-bmh-name-based-preallocation`。

**運作原理：**

啟用後，CAPM3 會在 Metal3Machine 的名稱與 BareMetalHost（BMH）名稱之間建立固定映射關係。這個映射會進一步與 IPPool 的 `preAllocations` 結合，確保同一台 BMH 在每次配置時都取得相同的 IP 地址。

**升級場景應用：**

在cluster升級過程中，舊的 Metal3Machine 物件會被刪除，新的 Metal3Machine 物件被建立。若升級策略是原地重建（重用同一台 BMH），啟用此功能後，新機器會繼承舊機器的 BMH 名稱映射，從而取得相同的 IP 地址，整個升級過程對外部系統（DNS、防火牆、監控）完全透明。

**適用場景：**

- node IP 被記錄在 DNS 中，IP 改變會導致解析失敗
- 防火牆規則、安全群組依賴固定 IP 白名單
- 監控系統（Prometheus、Zabbix）以 IP 為目標進行配置
- 合規要求需要網路存取記錄與node IP 一一對應

**注意事項：**

此功能需要 IPPool 中的 `preAllocations` 已正確設定，且 BMH 的命名規則符合 CAPM3 的映射邏輯。在啟用前，建議在測試環境充分驗證 BMH 名稱到索引的映射是否符合預期，再套用至生產環境。

## 常見誤解與注意事項

<Callout type="warning" title="常見誤解">

**誤解 1：「IPAM 只是可選功能，不設定也沒關係」**
→ 取決於環境。在使用 DHCP 的測試環境可以不設。但生產環境若需要靜態 IP（DNS、防火牆、監控依賴），IPAM 就是必須的，否則每次機器重建後 IP 可能改變。

**誤解 2：「ipam.metal3.io 和 ipam.cluster.x-k8s.io 是競爭關係，選一個就好」**
→ 不完全對。Metal3 自帶的 IPPool（`ipam.metal3.io`）整合最直接；CAPI 標準 IPAM（`ipam.cluster.x-k8s.io`）適合已有企業 IPAM 系統的環境。可以根據你的基礎設施需求選擇。

**誤解 3：「preAllocations 設定後就不需要 Node Reuse 了」**
→ 兩者解決不同問題。`preAllocations` 確保 IP 不變；Node Reuse 確保同一台 BMH 被重用（省去 deprovisioning 時間）。搭配使用效果最好。

**誤解 4：「IPPool 耗盡後機器會自動等待 IP 釋放」**
→ 是的，但會造成機器卡在 provisioning 佇列。建議在 IPPool 建立時預留足夠的 IP 範圍，並在 `preAllocations` 中為 Control Plane node固定 IP。

</Callout>
